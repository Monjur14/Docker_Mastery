Task 19  ·  Intermediate

Push Your Image
to Docker Hub

Authenticate with Docker Hub, retag your local image with your Docker Hub username, and push it to a public repository — making it pullable by anyone on the internet.

~15 minutes Task 18 completed Docker Hub account
📖Required concepts
🏪
Docker Hub as a Registry
Docker Hub is Docker's official public registry — a hosted store for images. A registry holds repositories, and each repository holds tagged image versions. Your URL looks like docker.io/username/reponame:tag.
🏷️
Naming Convention for Push
To push to Docker Hub, your image name must include your username as the first segment: username/reponame:tag. Docker uses this prefix to know which Hub account to push to. Locally-named images like myapp:1.1.0 need to be retagged first.
🔑
docker login & Access Tokens
docker login authenticates your CLI with Docker Hub and stores credentials locally. Use an Access Token (not your password) for the password prompt — tokens are scoped, revocable, and safer than your real credentials.
⬆️
Pushing Only Uploads New Layers
Like pulling, Docker only uploads layers that don't already exist on Hub. If you push a second version of the same image and most layers are unchanged, Docker skips them with "Layer already exists" — making pushes as fast as possible.
🌐
Public vs. Private Repositories
Free Docker Hub accounts get unlimited public repos and one private repo. A public repo means anyone can docker pull your image — no login required. Private repos require authentication to pull.
🔁
docker tag — Alias, Not a Copy
docker tag creates a new name that points to the same image layers — no data is duplicated. It's like creating a symlink. Both the old name and new name refer to identical content; only the label differs.
🏗️Image naming anatomy
johndoe / myapp : 1.1.0
username

Your Docker Hub account name — acts as the namespace. Docker Hub uses this to route the push to the right account. Must match exactly.

reponame

The repository name on Hub. Created automatically on first push if it doesn't exist yet. Keep it lowercase, no spaces.

tag

The version label. Omitting it defaults to latest — avoid relying on latest in production since it's mutable.

🔬Command anatomy
docker push johndoe / myapp:1.1.0
docker

The Docker CLI — sends the push request to the daemon.

push

Uploads image layers to the registry. Only transmits layers not already present on the remote.

username

Your Docker Hub username. Must match the account you authenticated with via docker login.

repo:tag

Repository name and version tag. The combination must match a locally tagged image exactly.

🪜Step-by-step guide
1
Create a Docker Hub account (if you don't have one)
Go to hub.docker.com and register. Your username is permanent and becomes the namespace for all your public images — choose it carefully.
ℹ️
Note your username exactly as shown on your profile page — it's case-sensitive when used in image names and commands.
2
Generate an Access Token on Docker Hub
Rather than using your Docker Hub password in the CLI, create a scoped Access Token. Go to hub.docker.com → Account Settings → Security → New Access Token. Give it a name like local-dev and set permissions to Read, Write, Delete.
🔒
Copy the token immediately — Docker Hub only shows it once. Store it in a password manager. Tokens can be revoked any time without changing your password.
3
Log in to Docker Hub from the CLI
Run docker login. Enter your Docker Hub username and paste the Access Token when prompted for a password.
shell 
docker login
expected output
Username: johndoe
Password: [paste your Access Token here — input is hidden]

Login Succeeded
Login with your Docker ID to push and pull images from Docker Hub.
✅
Credentials are saved to ~/.docker/config.json. You won't need to log in again on this machine until the token expires or is revoked.
4
Retag your local image with your Docker Hub username
Your local image is named myapp:1.1.0. To push it, Docker needs the full username/reponame:tag format. Replace johndoe with your actual username.
shell 
docker tag myapp:1.1.0 johndoe/myapp:1.1.0
ℹ️
docker tag doesn't copy or duplicate anything — it creates a second label pointing to the same image data. Both myapp:1.1.0 and johndoe/myapp:1.1.0 now refer to identical layers.
5
Verify both tags exist locally
List your images to confirm the retagged version appeared alongside the original.
shell 
docker images | grep myapp
expected output
REPOSITORY TAG IMAGE ID CREATED SIZE
johndoe/myapp 1.1.0 c9d2e3f4a1b8 2 hours ago 185MB
myapp 1.1.0 c9d2e3f4a1b8 2 hours ago 185MB
💡
Notice both rows share the same IMAGE ID — proof that docker tag created an alias, not a new image. No extra disk space used.
6
Push the image to Docker Hub
Upload the image. Docker will transmit each layer individually and skip any that already exist on Hub.
shell 
docker push johndoe/myapp:1.1.0
expected output
The push refers to repository [docker.io/johndoe/myapp]
4f2d8b3a1c9e: Pushing 1.536kB/1.536kB
9a7f3e2c1b4d: Pushing 2.048kB/3.584kB
7b6c8d4e2f1a: Layer already exists
2e9d5f7a3c8b: Layer already exists
a1b3c5e7d9f2: Layer already exists

1.1.0: digest: sha256:a3f9b2c1... size: 1163
🚀
"Layer already exists" means Docker Hub already has those layers (from a previous push or official base image). Only your new layers are actually uploaded — push is as efficient as pull.
7
Verify the push on Docker Hub
Open Docker Hub in your browser and confirm the repository and tag appeared.
url 
https://hub.docker.com/r/johndoe/myapp/tags
ℹ️
You should see the 1.1.0 tag listed with its compressed size, digest, and last pushed timestamp. The repository was created automatically on first push.
8
Test pulling your image from scratch
Delete your local copy to simulate a fresh machine, then pull it back from Hub to confirm the full round-trip works.
shell 
# Remove both local tags
docker rmi myapp:1.1.0
docker rmi johndoe/myapp:1.1.0

# Pull from Hub — as if on a different machine
docker pull johndoe/myapp:1.1.0

# Run it to confirm it works end-to-end
docker run -p 3000:3000 johndoe/myapp:1.1.0
expected output
1.1.0: Pulling from johndoe/myapp
7b6c8d4e2f1a: Pull complete
2e9d5f7a3c8b: Pull complete
Digest: sha256:a3f9b2c1...
Status: Downloaded newer image for johndoe/myapp:1.1.0
9
Also push the latest tag (optional but recommended)
It's convention to keep latest pointing to your most recent stable release. Create a second tag and push it — no new data is uploaded since layers are identical.
shell 
# Tag the same image as latest
docker tag johndoe/myapp:1.1.0 johndoe/myapp:latest

# Push the latest tag — layers already on Hub, nearly instant
docker push johndoe/myapp:latest
expected output
The push refers to repository [docker.io/johndoe/myapp]
4f2d8b3a1c9e: Layer already exists
9a7f3e2c1b4d: Layer already exists
7b6c8d4e2f1a: Layer already exists
latest: digest: sha256:a3f9b2c1... size: 1163
⚠️
In production, treat latest as a convenience alias only — never depend on it in CI/CD pipelines or Kubernetes manifests. Always pin to an explicit version tag for reproducibility.
10
Log out when done (optional)
On shared or CI machines, always log out after pushing to remove stored credentials from disk.
shell 
docker logout
🔒
On your personal machine you can stay logged in. On shared servers or CI runners, always log out after pushing — or better yet, use short-lived tokens scoped only to the repository you need.
📡Push output reference

Each layer in docker push output shows one of these statuses.

Status message Meaning Data uploaded?
Pushing N/NkB Layer is being uploaded — progress shown in bytes uploading
Pushed Layer upload completed successfully yes
Layer already exists Hub already has this layer — skipped entirely skipped
Mounted from library/node Layer exists in an official image — Hub cross-mounts it, no upload needed skipped
digest: sha256:… Final manifest digest — unique fingerprint of the full image complete
🔍What happened under the hood
Step 01
🔑
Token auth
CLI sent credentials to Hub. An auth token was stored in ~/.docker/config.json.
Step 02
🏷️
Retag
docker tag added a Hub-routable alias — same image ID, new name.
Step 03
🔍
Layer check
Hub compared each layer's digest against what it already stores.
Step 04
⬆️
Upload new layers
Only your app-specific layers were transmitted — base OS layers were cross-mounted.
Step 05
📋
Manifest written
Hub stored the image manifest linking all layers under your tag.
Step 06
🌐
Publicly available
Anyone can now docker pull johndoe/myapp:1.1.0 — no login required.
