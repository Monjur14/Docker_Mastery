Task 33  ·  Advanced

Container
Resource Limits

Restrict how much CPU and memory a container can consume using --cpus and --memory. Learn what happens when a container exceeds its quota — and how to verify enforcement is actually working.

~15 minutes Docker installed Terminal / Shell
📖 Required concepts
🚨
Why limits matter
Without limits, a single runaway container can consume all available CPU or RAM on the host — starving every other container and the host OS itself. In any multi-tenant or production environment, resource limits are not optional — they're the contract between a container and the host it shares.
⚙️
cgroups — the kernel mechanism
Docker enforces limits via Linux control groups (cgroups). When you pass --memory or --cpus, Docker writes cgroup configuration files. The kernel then enforces the ceiling directly — Docker itself never intervenes after setup.
🖥️
--cpus vs --cpu-shares
--cpus sets a hard ceiling — e.g. --cpus 0.5 means the container can never use more than half a CPU core, even if others are idle. --cpu-shares sets a relative weight — only throttles when the host is under contention. For predictable behaviour, prefer --cpus.
💾
--memory and OOM kills
--memory sets the maximum RAM a container can allocate. When the process tries to exceed it, the kernel's OOM killer terminates the container — exit code 137 (128 + SIGKILL). The container is stopped, not paused.
🔄
--memory-swap
By default Docker sets swap to 2× --memory, so a container can burst beyond its RAM limit into swap. Set --memory-swap equal to --memory to disable swap entirely, or -1 to allow unlimited swap. In production, disabling swap gives you predictable kill behaviour.
📊
docker stats — live monitoring
docker stats streams live CPU%, memory usage and limit, network I/O, and block I/O for running containers. It's the fastest way to confirm a limit is being enforced — watch the memory column hit its ceiling and stop climbing.
🔬 Command anatomy
docker run --cpus 0.5 --memory 128m --memory-swap 128m nginx
docker run

Create and start a new container from the specified image.

--cpus

Maximum CPU cores the container may use. 0.5 = half a core. 2 = two full cores. Fractional values are supported.

--memory

Hard RAM ceiling. Accepts b, k, m, g suffixes. Minimum is 6m. Processes that exceed this are OOM-killed.

--memory-swap

Total memory + swap combined. Setting it equal to --memory disables swap entirely — the container lives and dies within its RAM quota.

nginx

The image to run. Any image works — the limits apply to whatever process runs inside.

📊 What limits look like on a host

A host with 4 CPU cores and 8 GB RAM runs three containers, each with explicit limits. The host OS and other processes still need headroom — never sum limits to 100%.

🖥️  CPU allocation
web (--cpus 1.0)
worker (--cpus 0.5)
db (--cpus 2.0)
host OS + headroom
3.5 / 4.0 cores allocated
💾  Memory allocation
web (--memory 256m)
worker (--memory 128m)
db (--memory 2g)
host OS + headroom
2.4 / 8.0 GB reserved
🪜 Step-by-step guide
1
Run a container with no limits — see the default
Start nginx with no constraints and inspect what Docker reports as its memory limit. Without flags, the container can access all host memory.
shell 
docker run -d --name unlimited nginx
docker inspect unlimited --format 'Memory: {{.HostConfig.Memory}}  NanoCpus: {{.HostConfig.NanoCpus}}'
expected output
Memory: 0 NanoCpus: 0   ← 0 means no limit set
⚠️
Memory: 0 and NanoCpus: 0 means unlimited — this container can consume every byte of RAM and every CPU cycle on the host if it chooses to.
2
Run a container with CPU and memory limits
Start a second nginx container restricted to half a CPU core and 128 MB of RAM, with swap disabled.
shell 
docker run -d \
  --name limited \
  --cpus "0.5" \
  --memory "128m" \
  --memory-swap "128m" \
  nginx
ℹ️
Setting --memory-swap equal to --memory means zero swap is available. The container lives and dies within its 128 MB RAM quota — no bursting into slower swap memory.
3
Verify limits are set via docker inspect
Inspect the container's HostConfig to confirm the kernel cgroup values Docker wrote. NanoCpus is CPUs × 10⁹.
shell 
docker inspect limited \
  --format 'Memory={{.HostConfig.Memory}} Swap={{.HostConfig.MemorySwap}} NanoCpus={{.HostConfig.NanoCpus}}'
expected output
Memory=134217728 Swap=134217728 NanoCpus=500000000
Memory 134217728 128 × 1024 × 1024 bytes = 128 MiB. This is the raw cgroup value Docker wrote.
MemorySwap 134217728 Equal to Memory → swap disabled. If this were 268435456 (256 MiB), swap would be 128 MiB.
NanoCpus 500000000 0.5 CPUs × 10⁹ = 500,000,000. Docker converts your --cpus value to nanoseconds of CPU time per second.
4
Watch live usage against the limit with docker stats
Open a second terminal and stream resource stats for both containers side-by-side. The MEM LIMIT column shows the ceiling you set.
shell — new terminal tab 
docker stats unlimited limited
expected output (streaming)
NAME CPU % MEM USAGE / LIMIT MEM % NET I/O
unlimited 0.0% 6.3MiB / 15.2GiB 0.04% ...
limited 0.0% 6.3MiB / 128MiB 4.93% ...
💡
Notice unlimited shows the full host RAM as its limit, while limited shows exactly 128MiB. Press Ctrl+C to stop streaming. Add --no-stream for a one-shot snapshot instead.
5
Apply a CPU limit and stress-test it
Run a CPU stress container capped at half a core and confirm it can't saturate the full host CPU — even with all threads pinned.
shell 
# Run stress with 4 CPU workers but cap the container at 0.5 cores
docker run -d --name cpu-test \
  --cpus "0.5" \
  progrium/stress \
  --cpu 4 --timeout 30s
shell — watch it in another tab 
docker stats cpu-test --no-stream
expected output
NAME CPU % MEM USAGE / LIMIT MEM %
cpu-test 50.0% 892KiB / 15.2GiB 0.00%
↑ capped at exactly 50% regardless of 4 stress workers
✅
Even though 4 threads are all trying to use 100% CPU, the container is hard-capped at 50% by the cgroup limit. The kernel throttles the threads proportionally.
6
Trigger an OOM kill — observe exit code 137
Run a container that deliberately allocates more memory than its limit. The kernel OOM killer will terminate it with exit code 137.
shell 
# Allow 64m RAM, then try to allocate 200m inside — will be OOM-killed
docker run --name oom-demo \
  --memory "64m" \
  --memory-swap "64m" \
  python:3.12-slim \
  python3 -c "x = ' ' * (200 * 1024 * 1024)"
shell 
# Check what happened
docker inspect oom-demo \
  --format 'ExitCode={{.State.ExitCode}} OOMKilled={{.State.OOMKilled}}'
expected output
ExitCode=137 OOMKilled=true
⚠️
Exit code 137 = 128 + 9 (SIGKILL). The OOMKilled flag is your definitive signal that the memory limit was breached. In production, alert on this flag rather than guessing from the exit code alone.
7
Reference — all resource limit flags
A complete reference of the flags you'll reach for most often.
--cpus "N"
Hard CPU ceiling. Fractional values allowed — 0.25, 1.5, 4. Enforced at all times, even when host cores are idle.
--cpu-shares N
Relative CPU weight. Default 1024. Only matters under contention — a container with 2048 shares gets twice the CPU time of one with 1024 when the host is busy.
--cpuset-cpus "0,1"
Pin to specific cores. Comma-separated or range — "0-3" pins to cores 0 through 3. Useful for NUMA-aware workloads.
--memory "Nm"
Hard RAM ceiling. Accepts b k m g suffixes. Processes that exceed this are OOM-killed immediately.
--memory-swap "Nm"
RAM + swap total. Set equal to --memory to disable swap. Set to -1 for unlimited swap. Default is 2× memory.
--memory-reservation "Nm"
Soft memory limit. Docker will try to reclaim memory down to this threshold when the host is under pressure, but won't kill the container.
--oom-kill-disable
Disable the OOM killer for this container. Dangerous — the host kernel may kill other processes instead. Only use with a hard --memory limit set.
--pids-limit N
Maximum number of processes the container can spawn. Prevents fork-bomb attacks. Set to -1 for unlimited.
8
Set limits in Docker Compose
In Compose files, resource limits live under deploy.resources — the same values, declaratively.
compose.yml 
services:
  web:
    image: nginx
    deploy:
      resources:
        limits:
          cpus: "0.5"       # hard ceiling
          memory: 128M   # hard RAM ceiling
        reservations:
          cpus: "0.1"       # guaranteed minimum
          memory: 64M    # soft floor
ℹ️
deploy.resources works with docker compose up (Compose v2). In the older Compose v1 format, limits were set directly under mem_limit and cpus at the service level.
9
Update limits on a running container
You don't need to stop and recreate a container to change its resource limits — docker update writes new cgroup values live.
shell 
# Raise the memory limit on a running container — no restart needed
docker update --memory "256m" --memory-swap "256m" limited

# Confirm the new value
docker inspect limited --format '{{.HostConfig.Memory}}'
expected output
268435456   ← 256 × 1024 × 1024 = 256 MiB
10
Clean up
Remove all containers created in this task.
shell 
docker rm -f unlimited limited cpu-test oom-demo
🔍 What happened under the hood
Step 01
📝
cgroup files written
Docker wrote limit values into cgroup v2 files under /sys/fs/cgroup/ on the host at container start.
Step 02
⚖️
Kernel enforces CPU
The CPU scheduler tracks nanoseconds of CPU time allocated. When the quota is exhausted, the container's threads are throttled until the next period.
Step 03
📊
Memory accounting
Every memory allocation inside the container increments a cgroup memory counter. Docker reads this counter to show usage in docker stats.
Step 04
💥
OOM kill triggered
When the counter would exceed the limit, the kernel's OOM killer sends SIGKILL to the offending process — Docker never intervenes.
Step 05
🏷️
OOMKilled flag set
Docker reads the cgroup's OOM event counter after the container exits and sets the OOMKilled field in the container's metadata.
Step 06
🔄
docker update live
docker update overwrites the cgroup files on the running container — the kernel picks up the new limits instantly, no restart.
