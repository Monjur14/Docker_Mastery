Task 06  ·  Beginner

Stop, Start & Restart
Containers

Manage the lifecycle of an existing container — stop, start, and restart — without destroying it or losing its filesystem state.

~8 minutes Docker installed A running container (nginx)
📖 Required concepts
🔄
The lifecycle isn't just Run
docker run always creates a brand new container. Once a container already exists, you don't run it again — you stop, start, or restart that exact same container, keeping its ID, filesystem changes, and configuration intact.
🛑
Stop ≠ Remove
Stopping a container halts its main process but keeps the container on disk — its filesystem, logs, and metadata all persist. This is very different from docker rm, which deletes the container entirely.
📡
SIGTERM, then SIGKILL
docker stop sends a SIGTERM signal, giving the process a chance to shut down gracefully (close connections, flush data). If it hasn't exited after a grace period (default 10s), Docker sends SIGKILL, force-terminating it.
▶️
Start resumes, it doesn't recreate
docker start resumes a stopped container using its original configuration — same image, ports, volumes, env vars — exactly as it was when created. You can't change those settings with start; that requires a new container.
🔁
Restart = stop + start
docker restart is a convenience command that stops the container (SIGTERM, then SIGKILL if needed) and immediately starts it again. Useful for applying config changes that require a fresh process without recreating the container.
⚙️
Restart policies
Beyond manual control, containers can be given a restart policy (no, on-failure, always, unless-stopped) so the daemon automatically restarts them on crash or on daemon/host reboot — no manual intervention needed.

Command What it does Requires
docker stop Gracefully halts a running container (SIGTERM → wait → SIGKILL). Container must be running
docker start Resumes a stopped container with its original config. Container must exist (any non-running state)
docker restart Stops then starts the container in one step. Works whether running or stopped
docker kill Immediately force-kills — sends SIGKILL right away, no grace period. Container must be running
🪜 Step-by-step guide
1
Start a long-running container
Run an nginx container in the background so it stays alive for us to manage. -d detaches it, --name gives it a memorable name instead of a random one.
shell 
docker run -d --name web nginx
2
Confirm it's running
List active containers — you should see web with status Up.
shell 
docker ps
expected output
CONTAINER ID IMAGE COMMAND STATUS PORTS NAMES
7f3a9c1b2e4d nginx "/docker-entrypoint.…" Up 5 seconds 80/tcp web
3
Stop the container
Send a graceful stop signal. Docker waits up to 10 seconds for the process to exit cleanly before force-killing it.
shell 
docker stop web
expected output
web
ℹ️
Docker echoes back the container name/ID once it's stopped. You can also target it by its container ID instead of name.
4
Verify it's stopped, not gone
docker ps alone only shows running containers, so it'll disappear. Use -a to confirm it still exists.
shell 
docker ps -a
expected output
CONTAINER ID IMAGE COMMAND STATUS NAMES
7f3a9c1b2e4d nginx "/docker-entrypoint.…" Exited (0) 4 seconds ago web
5
Start it again
Resume the exact same container — same ID, same config, same filesystem — no new container is created.
shell 
docker start web
💡
Run docker ps again — you'll see the same CONTAINER ID as step 2, now back to Up. Compare that to docker run, which would have created a totally new container.
6
Restart in one step
Instead of stop then start, use restart to do both at once — handy after changing a mounted config file that the app only reads at startup.
shell 
docker restart web
7
Set a custom stop timeout
If your app needs longer than 10 seconds to shut down gracefully (e.g. draining requests), extend the grace period with -t.
shell 
# Give the container 30 seconds to shut down gracefully
docker stop -t 30 web
⚠️
Need it dead right now, no grace period? Use docker kill web — it sends SIGKILL immediately, skipping the graceful shutdown entirely.
8
Apply an automatic restart policy
Restart policies handle crashes and reboots for you, without needing to manually run docker start. This only applies to new containers, so we'll spin up a second one.
shell 
docker run -d --name web2 --restart unless-stopped nginx
ℹ️
unless-stopped restarts the container on crash or daemon reboot, but respects a manual docker stop — it won't come back until you explicitly start it again. Use always if you want it to restart even after a manual stop, on daemon startup.
9
Clean up
Stop and remove both containers when you're done experimenting.
shell 
docker rm -f web web2
💡
-f forces removal even if the container is still running — it stops it first, then removes it, in one command.
🔍 What happens under the hood
Step 01
📡
SIGTERM sent
Daemon sends SIGTERM to the container's main process (PID 1 inside it).
Step 02
⏳
Grace period
Docker waits (default 10s) for the process to exit on its own.
Step 03
💀
SIGKILL if needed
If still running after the timeout, Docker force-kills it.
Step 04
💾
Filesystem preserved
The container's writable layer stays on disk — nothing is deleted.
Step 05
🔓
Ports released
Bound host ports are freed up while the container is stopped.
Step 06
▶️
Start reuses config
docker start re-attaches networking and re-launches with original settings.
🎯 What you learned
  • The difference between stopping a container and removing it entirely
  • How docker stop sends SIGTERM first, then SIGKILL after a grace period
  • That docker start resumes the same container with its original config — it doesn't create a new one
  • How docker restart combines stop and start in a single command
  • When to reach for docker kill instead, for an immediate, non-graceful stop
  • How restart policies (unless-stopped, always) automate recovery from crashes and reboots
