📖 Required concepts
🌐
User-defined bridge networks
When you create a custom network and attach containers to it, Docker gives each container a DNS name equal to its container name. The Node app can reach Mongo at mongodb://mongo:27017 — no IP address needed, ever. The default bridge network does not have this DNS feature.
💾
Named volumes for persistence
MongoDB stores its data at /data/db inside the container. Without a volume, all data is lost when the container is removed. A named volume lives independently of the container — delete and recreate the Mongo container as many times as you want, the data survives.
📦
Container startup order
Containers start in parallel by default — MongoDB must be ready to accept connections before the Node app tries to connect. You control this manually here by starting Mongo first and adding retry logic in the app. Task 34 (Healthchecks) handles this more elegantly.
🔌
Inter-container communication
Once on the same custom network, containers communicate over the internal virtual network. Mongo's port 27017 is never exposed to your host — only to other containers on app-network. This is the principle of least exposure.

package.json 
{
  "name": "node-mongo-app",
  "version": "1.0.0",
  "main": "server.js",
  "scripts": { "start": "node server.js" },
  "dependencies": {
    "express": "^4.18.2",
    "mongoose": "^8.0.0"
  }
}
server.js 
const express  = require('express');
const mongoose = require('mongoose');

const app  = express();
const PORT = process.env.PORT || 3000;
const MONGO_URI = process.env.MONGO_URI
  || 'mongodb://mongo:27017/mydb';

app.use(express.json());

// Simple Note model
const Note = mongoose.model('Note', {
  text: String,
  createdAt: { type: Date, default: Date.now }
});

// GET  /notes  — list all notes
app.get('/notes', async (req, res) => {
  const notes = await Note.find().sort({ createdAt: -1 });
  res.json(notes);
});

// POST /notes  — create a note
app.post('/notes', async (req, res) => {
  const note = await Note.create({ text: req.body.text });
  res.status(201).json(note);
});

// GET  /health — readiness probe
app.get('/health', (req, res) => {
  const state = mongoose.connection.readyState;
  res.json({ status: state === 1 ? 'ok' : 'degraded', mongo: state });
});

// Connect to Mongo with retry, then start server
async function start() {
  let retries = 5;
  while (retries) {
    try {
      await mongoose.connect(MONGO_URI);
      console.log('✅ Connected to MongoDB');
      break;
    } catch (err) {
      console.log(`⏳ Mongo not ready, retrying... (${retries} left)`);
      retries--;
      await new Promise(r => setTimeout(r, 3000));
    }
  }
  app.listen(PORT, () => console.log(`🚀 API on port ${PORT}`));
}

start();
Dockerfile 
FROM node:20-alpine
WORKDIR /app
COPY package*.json ./
RUN npm install --omit=dev
COPY . .
EXPOSE 3000
CMD ["node", "server.js"]
🪜 Step-by-step guide
Phase 1 Network & Volume infrastructure first
1
Create the custom bridge network
This network lets containers find each other by name. Create it before starting any containers.
shell 
docker network create app-network
expected output
7f3a1b9c2d4e5f6a8b0c1d2e3f4a5b6c7d8e9f0a1b2c3d4e5f6a7b8c9d0e1f
ℹ️
Inspect the new network at any time with docker network inspect app-network — it will list every container attached to it.
2
Create the named volume
The volume holds MongoDB's data directory. Creating it explicitly (rather than letting Docker auto-create it) is best practice — it makes the volume visible in docker volume ls even before any container uses it.
shell 
docker volume create mongo-data
shell 
docker volume inspect mongo-data
inspect output (truncated)
[
{
"Name": "mongo-data",
"Driver": "local",
"Mountpoint": "/var/lib/docker/volumes/mongo-data/_data",
"Scope": "local"
}
]
Phase 2 MongoDB Container start the database first
3
Start the MongoDB container
Pull and run the official Mongo image, connecting it to the volume and the network. Mongo's port is intentionally not published to the host — it's only reachable by containers on app-network.
shell 
docker run -d \
  --name mongo \
  --network app-network \
  -v mongo-data:/data/db \
  -e MONGO_INITDB_DATABASE=mydb \
  mongo:7
💡
No -p flag on Mongo — this is by design. Exposing the database port to the host is a security risk. The Node app reaches it over the internal Docker network instead.
4
Wait for MongoDB to be ready
MongoDB takes a few seconds to initialise its storage engine. Watch the logs until you see the Waiting for connections message before starting the Node app.
shell 
docker logs -f mongo
watch for this line — then Ctrl+C
…
{"t":{"$date":"…"},"s":"I", "c":"NETWORK", "msg":"Waiting for connections","attr":{"port":27017}}
✅
Once you see "Waiting for connections", press Ctrl+C to stop following logs (the container keeps running). MongoDB is ready.
Phase 3 Node.js API Container build and run the app
5
Build the Node.js image
From inside your node-mongo-app/ folder, build the app image.
shell 
# Run from inside node-mongo-app/
docker build -t node-mongo-app .
6
Run the Node.js container on the same network
Connect it to app-network so it can reach the mongo container by name. The MONGO_URI env var points to mongo — the container name, resolved by Docker DNS.
shell 
docker run -d \
  --name node-app \
  --network app-network \
  -p 3000:3000 \
  -e MONGO_URI=mongodb://mongo:27017/mydb \
  node-mongo-app
ℹ️
The hostname mongo in the URI resolves to the MongoDB container's internal IP because both containers share app-network. This only works on user-defined networks — not the default bridge.
7
Confirm both containers are running
Check that both services are up and the port mapping is active.
shell 
docker ps
expected output
CONTAINER ID IMAGE COMMAND STATUS PORTS NAMES
b1c2d3e4f5a6 node-mongo-app "node server.js" Up 12 seconds 0.0.0.0:3000->3000/tcp node-app
a9b8c7d6e5f4 mongo:7 "docker-entrypoint.s…" Up 45 seconds mongo
Phase 4 Test & Verify hit the api and confirm persistence
8
Check the health endpoint
The /health route reports whether the app connected to MongoDB. Wait a few seconds after starting for the retry logic to succeed.
shell 
curl http://localhost:3000/health
expected output
{"status":"ok","mongo":1}
⚠️
If you see "degraded", check the Node app logs: docker logs node-app. The retry loop runs up to 5 times at 3-second intervals — wait ~15 seconds and try again.
9
Create and retrieve notes
Exercise the full API — write a record to MongoDB via the Node app, then read it back.
shell 
# Create a note
curl -X POST http://localhost:3000/notes \
  -H "Content-Type: application/json" \
  -d '{"text":"Hello from Docker!"}'

# List all notes
curl http://localhost:3000/notes
expected output
# POST response
{"_id":"6570a1…","text":"Hello from Docker!","createdAt":"2024-12-06T…","__v":0}

# GET response
[{"_id":"6570a1…","text":"Hello from Docker!","createdAt":"2024-12-06T…","__v":0}]
10
Prove data survives container deletion
This is the key test for named volumes. Delete the Mongo container, recreate it from scratch, and verify your notes are still there.
shell 
# 1. Stop and destroy the Mongo container entirely
docker rm -f mongo

# 2. Recreate it (same command as step 3)
docker run -d \
  --name mongo \
  --network app-network \
  -v mongo-data:/data/db \
  -e MONGO_INITDB_DATABASE=mydb \
  mongo:7

# 3. Wait ~5 seconds, then query the API
curl http://localhost:3000/notes
expected — data still there after container restart
[{"_id":"6570a1…","text":"Hello from Docker!","createdAt":"2024-12-06T…","__v":0}]
✅
Your note survived because it lives in mongo-data, which is managed by Docker independently of any container. The container was recreated from scratch — the volume was not touched.
11
Inspect the network — see both containers
Confirm Docker's internal DNS by inspecting the network and seeing both containers registered with their names.
shell 
docker network inspect app-network \
  --format '{{range .Containers}}{{.Name}} → {{.IPv4Address}}{{"\n"}}{{end}}'
expected output
node-app → 172.18.0.3/16
mongo → 172.18.0.2/16
Phase 5 Teardown clean up everything
12
Stop and remove all resources
Remove containers, network, and (optionally) the volume. Run these in order — you can't remove a network while containers are using it.
shell 
# 1. Stop and remove both containers
docker rm -f node-app mongo

# 2. Remove the network
docker network rm app-network

# 3. Remove the volume (DESTROYS ALL DATA)
docker volume rm mongo-data

# 4. Remove the app image
docker rmi node-mongo-app
⚠️
Step 3 permanently deletes your data. Skip docker volume rm if you want to keep the database contents for Task 28 (Docker Compose), which reuses the same volume name.
📋 All commands at a glance
Complete run sequence
docker network create app-network Create the shared bridge network with automatic DNS
docker volume create mongo-data Create the persistent volume for MongoDB data
docker run -d --name mongo --network app-network -v mongo-data:/data/db mongo:7 Start MongoDB — internal only, no host port published
docker logs -f mongo Wait for "Waiting for connections" before proceeding
docker build -t node-mongo-app . Build the Node.js API image from the Dockerfile
docker run -d --name node-app --network app-network -p 3000:3000 -e MONGO_URI=mongodb://mongo:27017/mydb node-mongo-app Start the API, connecting to Mongo by container name
curl localhost:3000/health Verify both containers are connected
🔍 What happened under the hood
Step 01
🌐
Virtual network
Docker created a virtual L2 network with its own subnet and a built-in DNS resolver.
Step 02
💾
Volume allocated
A directory was allocated on the host inside Docker's storage area, ready to be mounted.
Step 03
🗄️
Mongo registered
The mongo container joined the network and its name was registered with the DNS resolver.
Step 04
🔗
DNS resolution
When node-app resolved mongo, Docker's resolver returned the Mongo container's internal IP.
Step 05
🔌
TCP handshake
Mongoose opened a persistent connection to Mongo over the virtual network interface.
Step 06
📝
Data persisted
Documents written to MongoDB were stored in mongo-data, independent of the container lifecycle.
🎯 What you learned
  • How to create a custom bridge network and why container-name DNS only works on user-defined networks
  • How to wire two containers together using --network and reference one from the other by name
  • How to attach a named volume to MongoDB and prove data survives container deletion
  • Why Mongo's port should not be published to the host — only the API container needs access
  • How to add retry logic in an app to handle the database not being ready immediately
  • The correct teardown order: containers → network → volume → images
