Task 17  ·  Intermediate

Tag Images with
Semantic Versions

Apply specific version tags like myapp:1.0.0 instead of relying on the default latest tag — so every image is traceable, rollbacks are safe, and teammates always know what they're running.

~15 minutes Completed Tasks 14 – 16 Built image from Task 15
📖 Required concepts
🏷️
What is a Tag?
A tag is a human-readable pointer to a specific image ID. The full name of an image is name:tag. If you omit the tag, Docker silently appends :latest — which sounds helpful but is a common source of confusion in teams.
⚠️
The Problem with latest
latest is just a tag — it has no special meaning to Docker. It doesn't automatically update. Two machines pulling myapp:latest at different times may get completely different images without any warning.
🔢
Semantic Versioning (SemVer)
SemVer uses the format MAJOR.MINOR.PATCH. Increment PATCH for bug fixes, MINOR for new backwards-compatible features, and MAJOR for breaking changes. Most of the ecosystem — npm, PyPI, Docker Hub — follows this convention.
🪝
One Image, Multiple Tags
A single image ID can carry many tags at once. It's common to tag a release as both 1.2.3 and 1.2 and latest simultaneously — they all point at the same image, giving callers flexibility in how precisely they pin their dependency.
🔁
Tagging vs. Rebuilding
docker tag adds a new name to an already-built image without copying or rebuilding anything. It's instant and costs no disk space — both names share the same underlying layers.
📌
Immutable Releases
Once you push myapp:1.0.0, treat it as immutable — never rebuild and re-push under the same version tag. If something changes, that's 1.0.1. Immutability is what makes rollbacks reliable.
🔢 Anatomy of a version tag
2.4.1
Major
Breaking change. Callers must update their code to keep working.
Minor
New feature, backwards compatible. Existing callers are unaffected.
Patch
Bug fix only. Safe to update without reading the changelog.
🔬 Command anatomy
docker build -t myapp : 1.0.0 .
docker

The CLI binary.

build -t

Build an image and assign it a tag. -t is short for --tag. Can be repeated for multiple tags in one build.

name

The image repository name. Locally this is freeform; on Docker Hub it becomes username/myapp.

:tag

The version label — anything after the colon. Omit it and Docker silently uses :latest.

.

The build context — the directory Docker packages and sends to the daemon.

🪜 Step-by-step guide
1
Build your first versioned image
Use -t to assign an explicit version tag at build time. Start at 1.0.0 for a new project.
shell 
docker build -t myapp:1.0.0 .
ℹ️
This image gets exactly one tag: myapp:1.0.0. Unlike omitting the tag, Docker will NOT automatically create myapp:latest — you'd have to add it explicitly.
2
Build with multiple tags at once
Repeat -t to assign several tags in a single build. All three point at the same image ID — nothing is duplicated.
shell 
docker build \
  -t myapp:1.0.0 \
  -t myapp:1.0 \
  -t myapp:latest \
  .
💡
The floating tags (1.0, latest) let users opt into automatic minor/patch updates while pinned consumers can lock to 1.0.0 forever. This is how the official Node.js and Python images are tagged on Docker Hub.
3
Verify all tags point to the same image
List your images filtered by name. Notice the IMAGE ID is identical across all three tags — they're the same image bytes, just different names.
shell 
docker images myapp
expected output
REPOSITORY TAG IMAGE ID CREATED SIZE
myapp 1.0.0 a3f72b9c1d4e 10 seconds ago 145MB
myapp 1.0 a3f72b9c1d4e 10 seconds ago 145MB
myapp latest a3f72b9c1d4e 10 seconds ago 145MB
✅
All three rows share the same IMAGE ID (a3f72b9c1d4e). Three names, one set of layers on disk.
4
Add a tag to an existing image with docker tag
If you forgot a tag, or want to add one after the build, use docker tag. No rebuild, no copy — it's instant.
shell 
# Add a 'stable' alias to whatever 1.0.0 points at
docker tag myapp:1.0.0 myapp:stable
ℹ️
docker tag SOURCE TARGET — the source can be a name:tag or a raw image ID. Both work.
5
Simulate a patch release
Make a small change to index.js (e.g. fix a typo), then rebuild under 1.0.1. Move the floating tags to the new release.
shell 
# Build the patch release — pin it exactly
docker build -t myapp:1.0.1 .

# Move the floating tags to the new release
docker tag myapp:1.0.1 myapp:1.0
docker tag myapp:1.0.1 myapp:latest
⚠️
Notice myapp:1.0.0 is left untouched. Anyone pinned to that exact tag still has the original image. Only the floating tags (1.0, latest) moved forward.
6
Inspect your tag history
Now you can see the version history at a glance. Two image IDs, four tags — the old 1.0.0 is preserved and the floating tags point at the latest patch.
shell 
docker images myapp
expected output
REPOSITORY TAG IMAGE ID CREATED SIZE
myapp 1.0.1 c9e14b2f8a7d 30 seconds ago 145MB
myapp 1.0 c9e14b2f8a7d 30 seconds ago 145MB
myapp latest c9e14b2f8a7d 30 seconds ago 145MB
myapp 1.0.0 a3f72b9c1d4e 5 minutes ago 145MB
7
Simulate a rollback
Because 1.0.0 still exists locally, rolling back is just a docker run away — no rebuild required.
shell 
# Something broke in 1.0.1 — roll back instantly
docker run --rm myapp:1.0.0
💡
This is the core value of versioned tags. With latest only, a rollback means a rebuild. With pinned versions, it's deterministic and instant.
8
Remove a specific tag
Removing a tag with docker rmi only deletes that name — not the underlying image — as long as another tag still points to the same ID.
shell 
# Remove the 'stable' alias — image layers are NOT deleted
docker rmi myapp:stable
expected output
Untagged: myapp:stable
# No "Deleted:" lines — layers still referenced by myapp:1.0.0
📅 A release timeline in tags
1.0.0 Initial release
First production build. Also tagged 1.0 and latest.
1.0.1 Bug fix
Fixed a crash on empty input. 1.0 and latest moved here. 1.0.0 preserved.
1.1.0 New feature
Added a /health endpoint. Backwards compatible — only MINOR bumped. latest moved here.
2.0.0 Breaking change — you are here
Restructured the API response format. MAJOR bumped. The 1.x images are still available for anyone not ready to migrate.
📋 Common tagging conventions
Tag When to use Notes
myapp:1.0.0
Every release
Pinned, immutable. Never re-push over an existing version tag.
myapp:1.0
Floating minor
Points to latest patch in the 1.0.x line. Move forward on each patch.
myapp:1
Floating major
Points to latest in 1.x.x. Move forward on each minor/patch.
myapp:latest
Convenience alias
Points to the newest stable release. Never rely on it in production.
myapp:main
CI/CD branch builds
Tracks the main branch tip. Useful for staging, not production.
myapp:sha-a1b2c3d
CI/CD commit pinning
Fully immutable — tied to a specific Git commit. Best for automated deployments.
🔍 What happened under the hood
Step 01
🏗️
Image built
Layers assembled and assigned a content-addressed SHA256 digest.
Step 02
🏷️
Tag stored
Docker's local store maps myapp:1.0.0 → the image ID.
Step 03
🔗
Alias added
docker tag adds another pointer — no copy, no bytes moved.
Step 04
🔄
Tag moved
Re-tagging latest updates the pointer; the old image remains.
Step 05
⏪
Rollback
Old pinned tag still resolves — layers already on disk, start is instant.
Step 06
🗑️
Untag only
rmi on one tag removes the name; layers survive while any other tag references them.
