Task 04  ·  Beginner

List & Inspect
Containers

View active and stopped containers with docker ps, then dig into a container's full configuration, network, and mount metadata with docker inspect.

~7 minutes Docker installed Completed Tasks 1–3
📖 Required concepts
📋
docker ps vs. docker ps -a
docker ps shows only running containers. Add -a (--all) to see every container regardless of state — including ones that exited or were never started.
🔄
Container states
Containers move through Created → Running → (Paused) → Exited, and can be Removed entirely. The STATUS column in docker ps -a always tells you which state a container is currently in.
🎛️
Filtering and formatting output
--filter narrows results by status, name, or image. --format lets you reshape the table using Go template syntax — useful for scripting.
🔬
docker inspect
docker inspect returns a full JSON document describing a container's config, mounts, network settings, resource limits, and state — everything Docker knows about it.
🎯
Extracting a single field
Rather than reading the whole JSON blob, --format (or piping through jq) lets you pull out exactly one value, like an IP address, straight into a script variable.
🆔
Referencing containers
Most container commands accept either the full ID, a short ID (first ~12 characters, as long as it's unique), or the container's name — whichever is most convenient.
🔬 Command anatomy
docker inspect --format '{{.State.Status}}' web
docker

The CLI binary — how you talk to the Docker daemon from your terminal.

inspect

Returns low-level, detailed JSON metadata about a container (or image, network, volume).

--format

Applies a Go template to the JSON so only the field(s) you care about are printed.

'{{.State.Status}}'

The template itself — dot-notation path into the JSON tree, here pointing at State.Status.

web

The target container — referenced here by name, but ID (full or short) works identically.

🪜 Step-by-step guide
1
Set up a few containers to inspect
Start one running container and let one exit immediately, so you have a mix of states to explore.
shell 
# A container that keeps running
docker run -d -p 8080:80 --name web nginx

# A container that exits right away
docker run --name hello hello-world
2
List running containers
Plain docker ps shows only what's currently active.
shell 
docker ps
expected output
CONTAINER ID IMAGE COMMAND STATUS PORTS NAMES
a7f3c9d21e4b nginx "/docker-entrypoint.…" Up 2 minutes 0.0.0.0:8080->80/tcp web
ℹ️
Notice hello doesn't show up here — it already exited, so it's invisible to plain docker ps.
3
List all containers, including stopped ones
Add -a to see the full history of containers on this machine.
shell 
docker ps -a
expected output
CONTAINER ID IMAGE COMMAND STATUS NAMES
a7f3c9d21e4b nginx "/docker-entrypoint.…" Up 3 minutes web
b2e8d4f10a3c hello-world "/hello" Exited (0) 90 seconds ago hello
4
Filter and reformat the list
Combine --filter and --format to get exactly the view you need.
shell 
# Only exited containers
docker ps -a --filter status=exited

# Custom columns: ID, image, status
docker ps -a --format "table {{.ID}}\t{{.Image}}\t{{.Status}}"
5
Inspect a container's full metadata
docker inspect dumps everything Docker knows about the container as JSON.
shell 
docker inspect web
expected output (truncated)
[
  {
    "Id": "a7f3c9d21e4b8f0c...",
    "Name": "/web",
    "State": { "Status": "running", ... },
    "NetworkSettings": { "IPAddress": "172.17.0.2" ... },
    "Mounts": [ ... ],
    // ... config, host config, resource limits, and more
  }
]
ℹ️
The output is always an array, even for a single container — that's why it starts with [. Pass multiple names/IDs to inspect several at once.
6
Extract a single field
Use --format to pull just the value you need — no JSON parsing required.
shell 
# Get the container's internal IP address
docker inspect --format '{{.NetworkSettings.IPAddress}}' web

# Get just the running state
docker inspect --format '{{.State.Status}}' web

# Get the exit code of a stopped container
docker inspect --format '{{.State.ExitCode}}' hello
expected output
172.17.0.2
running
0
💡
Prefer JSON parsing over templates? Pipe into jq: docker inspect web | jq '.[0].State.Status'
7
Clean up
Remove both practice containers now that you're done inspecting them.
shell 
docker rm -f web hello
🔍 What happened under the hood
Step 01
🗂️
Daemon tracks state
The daemon maintains a live record of every container's metadata, whether running or stopped.
Step 02
📋
ps queries the list
docker ps asks the daemon for a filtered summary table of that tracked state.
Step 03
🔬
inspect queries one object
docker inspect asks for the complete, unfiltered record of a single container.
Step 04
🧾
JSON returned
The daemon serializes its internal object into JSON and sends it back to the CLI.
Step 05
🎯
Template applied
If --format was given, the CLI walks the JSON tree and prints only the requested field.
Step 06
🖨️
Printed to terminal
The final table or value is rendered to stdout for you to read or pipe elsewhere.
🎯 What you learned
  • The difference between docker ps (running only) and docker ps -a (all states)
  • The full container lifecycle: Created → Running → Paused → Exited
  • How to filter and reformat docker ps output with --filter and --format
  • How docker inspect returns full JSON metadata for a container
  • How to extract a single field (IP address, status, exit code) with Go template syntax
  • Containers can be referenced by full ID, short ID, or name
