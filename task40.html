Task 40  ·  Advanced

Docker Daemon
Security Tuning

Harden your Docker setup by enabling rootless mode — eliminating daemon root access entirely — and by writing a custom Seccomp profile to restrict the Linux syscalls your containers are allowed to make.

~35 minutes Tasks 35 & 36 complete Linux host recommended
⚠️The default Docker attack surface

Out of the box, Docker runs with significant privileges. A container escape, a misconfigured mount, or a compromised image could give an attacker root access to your host. This task closes those gaps layer by layer.

Attack surface — default Docker install, worst-first
Root daemon
dockerd runs as root. The Docker socket (/var/run/docker.sock) grants full root equivalent to anyone who can write to it.
→ Fixed by: Rootless mode (Part A)
Unrestricted syscalls
Containers can call hundreds of Linux kernel syscalls by default — many of which are not needed and could be weaponised for privilege escalation.
→ Fixed by: Custom Seccomp profile (Part B)
Excess capabilities
Containers run with a large set of Linux capabilities (CAP_NET_ADMIN, CAP_SYS_PTRACE, etc.) most apps don't need. Each is a potential escalation vector.
→ Mitigated by: --cap-drop=ALL --cap-add=... (covered in Task 35)
Running as root in container
If a process escapes the container namespace, root inside = root outside on a default install.
→ Mitigated by: Non-root USER in Dockerfile (Task 35)
Mandatory access control
Without AppArmor or SELinux profiles, no MAC policy constrains what a container process can access on the host filesystem.
→ Covered in Part C: AppArmor profiles
📖Required concepts
👤
Rootless Docker
In rootless mode the entire Docker daemon — including the container runtimes — runs as a regular unprivileged user. Even if an attacker escapes a container and compromises the daemon, they only have your user's permissions, not root. The Docker socket is private to your user account.
🔒
Seccomp (Secure Computing Mode)
Seccomp is a Linux kernel feature that filters syscalls. A Seccomp profile is a JSON allowlist (or blocklist) of syscall names. When a container process attempts a syscall not on the list, the kernel kills it with SIGSYS. Docker ships a default profile that blocks ~44 dangerous syscalls; you can write tighter custom profiles.
🛡️
AppArmor
AppArmor is a Linux mandatory access control (MAC) system that restricts what files, capabilities, and network operations a process can access — regardless of its UID or privileges. Docker loads a default AppArmor profile (docker-default) for containers on systems where AppArmor is available.
📜
Defence in depth
No single hardening measure is enough. The goal is layered defences: rootless mode limits daemon blast radius, Seccomp limits syscall surface, AppArmor limits filesystem access, non-root USER (Task 35) limits container privilege, and secrets management (Task 36) limits credential exposure. Each layer makes exploitation harder.
⚙️
daemon.json
The Docker daemon's configuration file at /etc/docker/daemon.json (root mode) or ~/.config/docker/daemon.json (rootless mode) controls daemon-wide defaults — including the default Seccomp profile, logging driver, storage driver, and more.
🧪
Testing without breaking things
Overly aggressive Seccomp profiles silently break applications. The workflow is: audit (run with --security-opt seccomp=unconfined and trace syscalls), build a profile allowing only what was observed, then enforce. Start from Docker's default profile — don't write from scratch.
Root mode (default)
dockerd runs as root
Docker socket at /var/run/docker.sock — world-readable by the docker group. Equivalent to passwordless sudo for group members. Container escape → host root.
Rootless mode
dockerd runs as your user
Socket at $XDG_RUNTIME_DIR/docker.sock. Only accessible to your UID. Daemon compromised → only user-level access. Recommended for all developer workstations and single-user servers.
🪜Step-by-step guide
A
Part A — Rootless Mode
1
Check prerequisites
Rootless mode requires newuidmap and newgidmap to be installed and your user to have subordinate UID/GID ranges configured in /etc/subuid and /etc/subgid.
shell
# Install prerequisites (Ubuntu/Debian)
sudo apt-get install -y uidmap dbus-user-session

# Verify your user has subuid/subgid ranges
grep $(whoami) /etc/subuid   # should show: username:100000:65536
grep $(whoami) /etc/subgid

# If missing, add them
sudo usermod --add-subuids 100000-165535 \
              --add-subgids 100000-165535 $(whoami)
expected /etc/subuid output
alice:100000:65536
2
Install rootless Docker
Run the official rootless setup script as your regular user — not as root. It installs a user-scoped daemon and sets up the socket.
shell — as regular user, not root
dockerd-rootless-setuptool.sh install
expected output (key lines)
[INFO] Creating /home/alice/.config/systemd/user/docker.service
[INFO] Starting systemd service docker.service
...
[INFO] Installed docker.service successfully.

Make sure the following environment variables are set (or add to ~/.bashrc):
export PATH=/usr/bin:$PATH
export DOCKER_HOST=unix:///run/user/1000/docker.sock
ℹ️
If dockerd-rootless-setuptool.sh isn't found, install it first with sudo apt-get install docker-ce-rootless-extras or fetch the convenience script from get.docker.com.
3
Configure your shell and start the daemon
Export the environment variables the installer printed, then enable the user-scoped systemd service so the daemon starts automatically on login.
shell
# Add to ~/.bashrc (or ~/.zshrc)
echo 'export DOCKER_HOST=unix:///run/user/$(id -u)/docker.sock' >> ~/.bashrc
source ~/.bashrc

# Enable the user daemon to start on login
systemctl --user enable docker
systemctl --user start docker

# Keep daemon running even when not logged in (servers)
sudo loginctl enable-linger $(whoami)
4
Verify rootless mode is active
Confirm that docker info reports rootless mode and that the socket path is under your user's runtime directory — not /var/run.
shell
docker info --format 'Security Options: {{.SecurityOptions}}'
docker info --format 'Rootless: {{.ClientInfo.Context}}'

# Verify socket is user-scoped
echo $DOCKER_HOST

# Run a container — should work exactly as before
docker run --rm hello-world
expected docker info output
Security Options: [name=rootless name=seccomp,profile=builtin name=apparmor]
DOCKER_HOST=unix:///run/user/1000/docker.sock
✅
The daemon is now running under your UID. The system-wide Docker socket no longer exists — or if root Docker is also installed, it's irrelevant to your user session. A container escape gives an attacker only your user's permissions.
B
Part B — Custom Seccomp Profile
5
Understand what a Seccomp profile looks like
A Seccomp profile is a JSON document with a defaultAction and a list of syscalls that deviate from that default. The two common strategies: allowlist (default: KILL, allow specific calls) or blocklist (default: ALLOW, block specific calls). Allowlists are more secure.
Syscall What it does Default profile
ptrace Attach a debugger to any process — can be used to dump memory or inject code into other containers. BLOCK
kexec_load Load a new kernel — trivial host escape if allowed. BLOCK
mount Mount filesystems — blocked by default, re-bind-mounting host paths is a common escape vector. BLOCK
open, read, write Basic I/O — every app needs these. ALLOW
clone Create threads and processes — needed for most runtimes. Restricted to non-namespace-creating flags. ALLOW
6
Download Docker's default profile as a starting point
Never write a Seccomp profile from scratch — start from Docker's well-tested default and tighten from there. The default already blocks ~44 dangerous syscalls.
shell
# Download the official default Seccomp profile
curl -o seccomp-default.json \
  https://raw.githubusercontent.com/moby/moby/master/profiles/seccomp/default.json

# Count how many syscalls it covers
jq '.syscalls | length' seccomp-default.json
expected output
355 # ~355 entries covering ~340 allowed + ~44 explicitly blocked
7
Write a tight custom Seccomp profile for a Node.js app
This profile uses allowlist mode: the default action kills the process with SIGSYS, and only the syscalls a typical Node.js HTTP server actually needs are permitted. Save this as seccomp-node.json.
seccomp-node.json
{
  "defaultAction": "SCMP_ACT_ERRNO",   // deny anything not listed

  "syscalls": [
    {
      "names": [
        // process lifecycle
        "execve", "exit", "exit_group", "wait4", "waitid",
        // memory management
        "mmap", "mprotect", "munmap", "brk", "madvise",
        // file I/O
        "open", "openat", "read", "write", "close",
        "stat", "fstat", "lstat", "lseek", "pread64",
        "readlink", "getcwd", "chdir",
        // directory listing
        "getdents", "getdents64",
        // networking
        "socket", "connect", "accept", "accept4",
        "bind", "listen", "getsockname", "getpeername",
        "setsockopt", "getsockopt", "sendto", "recvfrom",
        "sendmsg", "recvmsg", "shutdown",
        // threads and signals
        "clone", "futex", "set_robust_list", "get_robust_list",
        "sigaltstack", "rt_sigaction", "rt_sigprocmask",
        "rt_sigreturn", "kill",
        // identity and time
        "getuid", "getgid", "geteuid", "getegid",
        "getpid", "gettid", "clock_gettime", "nanosleep",
        // polling / epoll (Node.js event loop)
        "epoll_create", "epoll_create1", "epoll_ctl",
        "epoll_wait", "epoll_pwait", "poll", "ppoll",
        "select", "pselect6",
        // misc required by Node.js / V8
        "ioctl", "fcntl", "pipe", "pipe2", "eventfd2",
        "prctl", "arch_prctl", "set_tid_address",
        "uname", "sysinfo", "tgkill"
      ],
      "action": "SCMP_ACT_ALLOW"
    }
  ]
}
⚡
This list is illustrative, not exhaustive. Real apps often need additional syscalls discovered through testing. Use the audit workflow in Step 9 to find missing syscalls before enforcing this profile in production.
8
Apply the Seccomp profile to a container
Pass the profile JSON to --security-opt. Docker loads it and hands it to the kernel before the container process starts.
shell — run with custom Seccomp profile
docker run --rm \
  --security-opt seccomp=./seccomp-node.json \
  -p 3000:3000 \
  my-node-app
docker-compose.yml — security_opt block
services:
  app:
    image: my-node-app
    security_opt:
      - seccomp:./seccomp-node.json
    ports:
      - "3000:3000"
9
Audit: find what syscalls your app actually uses
Before enforcing a tight profile, run with strace inside the container to capture every syscall the app makes under realistic load, then build your allowlist from that output.
shell — audit mode (unconfined + strace)
# Run completely unconfined so nothing is blocked
docker run -it --rm \
  --security-opt seccomp=unconfined \
  --cap-add SYS_PTRACE \
  my-node-app sh

# Inside the container — trace the app's syscalls
apk add strace
strace -f -e trace=all -o /tmp/syscalls.txt node app.js &

# Exercise your app (run requests, trigger all code paths)
# Then extract the unique syscall names used
grep -oP '^[a-z_0-9]+' /tmp/syscalls.txt | sort -u
💡
Alternatively, the Inspektor Gadget project and tracee (by Aqua Security) can record syscall usage from outside the container without needing strace inside it — useful for production audits.
10
Verify the profile blocks a forbidden syscall
Confirm enforcement is working by trying to run a syscall your profile blocks — like ptrace. Under the custom profile it should fail; under unconfined it would succeed.
shell
# With our custom profile — ptrace should be blocked
docker run --rm \
  --security-opt seccomp=./seccomp-node.json \
  alpine strace -e trace=ptrace ls
expected output — syscall denied
strace: ptrace(PTRACE_TRACEME, ...): Operation not permitted
strace: attach: ptrace(PTRACE_SEIZE, 1, ...): Operation not permitted
✅
The kernel returned EPERM (Operation not permitted) — the syscall was blocked by Seccomp before it could execute. An attacker who gains code execution inside this container cannot use ptrace to inspect or manipulate other processes.
C
Part C — AppArmor Profile
11
Check AppArmor availability and Docker's default profile
AppArmor is available on Ubuntu/Debian by default. Docker automatically loads its docker-default profile for every container when AppArmor is present.
shell
# Check if AppArmor is enabled
aa-status --enabled && echo "AppArmor is active"

# See the profiles currently loaded
aa-status | grep docker

# Confirm a container uses the default profile
docker run --rm ubuntu cat /proc/1/attr/current
expected output
AppArmor is active
docker-default (enforce)
docker-default (enforce) ← /proc/1/attr/current
12
Write and load a custom AppArmor profile
Create a profile that tightens the default — explicitly denying write access to /etc/ and disabling certain capabilities. Save it to /etc/apparmor.d/docker-node-app.
/etc/apparmor.d/docker-node-app
#include <tunables/global>

profile docker-node-app flags=(attach_disconnected,mediate_deleted) {
  # Include Docker's base rules
  #include <abstractions/base>

  # Allow read access to most of the filesystem
  /** r,

  # Allow write only to specific directories
  /tmp/** rw,
  /run/secrets/** r,   # Docker secrets (read-only)
  /app/** rw,           # app working directory

  # Explicitly deny writes to sensitive locations
  deny /etc/** w,
  deny /bin/** w,
  deny /sbin/** w,
  deny /usr/bin/** w,
  deny /proc/** w,
  deny /sys/** w,

  # Allow networking
  network inet tcp,
  network inet udp,
  network inet6 tcp,
  network inet6 udp,

  # Deny dangerous capabilities
  deny capability sys_admin,
  deny capability sys_ptrace,
  deny capability net_admin,
}
shell — load and enforce the profile
# Parse and load the profile into the kernel
sudo apparmor_parser -r -W /etc/apparmor.d/docker-node-app

# Confirm it's loaded
sudo aa-status | grep docker-node-app
expected output
docker-node-app (enforce)
13
Run a container with the custom AppArmor profile
Apply both Seccomp and AppArmor profiles together for layered protection. Then verify that writing to /etc/ is denied.
shell
# Run with both profiles enforced
docker run --rm -it \
  --security-opt apparmor=docker-node-app \
  --security-opt seccomp=./seccomp-node.json \
  ubuntu bash

# Inside — try to write to /etc/ (should be denied)
echo "hacked" >> /etc/passwd
expected output — write denied
bash: /etc/passwd: Permission denied
🛡️
AppArmor denied the write at the MAC layer — regardless of file permissions or the process's UID. Even running as root inside the container, the AppArmor policy overrides traditional DAC permissions. This is the power of mandatory access control.
14
Set Seccomp profile as the daemon-wide default
Instead of passing --security-opt to every container, configure the daemon to use your profile by default for all containers.
~/.config/docker/daemon.json (rootless) or /etc/docker/daemon.json (root)
{
  "seccomp-profile": "/home/alice/seccomp-node.json",
  "log-driver": "json-file",
  "log-opts": {
    "max-size": "10m",
    "max-file": "3"
  }
}
shell — reload daemon config
# Rootless mode
systemctl --user restart docker

# Root mode
sudo systemctl restart docker
⚠️
Restarting the daemon stops all running containers. In production, apply configuration changes during a maintenance window. Use dockerd --validate to check daemon.json syntax before restarting.
🔍What happened under the hood
Part A
👤
Rootless daemon
dockerd runs as UID 1000. Socket is at /run/user/1000/docker.sock — inaccessible to other users.
Part A
🗂️
User namespace
Root inside the container maps to an unprivileged UID on the host via /etc/subuid. Container escape → non-root on host.
Part B
🔒
Syscall filter loaded
The kernel BPF program enforcing the Seccomp profile runs on every syscall before it reaches the kernel handler.
Part B
🚫
Denied syscalls
Blocked calls return EPERM immediately — the kernel never executes them. Zero performance cost for allowed calls.
Part C
🛡️
MAC enforcement
AppArmor policy checked on every file access, network call, and capability request — independent of UID.
All
🧅
Layered defence
An attacker must bypass rootless namespacing, Seccomp filter, AppArmor MAC, and non-root USER — all independently.
