Task 07  ·  Beginner

Container Removal
& Cleanup

Stop a container, delete it, and reclaim disk space by pruning dangling images, unused volumes, and other leftover resources. Keep your Docker environment lean.

~10 minutes Docker installed Terminal / Shell
📖 Required concepts
🗑️
Stop before you remove
You cannot remove a running container with docker rm — it will error. You must stop it first, or pass -f (force) to stop and remove in one go.
🪣
What "dangling" means
A dangling image is one with no tag and no container referencing it — typically left behind when you rebuild an image with the same tag. It shows up as <none>:<none> in docker images. They're safe to delete.
🧹
Prune vs. remove
docker rm targets specific containers by name or ID. prune commands bulk-delete all resources of a given type that are safe to remove — faster for cleanups, but less surgical.
💽
Volumes are independent
Volumes are not deleted when a container is removed — by design. Data persists even after the container is gone. You have to explicitly prune volumes, or they accumulate indefinitely on disk.
☢️
docker system prune is aggressive
docker system prune removes all stopped containers, dangling images, unused networks, and build cache in one shot. Adding -a also deletes all unused images, not just dangling ones — use with care on dev machines.
📊
Check disk usage first
Before pruning blindly, run docker system df to see exactly how much space containers, images, volumes, and build cache are consuming — and how much is reclaimable.
🗺️ What can be cleaned up
📦 Containers
Stopped containers sitting on disk, using no CPU but still consuming filesystem space.
docker container prune
🖼️ Images
Dangling <none> images and old tagged images no longer referenced by any container.
docker image prune
💾 Volumes
Named volumes not attached to any container. Can hold gigabytes of old database data.
docker volume prune
🌐 Networks
Custom networks not connected to any running container — typically left from removed Compose projects.
docker network prune
⚡ Build cache
Cached layers from docker build. Often the biggest consumer of space on active dev machines.
docker builder prune
☢️ Everything
All of the above at once (except volumes unless --volumes is passed).
docker system prune
🔬 Command anatomy
docker rm -f my-container
docker

The CLI binary — passes the command to the Docker daemon.

rm

Remove a container. Shorthand for docker container rm. The container must be stopped unless -f is used.

-f

Force — stops the container first if it's running, then removes it. Without this flag, removing a running container is an error.

name / ID

The container to target — by name (e.g. web) or full/short container ID. You can pass multiple separated by spaces.

🪜 Step-by-step guide
1
Spin up some containers to clean up
Create a few containers so we have something to work with. One runs in the background; two are one-off jobs that exit immediately.
shell 
# Long-running container
docker run -d --name web nginx

# Two one-off containers (exit immediately)
docker run --name job1 alpine echo "done"
docker run --name job2 alpine echo "done"
2
Inspect disk usage before cleanup
Get a clear picture of what's consuming space and what's reclaimable before touching anything.
shell 
docker system df
expected output
TYPE TOTAL ACTIVE SIZE RECLAIMABLE
Images 3 1 197.7MB 184.3MB (93%)
Containers 3 1 0B 0B
Local Volumes 0 0 0B 0B
Build Cache 0 0 0B 0B
💡
Add -v for a verbose breakdown listing each image and container individually — useful for identifying the biggest offenders.
3
Remove a specific stopped container
Target a single container by name. This only works if the container is already stopped.
shell 
docker rm job1
expected output
job1
4
Force-remove a running container
The -f flag stops the container first (with SIGKILL, no grace period) then deletes it — all in one command.
shell 
docker rm -f web
⚠️
-f uses SIGKILL immediately — no graceful shutdown. If web were a database, this could corrupt in-flight writes. In production, prefer docker stop first, then docker rm.
5
Prune all stopped containers at once
Instead of removing containers one by one, bulk-delete every stopped container in a single command. Great for clearing out accumulated job containers.
shell 
docker container prune
expected output
WARNING! This will remove all stopped containers.
Are you sure you want to continue? [y/N] y
Deleted Containers:
a1b2c3d4e5f6789012345678901234567890abcd

Total reclaimed space: 0B
ℹ️
Docker always prompts for confirmation before pruning. Add -f to skip the prompt in scripts: docker container prune -f
6
Remove dangling images
Pull a fresh image and rebuild to manufacture a dangling layer, then clean it up.
shell 
# See any dangling images (<none>:<none>)
docker images -f "dangling=true"

# Remove only dangling images (safe — no containers use them)
docker image prune

# Remove ALL unused images (not just dangling)
docker image prune -a
🚨
image prune -a deletes every image not currently used by a running container — including ones you might want to reuse soon. Only run this if you're intentionally freeing up space.
7
Remove a specific image by name or ID
Target individual images with docker rmi when you know exactly what you want gone. You can pass multiple image names or IDs at once.
shell 
# Remove by name and tag
docker rmi nginx:latest

# Remove multiple at once
docker rmi nginx:latest alpine:latest
ℹ️
Docker will refuse to delete an image that's in use by any container — even a stopped one. Remove the container first, then the image.
8
Prune unused volumes
Volumes persist after container removal by design. List orphaned volumes, then prune the ones you no longer need.
shell 
# List all volumes
docker volume ls

# Remove all volumes not attached to a container
docker volume prune

# Remove a specific volume by name
docker volume rm my-db-data
🚨
Volume data is permanent once deleted. This includes database files, uploaded files, and any other data that wasn't stored elsewhere. Always verify the volume contents or back up before pruning.
9
The nuclear option — system prune
When you want to reclaim everything at once: stopped containers, dangling images, unused networks, and build cache.
shell 
# Remove everything unused (prompts for confirmation)
docker system prune

# Also include all unused images, not just dangling
docker system prune -a

# Include volumes too (use with extreme caution)
docker system prune -a --volumes
expected output
WARNING! This will remove:
- all stopped containers
- all networks not used by at least one container
- all dangling images
- all dangling build cache
Are you sure you want to continue? [y/N] y

Deleted Containers: ...
Deleted Images: ...
Total reclaimed space: 1.42GB
10
Verify the cleanup
Run docker system df again to confirm the reclaimable space is now zero.
shell 
docker system df
expected output (after full cleanup)
TYPE TOTAL ACTIVE SIZE RECLAIMABLE
Images 0 0 0B 0B
Containers 0 0 0B 0B
Local Volumes 0 0 0B 0B
Build Cache 0 0 0B 0B
📋 Prune command reference
Command What it removes Risk
docker container prune All stopped containers safe
docker image prune Only dangling images (<none>:<none>) safe
docker image prune -a All images not used by any container caution
docker volume prune All volumes not attached to a container data loss
docker network prune All networks not used by any container safe
docker builder prune All unused build cache layers safe
docker system prune Containers + dangling images + networks + build cache caution
docker system prune -a --volumes Everything — all unused resources including volumes data loss
🔍 What happens under the hood
Step 01
🛑
Process killed
On docker rm -f, the container's PID 1 receives SIGKILL immediately.
Step 02
🗂️
Writable layer deleted
The container's unique writable filesystem layer is removed from the storage driver.
Step 03
🌐
Network detached
Virtual Ethernet interfaces and IP allocations for the container are released.
Step 04
📋
Metadata removed
Container ID, name, config, and log files are deleted from Docker's internal database.
Step 05
💾
Volumes persist
Named and anonymous volumes survive untouched — they must be pruned separately.
Step 06
🖼️
Image stays cached
The underlying image layers remain on disk and can be reused instantly for the next container.
🎯 What you learned
  • How to remove a specific container with docker rm, and force-remove a running one with -f
  • How to bulk-delete all stopped containers at once using docker container prune
  • What dangling images are and how to safely remove them with docker image prune
  • That volumes survive container removal and must be pruned separately — with care, since data is permanent
  • How to inspect disk usage before and after with docker system df
  • The difference in scope and risk between targeted prune commands and docker system prune
