Task 16  ·  Intermediate

Use a
.dockerignore File

Add a .dockerignore to exclude node_modules, secrets, and other unnecessary files from your build context — making images smaller, builds faster, and avoiding accidental leaks.

~15 minutes Completed Task 14 & 15 Node.js project from Task 14
📖 Required concepts
📬
The Build Context
When you run docker build, Docker packages up your entire project directory and sends it to the Docker daemon as a build context. Every file in that directory is included — even ones your image doesn't need.
🚫
What is .dockerignore?
.dockerignore works exactly like .gitignore. It tells Docker which files and folders to exclude from the build context before sending it to the daemon. Excluded files are never read, sent, or available inside the build.
⚡
Why node_modules Must Be Excluded
node_modules can contain hundreds of megabytes of files. If you copy your source into the image and then run npm install, the image already gets fresh dependencies — sending the host's node_modules just bloats the context and can overwrite what was installed.
🔐
Preventing Secret Leaks
Files like .env, .env.local, and private keys that land in the build context can end up baked into an image layer — even if you delete them in a later step. Excluding them from the start is the only safe approach.
♻️
Layer Cache Invalidation
Docker busts a layer's cache whenever any file in the context changes. If node_modules or build artifacts are included, a trivial file change in a dependency can invalidate your COPY layer and force a full reinstall. Ignoring them keeps your cache stable and fast.
🔢
Pattern Syntax
.dockerignore uses glob patterns. A leading ! negates a rule (re-includes previously excluded files). Lines starting with # are comments. Patterns are evaluated top to bottom, with later rules taking precedence.
🔬 File anatomy
# comment · node_modules · *.log · !src/keep.js
# comment

Lines starting with # are ignored. Use them to document why a rule exists.

pattern

A plain name or path (e.g. node_modules) excludes that exact file or directory anywhere in the tree.

glob

* matches any string, ** matches across directory separators. So **/*.log excludes all log files at any depth.

! negate

A leading ! re-includes a file that a previous rule excluded. Order matters — later rules win.

⚠️ The problem without .dockerignore

Without a .dockerignore, building a typical Node.js project sends everything to the daemon — including hundreds of megabytes you don't need.

Without .dockerignore
247 MB
Build context includes node_modules, .git history, test fixtures, logs
With .dockerignore
1.2 MB
Build context contains only your source files and package manifests
⚠️
The build context is transferred to the daemon every single time you run docker build. A bloated context means slow builds even when nothing changed.
🪜 Step-by-step guide
1
Start with the Node.js project from Task 14
Navigate to your project directory. It should look like this before we add anything:
shell 
ls -la
expected output
Dockerfile
index.js
package.json
node_modules/ ← this is the problem
.env ← never bake this in
ℹ️
You need a local node_modules to run the app locally, but it should never be copied into a Docker image. The Dockerfile's npm install step handles that inside the container.
2
Measure the build context before .dockerignore
Build your image and watch the first line Docker prints — it tells you exactly how large the build context is.
shell 
docker build -t myapp:before .
expected output — note the context size
[+] Building 0.3s (2/2) FINISHED
=> [internal] load build context
=> => transferring context: 247.31 MB ← huge!
💡
Write down that number. You'll compare it after adding .dockerignore to see the improvement.
3
Create the .dockerignore file
Create .dockerignore in the same directory as your Dockerfile. This is the standard starter for any Node.js project:
.dockerignore 
# Dependencies — rebuilt inside the image by npm install
node_modules

# npm debug logs
npm-debug.log*
yarn-debug.log*
yarn-error.log*

# Environment files — never bake secrets into an image
.env
.env.*

# Git history — not needed at runtime
.git
.gitignore

# Editor & OS noise
.DS_Store
Thumbs.db
.vscode

# Test files & coverage
*.test.js
coverage/

# Build output (if applicable)
dist/
build/

# The Dockerfile itself and this file
Dockerfile
.dockerignore
ℹ️
Ignoring Dockerfile and .dockerignore themselves is optional but clean — they're never needed inside the running container.
4
Rebuild and observe the difference
Run the exact same build command again. Watch the transferring context line change dramatically.
shell 
docker build -t myapp:after .
expected output — context is now tiny
[+] Building 2.1s (8/8) FINISHED
=> [internal] load build context
=> => transferring context: 1.24 kB ← tiny!
✅
The context dropped from hundreds of MB to kilobytes. The image itself will be smaller too — no leftover host dependencies contaminating the layers.
5
Verify node_modules is not in the image
Run a container from your new image and list /app to confirm only your source files are there — not the host's node_modules.
shell 
docker run --rm myapp:after ls -la /app
expected output
index.js
package.json
package-lock.json
node_modules/ ← installed fresh by npm install in the Dockerfile
💡
node_modules is still present — but it was created inside the image by npm install during the build, not copied from your host. This is exactly what you want.
6
Compare the image sizes
List both images side by side to see the size difference. The "before" image may be identical if the Dockerfile's COPY was already selective — or dramatically larger if it used COPY . ..
shell 
docker images myapp
expected output
REPOSITORY TAG IMAGE ID CREATED SIZE
myapp after b3f91a2c7d4e 12 seconds ago 145MB
myapp before c1e82b4f9a3d 2 minutes ago 392MB
7
Use negation to re-include a specific file
Sometimes you want to exclude a whole directory but keep one file inside it. The ! prefix negates a rule. For example, to exclude all .env files except a non-secret template:
.dockerignore — negation example 
# Exclude all env files …
.env*
# … but re-include the example template
!.env.example
ℹ️
Order matters. The last matching rule wins. Putting !.env.example before .env* would have no effect — it would be overridden by the wildcard rule below it.
8
Clean up
Remove the "before" image now that you've confirmed the improvement.
shell 
docker rmi myapp:before
📋 Common .dockerignore patterns
Pattern What it excludes
node_modules
All local Node.js dependencies (any depth)
**/__pycache__
All Python bytecode cache dirs (recursive)
*.log
All log files anywhere in the project
.env*
All environment files (.env, .env.local, .env.prod)
.git
Git history and index — can be hundreds of MB
coverage/
Test coverage reports
**/*.test.js
All test files at any depth
.DS_Store
macOS metadata files
tmp/
Temporary files and scratch directories
