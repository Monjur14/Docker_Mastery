Task 34  ·  Advanced

Docker
Healthchecks

Add a HEALTHCHECK instruction to a Dockerfile and a healthcheck: block in Compose so Docker can tell the difference between a container that's running and one that's actually ready.

~20 minutes Docker + Compose installed Familiarity with Dockerfiles
📖Required concepts
❌
Running ≠ ready
Docker marks a container as running the instant its main process starts — but a web server might take two more seconds to bind its port, or a database might need ten seconds to initialise its data files. Without a healthcheck, Docker has no way to know the difference.
❤️
What a healthcheck is
A healthcheck is a command Docker runs inside the container on a schedule — every 30 seconds by default. Exit code 0 means healthy. Exit code 1 means unhealthy. Docker tracks consecutive failures and updates the container's health status accordingly.
⏳
The starting state
When a container first launches, its health status is starting — healthchecks run but failures don't count toward the unhealthy threshold yet. Only after the start-period elapses does Docker start counting failures.
🔗
depends_on + condition
In Compose, depends_on with condition: service_healthy makes Compose hold off starting a dependent service — like a Node.js app — until the dependency's healthcheck reports healthy. This replaces fragile wait-for scripts.
📍
Two places to define it
A healthcheck can live in the Dockerfile — baked into the image, applied everywhere the image runs — or in Compose, which overrides the image's healthcheck for that specific deployment. Both use the same parameters; Compose just wins if both are present.
🛠️
What to probe
For HTTP services, probe a dedicated /health endpoint that returns 200 only when the app is fully ready. For databases, use their native ping command. Avoid probing business logic endpoints — they may be slow or state-dependent.

ℹ️
Docker never restarts a container just because it's unhealthy — it only updates the status label. It's up to Compose (with restart: on-failure), Swarm, or Kubernetes to act on that signal and restart or replace the container.
🔬Instruction anatomy
HEALTHCHECK --interval=30s --timeout=5s --start-period=10s --retries=3 CMD curl -f http://localhost/health || exit 1
HEALTHCHECK

Dockerfile instruction that tells Docker how to test whether the container is healthy.

--interval

How often to run the check. Default: 30s. Shorter = faster detection but more overhead.

--timeout

If the command hasn't finished within this time, it counts as a failure. Default: 30s.

--start-period

Grace window after container start — failures during this time don't count toward the retry threshold. Use this for slow-starting services. Default: 0s.

--retries

How many consecutive failures before the status becomes unhealthy. Default: 3.

CMD

The shell command to run. Exit 0 = healthy, exit 1 = unhealthy. curl -f fails with a non-zero exit code on HTTP 4xx/5xx responses.

Parameter Default Advice
--interval 30s Use 10–15s in dev for faster feedback; 30s is fine in production.
--timeout 30s Keep well below --interval. 5s is a sensible ceiling for most HTTP checks.
--start-period 0s Set to the worst-case startup time of your service — e.g. 30s for MySQL, 10s for a Node.js app.
--retries 3 3 is right for most cases. Lower means faster detection; higher means more tolerance for transient blips.
🪜Step-by-step guide
1
Set up the project
Create a minimal Express app with a /health endpoint — the dedicated route your healthcheck will probe.
healthcheck-demo/
  ├── app.js
  ├── package.json
  ├── Dockerfile
  └── compose.yaml
shell
mkdir healthcheck-demo && cd healthcheck-demo
2
Write the app with a /health route
The app has two routes: the main endpoint and a dedicated health probe. The health route is intentionally lightweight — no DB queries, no business logic — so it always responds fast.
app.js
const express = require('express');
const app = express();
const PORT = process.env.PORT || 3000;

// Simulate a slow startup — app takes 5s to be "ready"
let ready = false;
setTimeout(() => { ready = true; }, 5000);

// Main route
app.get('/', (req, res) => {
  res.send('Hello from the app!');
});

// Health probe — only returns 200 once the app is ready
app.get('/health', (req, res) => {
  if (ready) {
    res.status(200).json({ status: 'ok', uptime: process.uptime() });
  } else {
    res.status(503).json({ status: 'starting' });
  }
});

app.listen(PORT, () => {
  console.log(`Server listening on port ${PORT}`);
});
💡
The 5-second setTimeout simulates a real-world slow startup — database connections being established, config being loaded, caches being warmed. The /health route returns 503 until then, causing the healthcheck to fail and the container to stay in starting.

4
Write the Dockerfile with HEALTHCHECK
Add the HEALTHCHECK instruction after exposing the port. Alpine images don't include curl, so we install it in the same RUN step as our dependencies.
Dockerfile
FROM node:22-alpine

# Install curl for the healthcheck probe
RUN apk add --no-cache curl

WORKDIR /app

COPY package.json .
RUN npm install --omit=dev

COPY app.js .

EXPOSE 3000

# Healthcheck baked into the image
HEALTHCHECK --interval=15s --timeout=5s --start-period=10s --retries=3 \
  CMD curl -f http://localhost:3000/health || exit 1

CMD ["node", "app.js"]
ℹ️
curl -f exits non-zero on HTTP 4xx and 5xx responses — not just network failures. Without -f, a 503 would still exit 0 and be mistakenly reported as healthy.
5
Write compose.yaml with a healthcheck override
Compose can define or override the image's healthcheck. It also uses condition: service_healthy to hold off starting dependent services until the check passes.
compose.yaml
services:

  app:
    build: .
    ports:
      - "3000:3000"
    restart: on-failure
      # Override (or supplement) the Dockerfile healthcheck
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:3000/health"]
      interval: 15s
      timeout: 5s
      start_period: 10s
      retries: 3

    # A hypothetical worker that must wait until app is healthy
  worker:
    image: alpine
    command: sh -c "echo 'App is healthy, starting work!' && sleep infinity"
    depends_on:
      app:
        condition: service_healthy
⚠️
Use CMD form (JSON array) in the Compose test: field, not shell string form. CMD-SHELL also works and accepts a plain string: test: ["CMD-SHELL", "curl -f http://localhost:3000/health || exit 1"]

Inspect the healthcheck history

9
Simulate an unhealthy container
Manually trigger a failure by making the health endpoint return 503 — exec into the container and set a flag, or simply stop the server process to watch the status degrade.
shell
# Probe the /health endpoint manually to confirm it's 200
docker exec healthcheck-demo-app-1 \
  curl -s http://localhost:3000/health

# Kill the node process inside the container
# (container stays running, but app port goes dark)
docker exec healthcheck-demo-app-1 \
  sh -c "kill 1"
docker ps after 3 failed checks (≈ 45 s)
NAMES STATUS
healthcheck-demo-app-1 Up 2 minutes (unhealthy)
ℹ️
With restart: on-failure in Compose, the container will restart after the main process exits. Try restart: unless-stopped to watch it automatically recover and return to healthy.
10
Add a healthcheck to MySQL
MySQL ships with mysqladmin ping — a built-in liveness probe that exits 0 only when the server is accepting connections. Add it to a Compose stack to make a backend app wait for MySQL to be truly ready.
compose.yaml — mysql with healthcheck
services:

  db:
    image: mysql:8.4
    environment:
      MYSQL_ROOT_PASSWORD: rootpass
      MYSQL_DATABASE: appdb
      MYSQL_USER: appuser
      MYSQL_PASSWORD: apppass
    volumes:
      - mysql-data:/var/lib/mysql
    healthcheck:
      test: ["CMD", "mysqladmin", "ping", "-h", "localhost",
             "-u", "root", "-prootpass"]
      interval: 10s
      timeout: 5s
      start_period: 30s
      retries: 5

  app:
    build: .
    ports:
      - "3000:3000"
    environment:
      DB_HOST: db
      DB_USER: appuser
      DB_PASSWORD: apppass
      DB_NAME: appdb
    depends_on:
      db:
        condition: service_healthy

volumes:
  mysql-data:
shell — override healthcheck at runtime
docker run -d \
  --name mysql-dev \
  -e MYSQL_ROOT_PASSWORD=rootpass \
  -e MYSQL_DATABASE=appdb \
  --health-cmd="mysqladmin ping -h localhost -u root -prootpass" \
  --health-interval=10s \
  --health-timeout=5s \
  --health-start-period=30s \
  --health-retries=5 \
  mysql:8.4
11
Disable the healthcheck when you don't need it
If a base image has a healthcheck baked in but you want to opt out — perhaps in a test environment — you can disable it entirely.
Dockerfile — disable inherited healthcheck
HEALTHCHECK NONE
compose.yaml — disable healthcheck for a service
healthcheck:
  disable: true
12
Tear down
shell
docker compose down --volumes --rmi local
🔍What happened under the hood
Step 01
🚀
Container starts
Status set to starting. Daemon schedules the first healthcheck after one interval.
Step 02
⏱️
Check fires
Daemon execs the CMD inside the container's filesystem, same network namespace — no host network used.
Step 03
📊
Exit code read
Exit 0 resets the failure streak and marks healthy. Exit 1 (or timeout) increments the failure counter.
Step 04
📋
Log stored
Result is appended to the container's health log (last 5 kept). Visible via docker inspect.
Step 05
📡
Event emitted
Status transitions emit Docker events — Compose, Swarm, and Kubernetes watch these to trigger orchestration actions.
Step 06
🔁
Cycle repeats
Daemon waits another interval and fires again — indefinitely for the container's lifetime.
