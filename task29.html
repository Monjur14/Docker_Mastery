Task 29  ·  Intermediate

Scale a Service with
Docker Compose

Use docker compose up --scale to run multiple replicas of your app service simultaneously, then observe how a load balancer distributes traffic across them.

~25 minutes Docker Compose v2 installed Familiarity with Compose services
📖Required concepts
📋
Replicas
A replica is an identical copy of a container running the same image and config. Scaling a service to 3 replicas means 3 separate containers — each with its own filesystem, process space, and IP — all serving the same role. If one crashes, the other two keep serving traffic.
⚖️
Load Balancing in Compose
Docker's built-in DNS round-robins requests across replicas of the same service name. To expose replicas to the host, you need a reverse proxy (like Nginx) as a separate service — it receives all external traffic and forwards requests to the app replicas by service name.
⚡
The Port Binding Conflict
Two containers cannot bind the same host port. If your docker-compose.yml maps ports: "3000:3000" and you try to scale to 3 replicas, Docker will error on the second container. You must remove ports from the scaled service and let the proxy handle host exposure.
🌐
Docker's Internal Network
All services in a Compose project share an internal network automatically. Containers reach each other by service name as the hostname. So http://app:3000 from the proxy resolves to one of the app replicas — Docker's embedded DNS picks which one.
🔢
deploy.replicas vs --scale
There are two ways to declare replica count. --scale app=3 on the CLI is ad hoc and overrides the compose file at runtime. deploy: replicas: 3 in the compose file is declarative and version-controlled — prefer it for anything beyond local experimentation.
🏷️
Container Naming with Replicas
Compose names replicas with an index suffix: project-app-1, project-app-2, project-app-3. You can exec into a specific replica by its full name, or target all replicas of a service by its service name in Compose commands.

🔬Command anatomy
docker compose up --scale app = 3 -d
docker

The Docker CLI.

compose up

Create and start all services defined in docker-compose.yml. Idempotent — re-running applies only what changed.

--scale

Override the replica count for one or more services at runtime. Repeatable: --scale app=3 --scale worker=2.

app

The service name from your compose file to scale. Must match exactly.

=3

The number of replicas to run. Set to 1 to scale back down without stopping other services.

-d

Detach — run containers in the background and return the prompt.

⚠️The port conflict — and how to fix it

Before scaling, you must remove any static ports mapping from your app service. Here's why:

❌ What fails when ports is set on a scaled service
replica 1 tries to bind host:3000 → ✓ succeeds
replica 2 tries to bind host:3000 → ✗ port already in use — container fails
replica 3 tries to bind host:3000 → ✗ port already in use — container fails
✅
The fix: remove ports from the app service and add a separate proxy service that binds host port 80. The proxy reaches the app replicas over the internal network — no host port conflicts possible.
🪜Step-by-step guide
1
Write a scalable docker-compose.yml
Key points: the app service has no ports mapping, and the proxy service is the only one that touches the host network.
docker-compose.yml 
services:

  app:
    build: .
    # No 'ports' here — replicas can't share a host port
    expose:
      - "3000"    # only exposes to the internal network
    environment:
      - NODE_ENV=production

  proxy:
    image: nginx:alpine
    ports:
      - "80:80"   # only the proxy binds a host port
    volumes:
      - ./nginx.conf:/etc/nginx/conf.d/default.conf:ro
    depends_on:
      - app
2
Create the Nginx load balancer config
Nginx uses the service name app as the upstream host. Docker's DNS resolves that name to one of the replicas automatically.
nginx.conf 
upstream app_servers {
    # Docker DNS round-robins 'app' across all replicas
    server app:3000;
}

server {
    listen 80;

    location / {
        proxy_pass         http://app_servers;
        proxy_set_header   Host              $host;
        proxy_set_header   X-Real-IP         $remote_addr;
        proxy_set_header   X-Forwarded-For   $proxy_add_x_forwarded_for;
    }
}
ℹ️
Docker's DNS returns all replica IPs for the service name. Nginx's round-robin upstream selection then distributes requests evenly across them — no extra config needed.
3
Make your app report its hostname
Add a response header or body that includes the container's hostname. This lets you see which replica handled each request, proving load distribution is working.
index.js (Node.js example) 
const http = require('http');
const os   = require('os');

http.createServer((req, res) => {
  res.writeHead(200, { 'Content-Type': 'text/plain' });
  // os.hostname() returns the container ID in Docker
  res.end(`Hello from replica: ${os.hostname()}\n`);
}).listen(3000);

6
Observe load distribution
Hit the proxy endpoint several times in a row. Each request should be answered by a different replica — the hostname in the response tells you which one.
shell 
# Send 6 requests and watch which replica responds
for i in $(seq 1 6); do
  curl -s http://localhost
done
expected output — different replica each time
Hello from replica: a3f72b9c1d4e (app-1)
Hello from replica: c9e14b2f8a7d (app-2)
Hello from replica: f1d83a6c5b2e (app-3)
Hello from replica: a3f72b9c1d4e (app-1 again — round-robin)
Hello from replica: c9e14b2f8a7d (app-2)
Hello from replica: f1d83a6c5b2e (app-3)
✅
You can see the requests cycling across all three container IDs — round-robin load balancing is working. Each replica is handling roughly equal traffic.
7
Follow logs across all replicas
Use docker compose logs to tail all replicas at once. The container name prefix tells you which replica emitted each line.
shell 
# Tail logs from all app replicas (not the proxy)
docker compose logs -f app
expected output
app-1 | GET / 200 — served request
app-2 | GET / 200 — served request
app-3 | GET / 200 — served request
app-1 | GET / 200 — served request
8
Scale up to 5 replicas without downtime
Re-run the scale command with a higher count. Compose adds only the 2 new containers — the 3 existing ones keep running and serving traffic uninterrupted.
shell 
docker compose up --scale app=5 -d
expected output — only 2 new containers started
[+] Running 6/6
✔ Container myproject-proxy-1 Running
✔ Container myproject-app-1 Running ← unchanged
✔ Container myproject-app-2 Running ← unchanged
✔ Container myproject-app-3 Running ← unchanged
✔ Container myproject-app-4 Started ← new
✔ Container myproject-app-5 Started ← new
9
Scale back down to 1 replica
Scaling down removes the excess containers gracefully. Compose stops and removes the highest-numbered replicas first.
shell 
docker compose up --scale app=1 -d
ℹ️
Compose removes replicas from the top down: app-5 → app-4 → app-3 → app-2. app-1 remains running. In-flight requests to the removed containers will be dropped unless you configure graceful shutdown in your app.
10
Declare replicas in the compose file instead
For anything beyond local testing, commit the replica count in the file using deploy.replicas. This way the count is version-controlled and reproduced automatically on every docker compose up.
docker-compose.yml — declarative replica count 
services:
  app:
    build: .
    expose:
      - "3000"
    deploy:
      replicas: 3   # version-controlled, no --scale flag needed
💡
deploy.replicas is also the correct syntax for Docker Swarm and most Kubernetes translators. Starting here keeps your config portable.
11
Tear everything down
Stop and remove all containers, networks, and the default volume created for this project.
shell 
docker compose down
📊What scaling changes
Before scaling
1
Single point of failure. One slow request blocks the next. Max throughput = one process.
After scaling to 3
3×
Concurrent requests handled in parallel. One replica can crash while the other two keep serving.
⚠️
Stateless apps scale easily. If your app stores session data in memory or writes to local files, replicas won't share that state. Use a shared store (Redis, a database) for anything that must be consistent across replicas before scaling horizontally.
