Task 22  ·  Intermediate

Inspect
Docker Volumes

List, inspect, and understand how named volumes work and where Docker stores their data on disk — using docker volume ls, inspect, --format, and df.

~10 minutes Docker installed Terminal / Shell
📖 Required concepts
🗂️
The volume subcommand group
All volume operations live under docker volume: create, ls, inspect, rm, and prune. These manage volumes independently of any container — you never need a container running to list or inspect a volume.
📍
Mountpoint — where data actually lives
Every named volume has a Mountpoint — an absolute path on the host where Docker stores its contents. On Linux this is directly readable at /var/lib/docker/volumes/<name>/_data. On Mac and Windows it's inside a hidden Linux VM, but the path is still reported accurately by inspect.
🔠
Go template formatting
docker inspect --format accepts Go templates to extract specific fields without parsing raw JSON. Useful in scripts where you only want, say, the Mountpoint and nothing else. Syntax: {{.FieldName}}.
🏷️
Volume drivers
The Driver field controls where and how the volume is stored. The default is local — data stays on this machine. Third-party drivers like rexray or nfs can store volume data on network shares or cloud block storage, making volumes portable across hosts.
🔍
Filtering and formatting ls output
docker volume ls supports --filter to narrow by driver, label, or dangling status. Combining it with --format lets you produce clean, scriptable output — only the column you care about, nothing else.
🧹
Dangling volumes
A dangling volume is one not referenced by any container. They accumulate silently over time and can consume significant disk space. docker volume prune removes all dangling volumes at once — with a confirmation prompt before deleting anything.
🔬 Command anatomy
docker volume inspect --format '{{.Mountpoint}}' pgdata
docker

The CLI binary — how you talk to the Docker daemon from your terminal.

volume inspect

Return the full JSON metadata for one or more named volumes.

--format

Apply a Go template to the JSON output, extracting only the fields you specify instead of printing the whole object.

{{.Mountpoint}}

Go template expression that reads the Mountpoint field — the absolute host path where this volume's data lives on disk.

pgdata

The name of the volume to inspect. Pass multiple names to inspect several at once.

🗺️ Where Docker stores volume data
Linux /var/lib/docker/volumes/<name>/_data/
Directly browsable as root. Every file inside is the volume's actual content.
Mac ~/Library/Containers/com.docker.docker/Data/vms/0/data/docker/volumes/<name>/_data/
Inside Docker Desktop's hidden Linux VM — not directly accessible from macOS Finder.
Windows \\wsl$\docker-desktop-data\data\docker\volumes\<name>\_data\
Accessible via WSL2 path in Explorer — but treat it as read-only from Windows to avoid corruption.
ℹ️
docker volume inspect always reports the path inside the environment where dockerd runs. On Mac and Windows that's the VM path — which is why the Mountpoint shown won't match a host filesystem path you can browse directly.
🪜 Step-by-step guide
1
Create a few volumes to work with
Set up three named volumes so the listing and filtering commands have something meaningful to operate on.
shell 
docker volume create pgdata
docker volume create redisdata
docker volume create uploads
2
List all volumes
The default output shows driver and name. All three volumes you created should appear, along with any pre-existing ones from earlier tasks.
shell 
docker volume ls
expected output
DRIVER VOLUME NAME
local pgdata
local redisdata
local uploads
3
Inspect a single volume — full JSON
Get the complete metadata object Docker holds for a volume. Every field is explained in the reference table below.
shell 
docker volume inspect pgdata
expected output
[
{
"CreatedAt": "2026-07-02T09:00:00Z", when docker volume create ran
"Driver": "local", storage backend
"Labels": null, key/value tags you can add
"Mountpoint": "/var/lib/docker/volumes/pgdata/_data",
"Name": "pgdata",
"Options": null, driver-specific config
"Scope": "local" local = this host only
}
]
CreatedAt ISO 8601 timestamp of when the volume was created — not when it last had data written to it.
Driver local stores data on this machine's filesystem. Third-party drivers (nfs, s3fs, rexray) can store data remotely.
Labels Key/value metadata you attach at create time with --label. Useful for identifying which app or team owns a volume.
Mountpoint The actual path on the host (or inside the VM) where the volume's files are stored. The directory Docker exposes into containers.
Options Driver-specific configuration — e.g. NFS server address or mount options. Null for the default local driver.
Scope local means the volume exists only on this Docker host. global (set by some drivers) means it's shared across a Swarm cluster.
4
Inspect multiple volumes at once
Pass several names to get a JSON array covering all of them in one call — handy for diffing config across volumes.
shell 
docker volume inspect pgdata redisdata uploads
ℹ️
The result is a JSON array with one object per volume, in the order you listed them. Pipe to jq for pretty-printing and field extraction if it's installed.
5
Extract specific fields with --format
Use Go templates to pull only the data you need — ideal for shell scripts that don't want to parse JSON.
shell 
# Just the host path where data lives
docker volume inspect --format '{{.Mountpoint}}' pgdata

# Name and driver on one line
docker volume inspect --format '{{.Name}}  driver={{.Driver}}' pgdata

# When was it created?
docker volume inspect --format '{{.CreatedAt}}' pgdata
expected output (first command)
/var/lib/docker/volumes/pgdata/_data
{{.Mountpoint}} Absolute path on the host (or VM) where the volume's files are stored.
{{.Name}} The volume's name as passed to docker volume create.
{{.Driver}} Storage driver — local for the default, or a plugin name.
{{.CreatedAt}} ISO 8601 creation timestamp.
{{.Scope}} local or global depending on the driver.
{{json .Labels}} Serialise the Labels map back to JSON — useful when labels hold structured data.
6
Filter the volume list
docker volume ls --filter narrows output without inspecting every volume individually.
shell 
# Show only volumes using the local driver
docker volume ls --filter driver=local

# Show only dangling volumes (not attached to any container)
docker volume ls --filter dangling=true

# Print just the volume names — one per line, no header
docker volume ls --format '{{.Name}}'
expected output (last command)
pgdata
redisdata
uploads
💡
Combine --format '{{.Name}}' with a shell loop to perform batch operations — like inspecting all volumes in a list without typing each name manually.
7
Check disk usage with docker system df
See exactly how much disk space volumes (and other Docker objects) are consuming on this host — and how much could be reclaimed.
shell 
docker system df
expected output
TYPE TOTAL ACTIVE SIZE RECLAIMABLE
Images 4 2 1.23GB 512MB (41%)
Containers 2 1 14kB 0B (0%)
Local Volumes 3 0 148MB 148MB (100%)
Build Cache 12 0 230MB 230MB
⚠️
ACTIVE 0 in the Volumes row means all three are dangling — not attached to any running container. Reclaimable: 100% means docker volume prune could free all of that space.
shell 
# Verbose breakdown — one row per volume
docker system df -v
expected output (volumes section)
VOLUME NAME LINKS SIZE
pgdata 0 144MB
redisdata 0 2.1MB
uploads 0 1.8MB
8
Browse a volume's contents directly (Linux)
On Linux you can read the Mountpoint path and list the files Docker is actually storing. On Mac/Windows, use a temporary container to browse instead.
shell — Linux only 
# Get the path, then list its contents
MOUNT=$(docker volume inspect --format '{{.Mountpoint}}' pgdata)
sudo ls -lah $MOUNT
shell — Mac / Windows alternative 
# Spin up a temporary Alpine container to peek inside the volume
docker run --rm \
  -v pgdata:/inspect:ro \
  alpine \
  sh -c 'ls -lah /inspect'
💡
The :ro suffix makes the mount read-only inside the Alpine container — you can browse without risk of accidentally modifying the data.
9
Create a volume with a label
Labels let you tag volumes with metadata — who owns it, which app it belongs to, what environment it's for. This makes filtering and auditing far easier at scale.
shell 
docker volume create \
  --label app=myapp \
  --label env=production \
  myapp-prod-data
shell 
# Filter ls by label to find all production volumes
docker volume ls --filter label=env=production

# Inspect and print the labels as JSON
docker volume inspect --format '{{json .Labels}}' myapp-prod-data
expected output (second command)
{"app":"myapp","env":"production"}
10
Clean up — prune dangling volumes and remove the rest
Remove all volumes created in this task. Use prune for the dangling ones, and rm for any still referenced.
shell 
# Remove all dangling volumes (prompts for confirmation)
docker volume prune

# Or remove specific volumes by name
docker volume rm pgdata redisdata uploads myapp-prod-data
⚠️
docker volume prune only removes volumes not referenced by any container — stopped or running. It will not touch a volume that a stopped container still points at. To remove those too, first run docker container prune, then prune volumes.
🔍 What happened under the hood
Step 01
🗃️
Daemon registry
Docker stores volume metadata in its own database — docker volume ls reads from there, not the filesystem.
Step 02
📂
_data directory
Each volume's files live in a _data/ subdirectory under the volumes root — isolating content from Docker's own metadata files.
Step 03
🔬
inspect reads metadata
The daemon returns the stored metadata object as JSON — no disk scan needed. Size data comes from a separate accounting layer.
Step 04
🧮
system df scans
docker system df walks the volume directories on disk to compute actual byte counts — which is why it can be slower than inspect.
Step 05
🏷️
Labels stored in DB
Labels are persisted in Docker's metadata store, not as files inside the volume — so they survive even if the volume's data directory is empty.
Step 06
🧹
Prune checks refs
Before deleting, prune checks the container-to-volume reference table. Any volume with at least one reference — even a stopped container — is skipped.
