Task 02  ·  Beginner

Pull & Run
Nginx Web Server

Fetch the official nginx image from Docker Hub and start it in the foreground. Watch live HTTP access logs stream to your terminal as you browse to it.

~10 minutes Task 1 complete Terminal + Browser
📖Required concepts
🌐
What is Nginx?
Nginx (pronounced "engine-x") is a high-performance web server, reverse proxy, and load balancer. It's one of the most widely deployed servers on the internet. Its official Docker image is minimal, well-maintained, and a perfect first real-world container to run.
⬇️
docker pull vs docker run
docker pull downloads an image from a registry to your local machine — nothing runs. docker run does a pull automatically if the image isn't local, then creates and starts a container. You can use them separately or let run handle both.
🔌
Port Mapping (-p flag)
Containers are isolated networks by default — a port open inside a container is not visible on your host. The -p HOST:CONTAINER flag creates a mapping: traffic to HOST port on your machine is forwarded into CONTAINER port. For example, -p 8080:80 means your browser visits localhost:8080 but Nginx inside hears it on port 80.
📺
Foreground vs Detached Mode
By default docker run runs in foreground mode — your terminal is attached to the container's stdout/stderr and you see logs live. The container stops when you press Ctrl+C. Detached mode (-d) runs in the background — you get the prompt back immediately. This task runs in foreground so you can see live logs.
📝
stdout / stderr & Container Logs
Docker captures anything a process writes to stdout (standard output) and stderr (standard error) as the container's logs. Nginx is configured in its official image to write access logs to stdout and error logs to stderr — this is why you see requests appear in your terminal in real time.
🏷️
Image Tags
An image tag specifies a version: nginx:latest, nginx:1.25, nginx:alpine. When you omit the tag, Docker defaults to :latest. The alpine variant is much smaller (under 50 MB vs ~190 MB) as it's built on Alpine Linux — a minimal distro designed for containers.
🔌How port mapping works
Your Machine
:8080
localhost:8080
-p 8080:80
Container
:80
nginx inside
ℹ️
The left side of -p is always the host port (your machine). The right side is the container port (inside Docker). You can map any available host port — if 8080 is busy, use 9090 or any free port above 1024.
🔬Command anatomy
docker run -p 8080:80 nginx
docker

The Docker CLI binary.

run

Pull (if needed) + create + start a container in foreground mode.

-p 8080:80

Publish port — map host port 8080 → container port 80. Nginx listens on 80 inside the container.

nginx

Official Nginx image from Docker Hub. Shorthand for docker.io/library/nginx:latest.

🪜Step-by-step guide
1
Pull the Nginx image explicitly
This downloads the image without running it yet — great for understanding the pull step separately. Watch the layer download progress.
shell
docker pull nginx
expected output
Using default tag: latest
latest: Pulling from library/nginx
a803e7c4b030: Pull complete
8b625c47d697: Pull complete
4d3239651a63: Pull complete
0f816efa513d: Pull complete
01d159b8db2f: Pull complete
Digest: sha256:32fdf92b4e...
Status: Downloaded newer image for nginx:latest
docker.io/library/nginx:latest
ℹ️
Each line like a803e7c4b030: Pull complete is one image layer being downloaded. Docker downloads them in parallel and caches each one separately.
2
Verify the image is stored locally
Confirm the image is in your local cache before running it.
shell
docker images nginx
expected output
REPOSITORY TAG IMAGE ID CREATED SIZE
nginx latest a8758716bb6a 2 weeks ago 187MB
3
Run Nginx in foreground with port mapping
Start the container. Your terminal will show Nginx startup messages, then wait — it stays running and streams logs. Don't close this terminal yet.
shell
docker run -p 8080:80 nginx
expected output — nginx startup
/docker-entrypoint.sh: /docker-entrypoint.d/ is not empty, will attempt to perform configuration
/docker-entrypoint.sh: Looking for shell scripts in /docker-entrypoint.d/
/docker-entrypoint.sh: Launching /docker-entrypoint.d/10-listen-on-ipv6-by-default.sh
/docker-entrypoint.sh: Launching /docker-entrypoint.d/20-envsubst-on-templates.sh
/docker-entrypoint.sh: Configuration complete; ready for start up

█ cursor blinks here — nginx is running and waiting for requests
💡
The blinking cursor means Nginx is up and waiting. Your terminal is now "inside" the container's output stream — this is foreground mode.
4
Open your browser and visit Nginx
Open a browser and navigate to the address below. You should see the Nginx welcome page.
url
http://localhost:8080
✅
You should see a page that says "Welcome to nginx!" — Nginx is serving HTTP from inside a Docker container to your browser.
5
Watch the live access logs appear
Switch back to your terminal. You'll see an HTTP access log entry appear for every request your browser made.
live log output in your terminal
172.17.0.1 - - [01/Jul/2026:10:23:45 +0000] "GET / HTTP/1.1" 200 615 "-" "Mozilla/5.0 ..."
172.17.0.1 - - [01/Jul/2026:10:23:45 +0000] "GET /favicon.ico HTTP/1.1" 404 555 ...

Each log line follows the standard Nginx combined log format:

PartMeaning
172.17.0.1IP address of the client (your browser, seen from inside Docker's network)
[01/Jul/2026:10:23:45 +0000]Timestamp of the request in UTC
"GET / HTTP/1.1"HTTP method, path, and protocol version
200HTTP status code — 200 means OK (success)
615Response size in bytes
"Mozilla/5.0 ..."User-agent string — identifies your browser
ℹ️
Refresh the browser a few times — watch a new log line appear for each request. The 404 for /favicon.ico is normal; Nginx's default page has no favicon.
6
Also test with curl (optional)
Open a second terminal and use curl to send a request from the command line instead of a browser.
shell — second terminal
curl -I http://localhost:8080
expected output
HTTP/1.1 200 OK
Server: nginx/1.25.3
Date: Tue, 01 Jul 2026 10:25:00 GMT
Content-Type: text/html
Content-Length: 615
Connection: keep-alive
ℹ️
curl -I sends a HEAD request — it fetches only the HTTP headers, not the body. The Server: nginx/1.25.3 header confirms Nginx is responding from inside the container.
7
Stop the container
Go back to the terminal running Nginx and press Ctrl + C. Docker sends a SIGTERM signal to Nginx, which gracefully shuts down.
terminal after Ctrl+C
^C
2026/07/01 10:30:00 [notice] 1#1: signal 2 (SIGINT) received, exiting
2026/07/01 10:30:00 [notice] 1#1: gracefully shutting down
2026/07/01 10:30:00 [notice] 1#1: exit

$ █ ← prompt is back
8
Run it with a custom container name
Re-run Nginx with --name so it's easier to reference instead of using auto-generated names.
shell
docker run --name my-nginx -p 8080:80 nginx
💡
Now docker stop my-nginx, docker logs my-nginx, and docker rm my-nginx all work by name — much friendlier than using a random ID.
9
Try the lightweight Alpine variant
Compare image sizes by pulling the Alpine-based Nginx — it's over 75% smaller.
shell
# Stop current container first
docker stop my-nginx

# Pull and run the alpine variant
docker run --name my-nginx-alpine -p 8080:80 nginx:alpine
shell — compare sizes
docker images nginx
expected output
REPOSITORY TAG IMAGE ID CREATED SIZE
nginx alpine b1b2c3d4e5f6 3 days ago 41.5MB
nginx latest a8758716bb6a 2 weeks ago 187MB
10
Clean up all containers and images
Stop and remove everything from this task to keep your system tidy.
shell
# Stop and remove named containers
docker stop my-nginx my-nginx-alpine
docker rm my-nginx my-nginx-alpine

# Remove both nginx images
docker rmi nginx nginx:alpine
🔍What happened under the hood
Step 01
⬇️
Pull layers
Docker fetched 6 image layers from Docker Hub in parallel.
Step 02
📦
Container created
A writable layer was stacked on top of the read-only Nginx image.
Step 03
🌐
Network bridged
Docker created a virtual NIC and mapped host port 8080 → container port 80.
Step 04
▶️
Nginx started
The entrypoint script ran, configured Nginx, and launched the master process.
Step 05
📝
Logs streamed
Nginx wrote access logs to stdout → Docker captured them → your terminal displayed them.
Step 06
⏹️
Graceful stop
Ctrl+C sent SIGTERM → Nginx drained connections and exited cleanly.

  • How to name containers with --name for easier management
  • How image tags work — nginx:alpine vs nginx:latest and their size difference
