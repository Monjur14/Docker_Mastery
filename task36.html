Task 36  ·  Advanced

Secure Secret
Management

Inject API keys and TLS certificates using Docker Compose secrets: so they are never visible in environment variables, image layers, docker inspect output, or process listings.

~20 minutes Completed Task 30 (env vars) Docker Compose v2
📖Required concepts
🔐
What is a Docker Secret?
A Docker secret is a blob of sensitive data — a password, API key, TLS certificate, or private key — that Docker mounts into a container as a read-only file under /run/secrets/<name>. It is never stored in an image layer, passed as an environment variable, or visible in docker inspect.
😱
Why Env Vars Aren't Enough
Environment variables are convenient but leak in multiple ways: they appear in docker inspect output, in /proc/<pid>/environ on the host, in crash dumps and logging frameworks that capture all env vars, and in child processes. A secret mounted as a file has a much smaller exposure surface.
📁
The tmpfs Mount
Compose (and Swarm) mount secrets on a tmpfs filesystem — stored only in memory, never written to disk. When the container stops, the secret vanishes. The mount point is /run/secrets/secret_name and the file is readable only by the container's process — not other containers.
📂
File-based vs External Secrets
In Compose for local dev, secrets come from files on the host (e.g. ./secrets/db_password). In Docker Swarm or production, they can come from the Swarm secret store — encrypted at rest and distributed only to nodes that run a service needing that secret. Same syntax either way.
📋
Two-level Declaration
Secrets are declared in two places in a compose file. At the top level (secrets:) you define where the secret data comes from. Under each service (secrets: list) you declare which secrets that service can access. A secret not listed for a service is completely invisible to it.
📖
Reading Secrets in App Code
Your app reads the secret by opening the file at /run/secrets/secret_name at startup. This is a deliberate, explicit read — unlike env vars which are just always there. Many frameworks support a _FILE suffix convention for this pattern.

top-level

The secrets: block at the root of the compose file declares every secret that exists in this project, and where its data comes from.

name

The secret's logical name — used to reference it in service declarations and as the filename inside /run/secrets/.

file:

Path to the file on the host whose content becomes the secret. The file can contain any bytes — a token string, a PEM certificate, a JSON key file.

per-service

Under each service, secrets: is an allow-list. Only listed secrets are mounted into that container. Other services — even in the same project — cannot see them.

🪜Step-by-step guide
1
Set up the project structure
Create a dedicated secrets/ directory alongside your compose file. This directory must be in .gitignore — it holds real secret data.
myproject/
  docker-compose.yml
  Dockerfile
  index.js
  .gitignore
  secrets/  ← gitignored, real secrets live here
    stripe_key.txt
    db_password.txt
    tls.crt
    tls.key
  secrets.example/  ← committed, placeholder values only
    stripe_key.txt  ← contains "sk_test_REPLACE_ME"
🔐
Add secrets/ to .gitignore before creating any files inside it. Commit a secrets.example/ with placeholder values so teammates know what files are needed.
2
Write the secret files
Each secret is a plain file. No quoting, no KEY= prefix — just the raw value. Docker mounts the entire file content at /run/secrets/<name>.
shell
# Create the secrets directory
mkdir -p secrets

# Write raw values — no KEY= prefix, no quotes
echo -n "sk_live_abc123xyz"          > secrets/stripe_key.txt
echo -n "V3ryStr0ngP@ssword!"        > secrets/db_password.txt

# Or copy an existing cert/key pair
cp /path/to/server.crt secrets/tls.crt
cp /path/to/server.key secrets/tls.key

# Restrict permissions (good practice)
chmod 600 secrets/*.txt secrets/*.key
ℹ️
The -n flag on echo suppresses the trailing newline. Without it, your app will read the value with a \n appended — a common cause of "invalid token" errors with API keys. Always .trim() when reading secret files in code.
3
Declare secrets in docker-compose.yml
Register every secret at the top level, then grant access per service. A service only sees secrets explicitly listed under its secrets: key.
docker-compose.yml
services:

  app:
    build: .
    environment:
      # Non-sensitive config still fine as env vars
      NODE_ENV: production
      PORT: "3000"
    secrets:
      - stripe_key      # mounted at /run/secrets/stripe_key
      - db_password     # mounted at /run/secrets/db_password
    depends_on:
      - db

  db:
    image: mongo:7
    secrets:
      - db_password     # db gets only what it needs
    environment:
      # Mongo supports _FILE convention for secrets
      MONGO_INITDB_ROOT_PASSWORD_FILE: /run/secrets/db_password

# ── Top-level secret definitions ────────────────
secrets:
  stripe_key:
    file: ./secrets/stripe_key.txt
  db_password:
    file: ./secrets/db_password.txt
💡
The _FILE convention — used by Mongo, Postgres, Redis, and others — means the env var's value is a path to a file containing the secret, rather than the secret itself. This is the standard bridge between secrets-as-files and apps that expect env vars.
4
Start the stack and verify mounts
Bring up the project and confirm that secrets are mounted where you expect them — and that they're absent from environment variables.
shell
docker compose up -d

# ✓ Confirm the secret file is present inside the container
docker compose exec app ls -la /run/secrets/

# ✓ Read its value (for debugging only — remove in prod)
docker compose exec app cat /run/secrets/stripe_key

# ✓ Confirm it does NOT appear in env vars
docker compose exec app env | grep -i stripe
expected output
$ docker compose exec app ls -la /run/secrets/
total 8
-r-------- 1 root root 18 Jul 3 09:12 db_password
-r-------- 1 root root 22 Jul 3 09:12 stripe_key

$ docker compose exec app cat /run/secrets/stripe_key
sk_live_abc123xyz

$ docker compose exec app env | grep -i stripe
(no output — secret is not an env var)
✅
The files are mode 0400 (read-only, owner only) and owned by root. The grep returns nothing — the secret value is nowhere in the environment.
5
Read secrets in your Node.js app
Open the file at startup, trim any trailing whitespace, and use it. Keep the read synchronous so a missing secret crashes immediately with a clear error before the server begins accepting connections.
index.js
const fs   = require('fs');
const path = require('path');

// Helper: read a secret file or throw at startup
function readSecret(name) {
  const secretPath = path.join('/run/secrets', name);
  try {
    return fs.readFileSync(secretPath, 'utf8').trim();
  } catch {
    console.error(`Missing secret: ${secretPath}`);
    process.exit(1);  // fail fast — don't start without secrets
  }
}

// Read at startup — never stored in a variable longer than needed
const stripeKey  = readSecret('stripe_key');
const dbPassword = readSecret('db_password');

const dbUrl = `mongodb://root:${dbPassword}@db:27017/myapp`;

console.log('Secrets loaded. Starting server...');
// ... rest of server setup
6
Mount a TLS certificate pair
Certificates are binary-safe and work identically to text secrets. Declare both the cert and key as secrets, then read them in Node to configure HTTPS.
docker-compose.yml — TLS cert secrets
services:
  app:
    build: .
    ports:
      - "443:443"
    secrets:
      - tls_cert
      - tls_key
      - stripe_key

secrets:
  tls_cert:
    file: ./secrets/tls.crt
  tls_key:
    file: ./secrets/tls.key
  stripe_key:
    file: ./secrets/stripe_key.txt
index.js — HTTPS with cert secrets
const https = require('https'); const fs = require('fs'); const options = { // Read cert and key directly from secret mounts cert: fs.readFileSync('/run/secrets/tls_cert'), key: fs.readFileSync('/run/secrets/tls_key'), }; https.createServer(options, (req, res) => { res.end('Secure hello!\n'); }).listen(443);
7
Use a custom mount path or target name
By default the secret mounts as /run/secrets/<name>. You can override the mount path using the long-form service secret declaration.
docker-compose.yml — custom target path
services:
  app:
    build: .
    secrets:
      # Short form — mounts at /run/secrets/stripe_key
      - stripe_key

      # Long form — custom mount path and permissions
      - source: tls_cert
        target: /etc/ssl/certs/server.crt
        mode: 0444    # world-readable for nginx etc.
      - source: tls_key
        target: /etc/ssl/private/server.key
        mode: 0400    # owner-read only
ℹ️
Custom target paths are useful when integrating with software that looks for certs in a fixed path (Nginx, HAProxy) — you can point the secret directly to where the tool expects it without any code changes.
8
Confirm secrets don't appear in docker inspect
This is the acid test. Run docker inspect on the container and confirm no secret values leak through.
shell
# Check env vars exposed in inspect output
docker inspect myproject-app-1 \
  --format '{{json .Config.Env}}'

# Check that secret names (not values) are listed in Mounts
docker inspect myproject-app-1 \
  --format '{{json .HostConfig.Mounts}}'
expected output
$ docker inspect … --format '{{json .Config.Env}}'
["NODE_ENV=production","PORT=3000"]
← only non-secret vars appear; stripe_key value is absent

$ docker inspect … --format '{{json .HostConfig.Mounts}}'
[{"Type":"tmpfs","Target":"/run/secrets/stripe_key",...}]
← mount path visible, but not the secret value itself
✅
The secret value sk_live_abc123xyz appears nowhere in the inspect output. Only the mount path — not the content — is recorded in the container spec.
9
Rotate a secret without rebuilding
To update a secret value, replace the file on the host, then recreate only the affected service. The image is unchanged — only the mounted content updates.
shell
# 1. Replace the secret file on the host
echo -n "sk_live_newkey789" > secrets/stripe_key.txt

# 2. Recreate only the app service — db is unaffected
docker compose up -d --force-recreate app
ℹ️
--force-recreate stops and restarts the container so the new file is mounted fresh. A running container does not see changes to the secret file while it's live — the mount is snapshotted at container start.
10
Tear down — secrets are gone
When the stack stops, the tmpfs mounts are destroyed. The secret data that was in memory is gone. The source files on the host remain untouched.
shell
docker compose down
# Containers stopped → tmpfs mounts destroyed → secret data gone from memory
# Source files in ./secrets/ are still on the host — manage them carefully
✨What makes Docker secrets secure
🧠
Memory-only mount
Secret content lives on a tmpfs — RAM only. Never written to the container's writable layer or the host's disk during the container's lifetime.
🔒
0400 permissions
Files are read-only for the owner only. Other users in the same container cannot read them even if they have shell access.
🚪
Per-service access control
A secret not explicitly granted to a service is completely invisible to it — not mounted, not accessible, not even known to exist.
🔍
Absent from inspect
Secret values never appear in docker inspect, logs, or crash reports — only the mount path (not content) is recorded.
🗑️
Auto-cleanup on stop
When the container stops, the tmpfs is destroyed. No traces of the secret remain in the container filesystem.
📦
Image-independent
The same image runs in dev with test keys and in prod with live keys. Nothing about the secret is baked into any layer.
