Task 38  ·  Advanced

Container
Logging Drivers

Configure a container to forward logs to a file with rotation or an external system — moving beyond the default json-file driver that fills your disk.

~12 minutes Docker installed Terminal / Shell
📖 Required concepts
📜
What a logging driver is
Every container's stdout and stderr is captured by the Docker daemon and routed through a logging driver — a pluggable backend that decides where those bytes actually end up. Swapping the driver lets you send logs to rotating files, syslog, a remote aggregator, or anywhere else, without touching the application.
📁
The default: json-file
Docker defaults to the json-file driver, which writes every log line as a JSON object to a file under /var/lib/docker/containers/<id>/. Without limits it grows without bound — a busy container can fill a disk within hours on a production server.
♻️
Log rotation with json-file
Before switching drivers entirely, json-file supports two options: max-size caps each file and max-file sets how many rotated files to keep. This is the simplest production hardening step if you don't yet have a log aggregator.
🔌
local driver
The local driver is Docker's more efficient alternative to json-file. It writes in a compact binary format, rotates by default (5 files × 20MB), and is significantly faster at high log volumes — a good drop-in for most self-hosted setups.
🌐
External drivers
Docker ships built-in drivers for syslog, journald, gelf (Graylog), fluentd, awslogs, splunk, and more. Each has its own options and forwards logs over the network or into the host OS logging system — no agent container or sidecar required.
⚠️
docker logs needs json-file or local
docker logs only works when the container uses the json-file or local driver. Switch to syslog or fluentd and Docker can no longer show you a container's output — you must query the external system instead.
🔌 Built-in logging drivers
json-file
Default. JSON lines on disk. Use with max-size + max-file for safe self-hosting.
local
Recommended default. Binary format, auto-rotates, faster than json-file at scale.
syslog
Forwards to a local or remote syslog daemon (rsyslog, syslog-ng). Good for systemd hosts.
journald
Sends to the host's systemd journal. Query with journalctl.
fluentd
Streams to a Fluentd or Fluent Bit collector over TCP. Common in Kubernetes-adjacent setups.
awslogs
Ships directly to AWS CloudWatch Logs. Zero extra infrastructure if you're already on AWS.
gelf
Graylog Extended Log Format — used with Graylog, Logstash, or compatible receivers.
splunk
Sends to a Splunk HTTP Event Collector endpoint using a token you configure.
none
Discards all output. Useful for noisy containers whose logs you never need.
🔬 Command anatomy
docker run --log-driver json-file --log-opt max-size=10m --log-opt max-file=3 nginx
docker run

Create and start a container.

--log-driver

Selects the logging plugin for this container. Overrides the daemon-level default for this run only.

--log-opt

Passes a key=value option to the chosen driver. Repeat once per option. Each driver has its own set of supported options.

nginx

The image to run. Any image works — the logging config is orthogonal to what the container does.

🪜 Step-by-step guide
1
See the daemon's current default driver
Before changing anything, check what logging driver Docker is currently using by default.
shell 
docker info --format '{{.LoggingDriver}}'
expected output
json-file
2
Run a container with json-file rotation
The simplest production fix: keep the familiar driver but cap each log file at 10 MB and retain at most 3 rotated files — 30 MB maximum per container.
shell 
docker run -d --name rotated \
  --log-driver json-file \
  --log-opt max-size=10m \
  --log-opt max-file=3 \
  nginx
💡
Size values accept k, m, and g suffixes. 10m = 10 megabytes. When the active file hits 10 MB, Docker renames it and starts a fresh one, keeping the last 3.
3
Verify the driver config on the container
Inspect the container's logging configuration to confirm the driver and options were applied.
shell 
docker inspect --format '{{.HostConfig.LogConfig}}' rotated
expected output
{json-file map[max-file:3 max-size:10m]}
4
Switch to the local driver
Run a second container using the local driver — a faster, binary-format alternative that rotates automatically with no extra options needed.
shell 
docker run -d --name local-logs \
  --log-driver local \
  --log-opt max-size=20m \
  --log-opt max-file=5 \
  nginx
ℹ️
docker logs local-logs still works with the local driver — it's one of the two drivers the command supports.
5
Forward logs to syslog
Run a container that ships its logs to the host's syslog daemon. The address below targets a local UDP syslog receiver — adjust it for your environment.
shell 
docker run -d --name syslog-demo \
  --log-driver syslog \
  --log-opt syslog-address=udp://127.0.0.1:514 \
  --log-opt tag="nginx-{{.Name}}" \
  nginx
⚠️
This will fail if nothing is listening on port 514 on your host. On Linux you can start rsyslog with sudo systemctl start rsyslog first, or skip this step and continue with the daemon-level config below.
6
Suppress logs entirely with the none driver
Run a container whose output you never need — useful for background utilities that produce only noise.
shell 
docker run -d --name silent \
  --log-driver none \
  nginx
7
Set the daemon-level default in daemon.json
Instead of passing --log-driver to every docker run, set a global default so every new container inherits rotation automatically.
shell 
# Edit (or create) /etc/docker/daemon.json
sudo nano /etc/docker/daemon.json
json 
{
  "log-driver": "json-file",
  "log-opts": {
    "max-size": "10m",
    "max-file": "3"
  }
}
⚠️
The daemon must be restarted to pick up changes to daemon.json. On Linux: sudo systemctl restart docker. On Mac/Windows: restart Docker Desktop. This will stop all running containers.
8
Verify the new daemon default
After restarting the daemon, confirm the global default has changed.
shell 
docker info --format 'Logging Driver: {{.LoggingDriver}}'
expected output
Logging Driver: json-file
9
Confirm docker logs still works for the rotated containers
Generate some log output and verify the command can still read it from the json-file driver.
shell 
docker logs --tail 5 rotated
ℹ️
Try the same on silent and you'll see an error — docker logs is intentionally unsupported when the none driver is in use.
10
Clean up
Stop and remove all containers started in this task.
shell 
docker rm -f rotated local-logs silent
🔍 What happened under the hood
Step 01
📝
stdout captured
The daemon intercepts everything written to the container's stdout and stderr file descriptors.
Step 02
🔌
Driver selected
The daemon checks the container's log driver setting (per-container overrides the daemon default).
Step 03
📦
Message encoded
Each log line is wrapped with a timestamp and stream label before being handed to the driver.
Step 04
🗂️
Driver writes/ships
json-file writes to disk; syslog forwards over UDP/TCP; none discards; others send over the network.
Step 05
♻️
Rotation triggered
When the active file hits max-size, the daemon renames it and opens a fresh file, pruning old ones beyond max-file.
Step 06
🔎
docker logs reads back
For json-file and local only, the CLI reads the on-disk files and streams them back to your terminal.
📐 Common log-opt reference
Option Drivers What it does
max-size json-file, local Maximum size of each log file before rotation. e.g. 10m, 100k, 1g.
max-file json-file, local Number of rotated log files to keep. Oldest are pruned when this is exceeded.
tag syslog, gelf, fluentd Template string used to tag log entries. Supports {{.Name}}, {{.ID}}, etc.
syslog-address syslog Target syslog endpoint, e.g. udp://127.0.0.1:514 or unix:///dev/log.
syslog-facility syslog Syslog facility code (daemon, local0–local7, etc.) to classify the log source.
fluentd-address fluentd Host and port of the Fluentd or Fluent Bit collector, e.g. localhost:24224.
