Task 31  ·  Advanced  ·  Node.js variant

Multistage Builds
with Node.js

Build a Node.js Express app using all dev dependencies, run tests, then copy only production node_modules into a lean runtime image — dropping from ~600 MB to ~130 MB.

~25 minutes Task 18 completed Node.js + npm
📖Required concepts
📦
dependencies vs devDependencies
devDependencies — test frameworks, TypeScript, linters, bundlers — are needed only at build time. Production only needs dependencies. In Node.js, npm ci --omit=dev installs only production deps.
🔀
Three-stage Node.js pattern
Node.js apps benefit from three stages: a deps stage that installs all packages (including dev), a build stage that runs tests and compiles TypeScript, and a lean runtime stage that gets only prod node_modules + compiled output.
📋
COPY --from transfers files, not layers
When you COPY --from=deps /app/node_modules ./node_modules, only those files move to the runtime stage — none of the npm cache, intermediate layers, or .npm metadata from the deps stage comes along.
🪶
node:alpine vs node:slim
alpine variants use musl libc and are ~5 MB base. slim variants use glibc and are ~70 MB base but have better compatibility with native npm modules. Use alpine unless a native module fails — then switch to slim.
👤
Non-root USER in production
Node images ship with a built-in node user (UID 1000). Running as USER node in your runtime stage means that if an attacker exploits your app, they get a limited user — not root — on the container filesystem.
🧪
Running tests in a build stage
Because the build stage has all devDependencies available, it's the right place to run your test suite. If tests fail, docker build exits with a non-zero code — blocking the bad image from being produced at all.

Struck-through items never enter the runtime stage. Only prod node_modules and compiled output cross the boundary via COPY --from.

🔬Command anatomy
COPY --from=deps /app/node_modules ./node_modules
COPY

Standard Dockerfile instruction, extended with a cross-stage source flag.

--from

Reads from a previous stage's filesystem — not the build context on your host. Value is the stage name defined with AS deps.

src

Absolute path inside the deps stage where npm ci --omit=dev installed production packages.

dest

Where to place it in the current (runtime) stage — relative to WORKDIR.

🪜Step-by-step guide
1
Set up the project structure
Create a minimal Express + TypeScript app with a test file. This gives us all three concerns: devDependencies (TypeScript, Jest), production dependencies (Express), and compiled output.
shell
mkdir node-multistage && cd node-multistage
node-multistage/ ├── Dockerfile ├── .dockerignore ├── package.json ├── tsconfig.json └── src/ ├── app.ts └── app.test.ts
2
Create package.json
Notice the separation: express is a production dependency; typescript and jest are dev-only. This separation is what makes multistage shrinkage possible.
package.json
{
  "name": "node-multistage",
  "version": "1.0.0",
  "scripts": {
    "build": "tsc",
    "start": "node dist/app.js",
    "test":  "jest"
  },
  // Ships to production — kept in the runtime image
  "dependencies": {
    "express": "^4.19.2"
  },
  // Build-time only — stripped from the runtime image
  "devDependencies": {
    "@types/express": "^4.17.21",
    "@types/jest":    "^29.5.12",
    "@types/node":    "^20.12.7",
    "jest":           "^29.7.0",
    "ts-jest":        "^29.1.2",
    "typescript":     "^5.4.5"
  },
  "jest": {
    "preset": "ts-jest",
    "testEnvironment": "node"
  }
}
3
Create the app source and a test
A simple Express server in TypeScript, plus a Jest test. Having a real test lets us prove the build stage runs tests before producing the final image.
src/app.ts
import express, { Request, Response } from 'express';

export const app = express();

app.get('/', (_req: Request, res: Response) => {
  res.json({
    message: 'Hello from a multistage Node.js build!',
    version: process.env.APP_VERSION ?? '1.0.0',
  });
});

// Only start the server when run directly
if (require.main === module) {
  const PORT = process.env.PORT ?? 3000;
  app.listen(PORT, () =>
    console.log(`Server running on port ${PORT}`)
  );
}
src/app.test.ts
import request from 'supertest';
import { app } from './app';

describe('GET /', () => {
  it('returns a JSON message', async () => {
    const res = await request(app).get('/');
    expect(res.status).toBe(200);
    expect(res.body.message).toMatch(/multistage/i);
  });
});
tsconfig.json
{
  "compilerOptions": {
    "target":     "ES2020",
    "module":     "commonjs",
    "outDir":     "./dist",
    "rootDir":    "./src",
    "strict":     true,
    "skipLibCheck": true
  },
  "include": ["src"],
  "exclude": ["node_modules", "dist"]
}
4
Write the .dockerignore
Keep the build context clean — never send node_modules or dist from your host into the Docker daemon. The build installs them fresh inside the container.
.dockerignore
node_modules
dist
.git
*.log
.env*
5
Write the three-stage Dockerfile
This is the core of the task. Read through each stage and the comments — every decision has a reason.
dockerfile
# ── Stage 1: deps ─────────────────────────────────
# Install ONLY production dependencies here.
# We copy this node_modules into the runtime image.
FROM node:20-alpine AS deps

WORKDIR /app

# Copy manifests first — layer is cached until they change
COPY package.json package-lock.json ./

# --omit=dev skips devDependencies — lean prod modules only
RUN npm ci --omit=dev


# ── Stage 2: build ────────────────────────────────
# Install ALL deps (including dev), run tests, compile TS.
# This stage is large but is never shipped to production.
FROM node:20-alpine AS build

WORKDIR /app

COPY package.json package-lock.json ./

# Full install — includes jest, typescript, type defs, etc.
RUN npm ci

# Copy source code and config
COPY tsconfig.json ./
COPY src ./src

# Run tests — if any fail, docker build exits here
# and the runtime image is never produced
RUN npm test

# Compile TypeScript → dist/
RUN npm run build


# ── Stage 3: runtime ──────────────────────────────
# Minimal image: prod node_modules + compiled JS only.
# No TypeScript, no jest, no source .ts files.
FROM node:20-alpine AS runtime

WORKDIR /app

# Grab prod node_modules from the deps stage
COPY --from=deps  /app/node_modules ./node_modules

# Grab compiled JS from the build stage
COPY --from=build /app/dist        ./dist

# Copy package.json so Node can resolve the app name
COPY package.json ./

ENV  NODE_ENV=production
ENV  APP_VERSION=1.0.0

# Use the built-in non-root node user
USER   node
EXPOSE 3000
CMD    ["node", "dist/app.js"]
🔑
Two COPY --from lines do all the work: one pulls prod node_modules from deps, the other pulls compiled dist/ from build. The TypeScript source, test files, and devDependencies never enter the runtime stage.
6
Build the naive single-stage image first (baseline)
Before building the multistage version, create a naive single-stage image to get a size baseline to compare against.
dockerfile.naive
# Everything in one stage — devDeps stay in final image
FROM    node:20-alpine
WORKDIR /app
COPY    package.json package-lock.json ./
RUN     npm ci
COPY    . .
RUN     npm run build
EXPOSE  3000
CMD     ["node", "dist/app.js"]
shell
docker build -f Dockerfile.naive -t nodeapp:naive .
image size
nodeapp naive ~620 MB (node_modules full + devDeps inside)
7
Build the multistage image and watch tests run
Run the multistage build. In the output you'll see Jest execute inside the build stage — and if all tests pass, the build continues to produce the slim runtime image.
shell
docker build -t nodeapp:multistage .
expected build output (abridged)

Tests ran inside Docker during the build — not on your host. If a test fails here, the runtime stage never executes and no broken image is tagged.

9
Run and verify
Start a container from the multistage image and confirm the API responds correctly.
shell
docker run -d -p 3000:3000 --name nodeapp-ms nodeapp:multistage
curl http://localhost:3000
expected output
{"message":"Hello from a multistage Node.js build!","version":"1.0.0"}
10
Prove TypeScript is absent from the runtime image
Try to run tsc inside the running container — it should not exist.
shell
docker exec nodeapp-ms ./node_modules/.bin/tsc --version
expected output
OCI runtime exec failed: exec: "./node_modules/.bin/tsc":
stat ./node_modules/.bin/tsc: no such file or directory
✅
TypeScript doesn't exist in the runtime image — it never entered it. node_modules came from the deps stage which only ran npm ci --omit=dev.
11
Use --target to run only the build stage in CI
In a CI pipeline you can build just the test stage to validate code without producing the final image.
shell
# CI: build only up to the build stage (runs tests)
# If tests fail, exit non-zero → CI pipeline blocked
docker build --target build -t nodeapp:ci .

# CD: build the full image only after CI passes
docker build -t nodeapp:multistage .
ℹ️
Because Docker caches layers, the second docker build reuses everything from the first — the tests don't re-run, and only the runtime stage executes fresh.
12
Clean up
Stop and remove everything created during this task.
shell
docker stop nodeapp-ms
docker rm nodeapp-ms
docker rmi nodeapp:multistage nodeapp:naive nodeapp:ci
🔍What happened under the hood
Stage 01
📦
deps: prod install
npm ci --omit=dev produced a lean node_modules with only Express and its dependencies.
Stage 02
🧪
build: test + compile
Full npm ci brought in Jest and TypeScript. Tests ran, TS compiled to dist/.
Stage 03
🗑️
build discarded
Stages 1 and 2 filesystems are thrown away — only their artifacts are referenced via COPY --from.
Stage 04
📋
runtime assembled
Two cross-stage copies pulled prod node_modules and dist/ into a fresh Alpine base.
Stage 05
👤
non-root user set
USER node locked down the runtime — the app runs without root privileges.
Stage 06
✅
79% smaller
620 MB → 130 MB. No TypeScript, no Jest, no source .ts files in production.
🎯What you learned
  • Why separating dependencies from devDependencies in package.json is essential for multistage efficiency
  • The three-stage Node.js pattern: deps (prod install) → build (test + compile) → runtime (lean image)
  • How npm ci --omit=dev produces a production-only node_modules for the runtime stage
  • How running npm test inside the build stage gates the final image on passing tests
  • How two COPY --from instructions assemble the runtime from two different upstream stages
  • How USER node in the runtime stage hardens the container against privilege escalation
  • How --target build lets CI validate code without building the final image
