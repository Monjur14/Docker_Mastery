Task 32  ·  Advanced

Build Cache
Optimization

Reorder Dockerfile instructions strategically so frequently-changing layers — app code — come last. Turn multi-minute rebuilds into sub-second cache hits by understanding exactly how Docker invalidates layers.

~20 minutes Docker installed Terminal / Shell
📖 Required concepts
📦
Every instruction is a layer
Each RUN, COPY, and ADD instruction produces an immutable layer identified by a hash of its inputs. Docker caches each layer on disk so it can be reused across builds without re-executing the instruction.
🌊
Cache invalidation cascades downward
When a layer's cache is invalidated — because its instruction changed, or a file it COPYs was modified — every layer below it is also invalidated, even if those layers didn't change. This is the single most important rule: a miss at layer 3 forces a re-run of layers 4, 5, 6 … all the way to the end.
🧊
Change frequency determines layer order
The rule for ordering is simple: what changes least goes first, what changes most goes last. Base image and system packages almost never change. Dependency manifests (package.json) change occasionally. App source code changes on every save. Order them in that sequence.
🔍
How COPY cache works
For COPY instructions, Docker computes a checksum of the source files and compares it to the cached layer. If any file's checksum changes — even one byte — the cache for that layer misses, and all layers below it are invalidated.
📋
Dependency install split
The most impactful optimisation in most projects: COPY only the manifest first (package.json, requirements.txt, go.mod), run the install, then copy your source code. This way npm install is only re-run when dependencies actually change.
🚀
BuildKit and --no-cache
Modern Docker uses BuildKit by default, which adds parallel layer execution and smarter cache backends. Pass --no-cache to force a full rebuild from scratch — useful when verifying the cold-build time or flushing a stale cache. Use --progress=plain to see every layer's cache status.

🪜 Step-by-step guide
1
Create a minimal Node.js app to build
Set up a small project with a package.json and one source file — enough to demonstrate cache behaviour clearly.
shell 
mkdir cache-demo && cd cache-demo
package.json 
{
  "name": "cache-demo",
  "dependencies": { "express": "^4.18.0" },
  "scripts": { "start": "node index.js" }
}
index.js 
const express = require('express');
const app = express();
app.get('/', (req, res) => res.send('v1\n'));
app.listen(3000);
2
Write the naïve Dockerfile — the wrong order
This is what most beginners write first. It works, but it's slow: npm install re-runs on every single code change.
Dockerfile.slow 
FROM node:20-alpine
WORKDIR /app
COPY . .              # ← copies everything first
RUN npm install       # ← re-runs whenever ANY file changes
CMD ["node", "index.js"]
✗
Why this is slow: COPY . . includes every source file — so editing index.js invalidates that layer, which cascades into npm install. You re-download all dependencies on every rebuild, even when package.json hasn't changed.
3
Build the slow version twice — observe the difference
Build once to prime the cache, edit one line of index.js, then rebuild and watch how many layers are re-executed.
shell 
# First build — primes the cache
docker build -f Dockerfile.slow -t demo:slow --progress=plain .

# Edit one line of app code
sed -i 's/v1/v2/' index.js

# Rebuild — note which layers say CACHED vs re-running
docker build -f Dockerfile.slow -t demo:slow --progress=plain .
rebuild output — slow dockerfile
#1 [internal] load build definition
#2 [1/4] FROM node:20-alpine CACHED
#3 [2/4] COPY . . 0.1s
#4 [3/4] RUN npm install 18.4s ← re-runs every time
#5 [4/4] CMD 0.0s
4
Write the optimised Dockerfile — the right order
Split the copy into two stages: manifest first, install, then source. This way npm install is only re-triggered when package.json or package-lock.json changes.
Dockerfile.slow
FROM node:20-alpine
WORKDIR /app
COPY . .
RUN npm install
CMD ["node","index.js"]
Dockerfile.fast
FROM node:20-alpine
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
CMD ["node","index.js"]
Dockerfile.fast 
FROM node:20-alpine
WORKDIR /app

# 1. Copy ONLY the dependency manifests
COPY package*.json ./

# 2. Install — cached unless package*.json changes
RUN npm ci

# 3. Copy source code last — changes here don't bust the install cache
COPY . .

CMD ["node", "index.js"]
ℹ️
npm ci instead of npm install is the right choice in Docker — it installs from the lockfile exactly, errors on any mismatch, and produces a reproducible result every time.
5
Build the fast version twice — observe the speedup
Prime the cache with the first build, edit source code, then rebuild and confirm npm ci is now served from cache.
shell 
# First build — primes the cache (npm ci runs for real)
docker build -f Dockerfile.fast -t demo:fast --progress=plain .

# Edit source code only
sed -i 's/v2/v3/' index.js

# Rebuild — npm ci should now say CACHED
docker build -f Dockerfile.fast -t demo:fast --progress=plain .
rebuild output — fast dockerfile
#1 [internal] load build definition
#2 [1/5] FROM node:20-alpine CACHED
#3 [2/5] COPY package*.json ./ CACHED
#4 [3/5] RUN npm ci CACHED ← skipped entirely
#5 [4/5] COPY . . 0.1s
#6 [5/5] CMD 0.0s
Build completed in 0.4s
💡
The 18-second npm ci is now a 0-millisecond cache hit. On a project with hundreds of dependencies this difference is measured in minutes per build.
6
Verify the cache invalidates correctly when deps change
Confirm that adding a new dependency does bust the cache — the optimised Dockerfile should still behave correctly, not just fast.
shell 
# Add a dependency — this changes package.json
npm install --save lodash --package-lock-only

# Rebuild — npm ci should now re-run
docker build -f Dockerfile.fast -t demo:fast --progress=plain .
expected output
#2 [1/5] FROM node:20-alpine CACHED
#3 [2/5] COPY package*.json ./ 0.1s ← manifest changed
#4 [3/5] RUN npm ci 12.3s ← correctly re-runs
#5 [4/5] COPY . . 0.1s
✅
The cache invalidates at exactly the right layer — COPY package*.json — and re-runs npm ci. This is correct behaviour, not a bug.
7
See the cache hit rate per layer in a table
Here's what happens to each layer across the three scenarios you just tested. This is the mental model to carry into every Dockerfile you write.
layer instruction code change dep change
1 FROM node:20-alpine hit hit
2 COPY package*.json ./ hit miss
3 RUN npm ci hit re-run
4 COPY . . miss re-run
5 CMD ["node", "index.js"] re-run re-run
ℹ️
When only code changes, layers 1–3 are cache hits — including the slow npm ci. Only when dependencies change does npm ci re-run. Both cases are correct.
8
Apply the same pattern to Python and Go
The manifest-first split works in every language — only the file names differ.
Dockerfile — Python / pip 
FROM python:3.12-slim
WORKDIR /app
COPY requirements.txt .          # manifest first
RUN pip install --no-cache-dir -r requirements.txt
COPY . .                           # source last
CMD ["python", "app.py"]
Dockerfile — Go / modules 
FROM golang:1.22-alpine
WORKDIR /app
COPY go.mod go.sum ./             # manifest first
RUN go mod download
COPY . .                           # source last
RUN go build -o server .
CMD ["./server"]
9
Add a .dockerignore to stop irrelevant files busting the cache
Without .dockerignore, the COPY . . layer will miss its cache any time a .git file, test result, or editor temp file changes — even though those files aren't part of the image.
.dockerignore 
# Never send these to the build context
node_modules
.git
.gitignore
*.md
.env
dist
coverage
.DS_Store
*.log
💡
A good .dockerignore does two things: speeds up context transfer to the daemon, and prevents transient files from causing spurious cache misses on COPY . ..
10
Measure total build time with --no-cache vs cached
Use time to get a concrete before-and-after number — useful for convincing teammates that the ordering matters.
shell 
# Cold build — no cache at all
time docker build --no-cache -f Dockerfile.fast -t demo:fast .

# Edit source, then warm rebuild
sed -i 's/v3/v4/' index.js
time docker build -f Dockerfile.fast -t demo:fast .
typical timing results
Cold build (--no-cache): real 0m22.4s
Warm rebuild (code only): real 0m0.6s
Speedup: ~37×
🔍 What happened under the hood
Step 01
🔑
Layer key computed
For each instruction Docker computes a cache key: the instruction text plus a checksum of any files it COPYs.
Step 02
🗄️
Cache lookup
Docker checks its local layer store. If the key matches an existing layer, that layer is returned instantly — no execution.
Step 03
💥
Miss propagates
On a miss, the instruction executes and produces a new layer. All subsequent layers are also executed, since their parent hash changed.
Step 04
📋
Manifest checksum
COPY package*.json hashes only those two files. If they haven't changed, the checksum matches and npm ci is never touched.
Step 05
📁
Context filtering
.dockerignore strips excluded files before computing checksums — so editor temps and .git changes can't pollute the cache key.
Step 06
⚡
BuildKit parallelism
BuildKit can execute independent branches of a multi-stage build in parallel, compounding the savings from cache hits.
