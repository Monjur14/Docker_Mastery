Task 13  ·  Beginner

Map Multiple Ports
in One Container

Run a container that exposes several ports simultaneously using repeated -p flags, then verify each one is reachable from the host.

~8 minutes Docker installed Terminal / Shell
📖 Required concepts
🔌
Port mapping is not port exposure
EXPOSE in a Dockerfile is purely documentation — it tells users which ports the process inside listens on, but doesn't actually make them reachable. Publishing with -p at docker run is what actually opens a host-side door and routes traffic into the container.
🗺️
The host:container syntax
Each -p flag takes the form host_port:container_port. Traffic arriving on the host at host_port gets NAT'd into the container on container_port. The host port and container port do not need to match.
🔁
Repeating -p for multiple ports
There is no multi-port shorthand in a single -p. To publish several ports you simply repeat the flag once per port mapping. Docker applies each one independently, and they can map to any available host port you choose.
🌐
Binding to a specific interface
The full syntax is ip:host_port:container_port. Omitting the IP binds to 0.0.0.0 (all interfaces). Use 127.0.0.1:host_port:container_port to restrict a port to localhost only — useful for admin ports you don't want exposed to the network.
🎲
Random host port assignment
Using -p container_port (no host port) lets Docker pick a random available host port for you. Useful when running many containers of the same image to avoid conflicts. Check which port was assigned with docker port.
🛠️
docker port for quick lookups
docker port container_name prints the complete port mapping table for a running container — faster than parsing docker inspect when you just want to know what's exposed where.
🔬 Command anatomy
docker run -p 8080:80 -p 8443:443 nginx
docker

The CLI binary — how you talk to the Docker daemon from your terminal.

run

Create and start a new container from an image.

-p 8080:80

Publish port: route traffic from host port 8080 into the container on port 80.

-p 8443:443

A second, independent mapping: host port 8443 → container port 443. Repeat -p once per port.

nginx

The image to run.

ℹ️
Host and container ports don't need to match — 8080→80 is perfectly valid. Same-number mappings like 9000→9000 are just a common convention for clarity.
🪜 Step-by-step guide
1
Run a container with three port mappings
Use the hashicorp/http-echo image — a tiny server that responds on any port with a message you define. Start it with three different mappings in detached mode.
shell 
docker run -d --name multiport \
  -p 8080:8080 \
  -p 8081:8081 \
  -p 8082:8082 \
  hashicorp/http-echo \
  -listen=:8080 -text="port 8080 works"
ℹ️
http-echo listens on a single port per process. Steps 2 and 3 will start additional containers on the other ports to give you three independent services to verify.
2
Run Nginx with two ports mapped
A more realistic example: run Nginx mapping both port 80 (HTTP) and port 443 (HTTPS) from the container to different host ports.
shell 
docker run -d --name webserver \
  -p 8088:80 \
  -p 4443:443 \
  nginx
3
Confirm both containers are running
The PORTS column in docker ps lists every published mapping — a quick sanity check before you start hitting the ports.
shell 
docker ps
expected output
CONTAINER ID NAME IMAGE PORTS
a1b2c3d4e5f6 webserver nginx 0.0.0.0:8088->80/tcp, 0.0.0.0:4443->443/tcp
f6e5d4c3b2a1 multiport hashicorp/http-echo 0.0.0.0:8080->8080/tcp
4
Look up mappings for a single container
Use docker port to list just the port table for webserver.
shell 
docker port webserver
expected output
80/tcp -> 0.0.0.0:8088
443/tcp -> 0.0.0.0:4443
5
Verify port 8088 is reachable
Hit the Nginx HTTP port on the host and confirm it responds with the default welcome page.
shell 
curl -s -o /dev/null -w "%{http_code}" http://localhost:8088
expected output
200
💡
No curl? Open http://localhost:8088 in your browser instead — you should see the Nginx welcome page.
6
Verify port 8080 on the echo server
Hit the http-echo container and see the message it was configured to return.
shell 
curl http://localhost:8080
expected output
port 8080 works
7
Bind a port to localhost only
Prefix the host IP to restrict a port so only processes on the same machine can reach it — not external network clients.
shell 
docker run -d --name local-only \
  -p 127.0.0.1:9090:80 \
  -p 127.0.0.1:9443:443 \
  nginx
⚠️
Without the 127.0.0.1: prefix, Docker binds to 0.0.0.0, meaning the port is reachable from any network interface — including external ones. Always restrict admin or debug ports to localhost in production.
8
Let Docker assign a random host port
Omit the host port entirely and Docker picks a free one for you. Use docker port to find out which.
shell 
docker run -d --name random-port -p 80 nginx
docker port random-port
expected output
80/tcp -> 0.0.0.0:49153
9
Clean up all containers from this task
Stop and remove every container started in this task in one command.
shell 
docker rm -f multiport webserver local-only random-port
ℹ️
The -f flag forces removal of running containers — combining stop and remove into one step.
📐 Port flag syntax reference
Flag What it does
-p 8080:80 Maps host port 8080 to container port 80 on all host interfaces.
-p 80 Maps container port 80 to a random available host port. Check with docker port.
-p 127.0.0.1:8080:80 Maps to host port 8080 but localhost only — not reachable externally.
-P Publishes all ports declared with EXPOSE in the Dockerfile to random host ports.
🎯 What you learned
  • That EXPOSE is documentation only — -p at run time is what actually publishes a port
  • How to publish multiple ports by repeating the -p flag once per mapping
  • The full ip:host_port:container_port syntax and when each part matters
  • How to restrict a port to localhost only with the 127.0.0.1: prefix
  • How to let Docker assign a random host port and look it up with docker port
  • How to force-remove running containers in one step with docker rm -f
