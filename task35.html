Task 35  ·  Advanced

Non-root User
in Dockerfile

Create a dedicated system user in your Dockerfile and switch to it with USER so your app never runs as root — limiting the blast radius if a vulnerability is ever exploited.

~15 minutes Docker installed Familiarity with Dockerfiles
📖Required concepts
🚨
Containers run as root by default
Unless told otherwise, every process inside a Docker container runs as UID 0 — root. That means your Node.js server, your Python worker, your entire app — all root. If an attacker exploits a vulnerability in your app, they immediately have root inside the container.
🧱
Container root ≠ host root — but almost
Docker's namespaces isolate the container, so root inside a container isn't automatically root on the host. But kernel vulnerabilities, misconfigured volume mounts, and --privileged mode can all let container root escape to become host root. Running non-root eliminates this entire category of risk.
👤
The USER instruction
USER sets the user (and optionally group) for all subsequent RUN, CMD, and ENTRYPOINT instructions in the Dockerfile. It takes a username, a UID, or both. Everything before the USER line still runs as root.
📂
File ownership must be set before switching
The new user can only read and write files it owns or has permission to access. If you copy app files as root and then switch users, the app can't write to its own working directory. You must chown the relevant directories to the new user before the USER switch.
🔢
Use high UIDs for compatibility
System UIDs below 1000 are often reserved on Linux. Using a high UID like 10001 avoids collisions with system accounts on the host, and works correctly with Kubernetes security policies that commonly enforce non-root by rejecting UIDs below a threshold.
🚫
Some things still need root
Binding to ports below 1024 (like port 80) requires root or the NET_BIND_SERVICE capability. The solution is to bind your app to a high port (like 3000) and let Docker's port publishing (-p 80:3000) map it — no root needed inside the container.
⚖️Root vs non-root
Running as root
Can read/write any file in the container filesystem
Can install packages, modify system files mid-runtime
Container escape may grant host root privileges
Rejected by many Kubernetes PodSecurityAdmission policies
Violates principle of least privilege
Running as non-root
Can only access files it owns or has explicit permission to
Cannot install packages or alter system files
Container escape limited to unprivileged host user
Compatible with strict Kubernetes security contexts
Passes most container security scanners
🔬Instruction anatomy
RUN addgroup -g 10001 -S appgroup && adduser -u 10001 -S -G appgroup appuser
RUN

Runs during image build as root — fine here because we're creating a user, not running app code.

addgroup

Creates the group. On Alpine, addgroup; on Debian/Ubuntu use groupadd.

-g 10001

Assign GID 10001 — a high number that avoids clashing with system groups on the host.

-S

Create a system group (no login shell, no home directory, minimal footprint).

adduser

Creates the user. Alpine uses adduser -S; Debian/Ubuntu use useradd --system --no-create-home.

-u 10001

Assign UID 10001. Using a numeric UID (not just a name) ensures identity is consistent even if /etc/passwd isn't present.

appuser

The username. Short, descriptive, tied to the app — not a generic name like nobody which is shared across the system.

🪜Step-by-step guide
1
Prove the default is root
Run any image without a USER instruction and check who owns the process. The answer is almost always root.
shell
docker run --rm node:22-alpine whoami
expected output
root
🚨
This is the default for almost every official image. It means every RUN step in your Dockerfile and your app process at runtime both execute as UID 0 unless you explicitly change it.
2
Write a Dockerfile with a non-root user (Alpine)
The pattern: install deps as root → create user/group → chown app files to the new user → switch with USER → run the app. Order matters.
Dockerfile — Node.js on Alpine
FROM node:22-alpine

# 1. Create a dedicated group and user (still running as root here)
RUN addgroup -g 10001 -S appgroup && \
    adduser  -u 10001 -S -G appgroup appuser

WORKDIR /app

# 2. Install dependencies as root (needs write access to node_modules)
COPY package.json .
RUN npm install --omit=dev

# 3. Copy app source
COPY app.js .

# 4. Transfer ownership of /app to the new user BEFORE switching
RUN chown -R appuser:appgroup /app

# 5. Switch — everything below here runs as appuser
USER appuser

EXPOSE 3000

CMD ["node", "app.js"]
ℹ️
Steps 1–4 run as root because they need elevated privileges — creating users, installing packages, setting ownership. Step 5's USER drops privileges permanently for everything that follows, including the final CMD.
3
The same pattern for Debian/Ubuntu base images
Alpine and Debian use different user-creation commands. Here's the translation for images like node:22 (Debian) or python:3.12.
Dockerfile — Node.js on Debian
FROM node:22

# Debian uses groupadd / useradd
RUN groupadd --gid 10001 --system appgroup && \
    useradd  --uid 10001 --system \
             --gid appgroup \
             --no-create-home \
             --shell /sbin/nologin \
             appuser

WORKDIR /app

COPY package.json .
RUN npm install --omit=dev

COPY app.js .

RUN chown -R appuser:appgroup /app

USER appuser

EXPOSE 3000
CMD ["node", "app.js"]
💡
--no-create-home and --shell /sbin/nologin ensure the account has no home directory and can never be used to log in interactively — reducing attack surface further.
4
Build and verify
Build the image and confirm the running process is no longer root.
shell
# Build the image
docker build -t myapp:secure .

# Check the user at runtime
docker run --rm myapp:secure whoami
docker run --rm myapp:secure id
expected output
appuser
uid=10001(appuser) gid=10001(appgroup) groups=10001(appgroup)
💡
You can also inspect the image config without running a container: docker inspect myapp:secure --format '{{.Config.User}}' — it should return appuser.
5
Understand the permission boundaries
With non-root in place, verify what the user can and cannot touch — this is the security guarantee you just gained.
shell
# Can write to /app (we chown'd it)
docker run --rm myapp:secure \
  sh -c "touch /app/test.txt && echo 'write OK'"

# Cannot write outside /app
docker run --rm myapp:secure \
  sh -c "touch /etc/hacked 2>&1 || echo 'write denied — good'"

# Cannot install packages
docker run --rm myapp:secure \
  sh -c "apk add curl 2>&1 || echo 'install denied — good'"
expected output
write OK
write denied — good
install denied — good
Pathappuser can read?appuser can write?
/app yes yes — we chown'd it
/tmp yes yes — world-writable by default
/etc, /usr, /bin yes no — owned by root
/root no no — root's home dir, mode 700
Mounted volumes depends depends — host ownership rules apply
6
The optimised pattern with COPY --chown
Instead of a separate RUN chown layer, use the --chown flag directly on the COPY instruction. This sets ownership at copy time — fewer layers, smaller image.
Dockerfile — before (separate chown layer)
FROM node:22-alpine RUN addgroup -g 10001 -S appgroup && adduser -u 10001 -S -G appgroup appuser WORKDIR /app COPY package.json . RUN npm install --omit=dev COPY app.js . RUN chown -R appuser:appgroup /app # extra layer USER appuser
Dockerfile — after (--chown on COPY)
FROM node:22-alpine RUN addgroup -g 10001 -S appgroup && adduser -u 10001 -S -G appgroup appuser WORKDIR /app COPY --chown=appuser:appgroup package.json . # owned immediately RUN npm install --omit=dev COPY --chown=appuser:appgroup app.js . # owned immediately USER appuser
💡
--chown also accepts numeric IDs: COPY --chown=10001:10001 . . — useful in multi-stage builds where the user may not be defined in the earlier stage being copied from.
7
Use numeric UID in USER for maximum portability
Kubernetes and many container runtimes prefer a numeric UID in the USER instruction — it works even if /etc/passwd is absent in a scratch or distroless base image.
Dockerfile snippet
# Username is readable; numeric UID is portable
USER 10001

# Or set both user and group explicitly
USER 10001:10001
8
Combining USER with a healthcheck
The HEALTHCHECK command also runs as the user set by USER. Confirm curl is installed before the switch, and the healthcheck works without root.
Dockerfile — USER + HEALTHCHECK together
FROM node:22-alpine

# Install curl while still root (system-level package)
RUN apk add --no-cache curl

RUN addgroup -g 10001 -S appgroup && \
    adduser  -u 10001 -S -G appgroup appuser

WORKDIR /app
COPY --chown=appuser:appgroup package.json .
RUN npm install --omit=dev
COPY --chown=appuser:appgroup app.js .

# Switch before HEALTHCHECK and CMD
USER appuser

# healthcheck runs as appuser — curl is available, no root needed
HEALTHCHECK --interval=15s --timeout=5s --start-period=10s --retries=3 \
  CMD curl -f http://localhost:3000/health || exit 1

EXPOSE 3000
CMD ["node", "app.js"]
9
Override the user at runtime when needed
Sometimes you need to run a one-off command as root inside a container — for debugging or an admin task. You can override the image's USER at runtime without rebuilding.
shell
# Override user for a one-off debug shell
docker run --rm -it --user root myapp:secure sh

# Or use a numeric UID
docker run --rm --user 0 myapp:secure whoami
⚠️
Never use --user root in production or CI pipelines — it defeats the purpose entirely. Reserve runtime overrides for local debugging only.
10
Scan the image for root-user violations
Trivy, Docker Scout, and Snyk all flag containers running as root. Run a quick scan to confirm your image passes the check that previously would have failed.
shell — Docker Scout (built into Docker Desktop)
docker scout cves myapp:secure
docker scout recommendations myapp:secure
shell — Trivy (open source, standalone)
trivy image myapp:secure
scout / trivy — look for this in output
✓ Recommended tag: node:22-alpine
✓ Non-root user: appuser (uid=10001)
✓ No critical CVEs found
🔍What happened under the hood
Step 01
🔨
Build as root
All RUN steps before USER execute as UID 0 — needed to create accounts and install packages.
Step 02
👤
User created
An entry is written to /etc/passwd and /etc/group inside the image layer.
Step 03
🔑
Ownership set
COPY --chown writes inode ownership metadata — UID/GID stored in each file's inode on the image layer.
Step 04
🔒
USER recorded
The USER instruction writes the UID into the image config JSON — read by the runtime when starting the container.
Step 05
🚀
Runtime applies UID
containerd/runc reads the config and sets the UID before launching the container's PID 1. No capability to elevate to root.
Step 06
🛡️
Kernel enforces limits
Linux's DAC (Discretionary Access Control) denies any write to files not owned by UID 10001 — kernel-level enforcement, not Docker magic.
