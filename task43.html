Task 43  ·  Advanced

Multi-architecture Builds
with Buildx

Use docker buildx to build a single image that runs natively on both amd64 (Intel/AMD servers) and arm64 (Apple Silicon, AWS Graviton, Raspberry Pi) — then push a multi-arch manifest so Docker picks the right one automatically.

~20 minutes Docker Desktop / Engine Docker Hub account
📖 Required concepts
🏗️
Why multi-arch images matter
A regular image built on your Intel laptop only runs on amd64. Pull it onto an Apple M-series Mac or an AWS Graviton server and Docker emulates it in software — slow and sometimes broken. A multi-arch image ships a native binary for each platform under one tag, and Docker picks the right one automatically at pull time.
🔨
What is Buildx?
Buildx is Docker's extended build CLI, bundled with Docker Desktop and Engine since v20.10. It uses BuildKit under the hood and adds cross-platform builds, build caching to external registries, and multiple builder instances. The command is docker buildx build.
🗂️
OCI image manifest list
A multi-arch image is actually a manifest list (also called an image index) stored in the registry. It's a JSON document that maps each os/arch combination to a specific image digest. The tag you push (myapp:latest) points at the manifest list, not at any single image.
⚡
QEMU emulation
To build an arm64 image on an amd64 machine, BuildKit uses QEMU — a CPU emulator registered as a binary format handler in the Linux kernel. Your RUN instructions execute inside an emulated arm64 environment. It's slower than native but produces a correct binary. Docker Desktop sets this up automatically.
🏭
Builder instances
Buildx uses builder instances — named BuildKit daemons that can target different platforms. The default builder only supports your host platform. You need to create a new builder using the docker-container driver, which runs BuildKit inside a container and supports cross-platform builds out of the box.
📤
--push vs. --load
Multi-arch images cannot be loaded into the local Docker daemon — the local store only holds single-platform images. You must use --push to write the manifest list directly to a registry. Use --load only when building a single platform for local testing.
🗺️ How a multi-arch manifest works

One tag in the registry points to a manifest list. Docker inspects the pulling machine's os/arch and fetches only the matching image — no extra steps needed by the user.

🔬 Command anatomy
docker buildx build --platform linux/amd64,linux/arm64 -t yourname/myapp:latest --push .
docker buildx

The extended build CLI. Uses BuildKit and supports cross-platform builds, external caches, and multiple builder drivers.

build

Build an image from a Dockerfile and context. Same as docker build but with Buildx capabilities.

--platform

Comma-separated list of os/arch targets. BuildKit builds each one, either natively or via QEMU emulation.

-t

Tag applied to the manifest list in the registry. Both platform variants live under this single tag.

--push

Push the completed manifest list and all platform images directly to the registry. Required for multi-arch — --load only works with a single platform.

.

Build context — the current directory sent to BuildKit. The Dockerfile is read from here.

🪜 Step-by-step guide
1
Confirm Buildx is available
Buildx ships with Docker Desktop and modern Docker Engine. Verify it's present and check which builder is currently active.
shell 
docker buildx version
docker buildx ls
expected output
github.com/docker/buildx v0.14.0 ...

NAME/NODE DRIVER/ENDPOINT STATUS BUILDKIT PLATFORMS
default * docker running v0.13.0 linux/amd64
default default running v0.13.0 linux/amd64
⚠️
The default builder only supports your host platform. The next step creates a new builder with the docker-container driver that supports cross-platform builds.
2
Install QEMU emulators (Linux only)
Docker Desktop on Mac and Windows already includes QEMU. On a Linux host, register the QEMU binary handlers with the kernel so it can execute foreign-architecture binaries during builds.
shell — Linux hosts only 
docker run --privileged --rm \
  tonistiigi/binfmt --install all
expected output
installing: arm64 OK
installing: arm OK
installing: riscv64 OK
... (several more architectures)
ℹ️
This registers QEMU interpreters in /proc/sys/fs/binfmt_misc. The registration persists until reboot — you'll need to re-run this after a system restart on Linux.
3
Create a multi-platform builder instance
Create a new named builder using the docker-container driver, which runs BuildKit in a container and supports cross-compilation. Set it as the active builder.
shell 
# Create a new builder with the container driver
docker buildx create \
  --name multiarch \
  --driver docker-container \
  --bootstrap

# Set it as the active builder for this session
docker buildx use multiarch

# Confirm it supports both target platforms
docker buildx inspect --bootstrap
expected output (inspect)
Name: multiarch
Driver: docker-container
Status: running
Platforms: linux/amd64, linux/amd64/v2, linux/amd64/v3, linux/arm64, linux/arm/v7, linux/arm/v6 ...
💡
--bootstrap starts the BuildKit container immediately so you can inspect it right away, rather than lazily on first build. You only need to create this builder once.
4
Write a simple app and Dockerfile to build
A tiny Go HTTP server is the ideal test — it cross-compiles to native binaries for any platform without QEMU emulation, making builds fast. If you have your own app, use that instead.
shell 
mkdir multiarch-demo && cd multiarch-demo
main.go 
package main

import (
    "fmt"
    "net/http"
    "runtime"
)

func main() {
    http.HandleFunc("/", func(w http.ResponseWriter, r *http.Request) {
        fmt.Fprintf(w, "arch=%s os=%s\n", runtime.GOARCH, runtime.GOOS)
    })
    http.ListenAndServe(":8080", nil)
}
Dockerfile 
# Build stage — Go cross-compiles natively, no QEMU needed
FROM --platform=$BUILDPLATFORM golang:1.22-alpine AS build
WORKDIR /src
COPY . .
# TARGETOS and TARGETARCH are injected by Buildx automatically
ARG TARGETOS TARGETARCH
RUN GOOS=$TARGETOS GOARCH=$TARGETARCH go build -o /app .

# Runtime stage — runs on the target architecture
FROM scratch
COPY --from=build /app /app
EXPOSE 8080
ENTRYPOINT ["/app"]
ℹ️
--platform=$BUILDPLATFORM on the build stage tells Docker to always run the Go compiler on your host architecture — fast native compilation. $TARGETOS and $TARGETARCH are Buildx-injected build args that tell Go which platform to compile for.
5
Log in to Docker Hub
You need to push to a registry — authenticate first. Replace yourname with your actual Docker Hub username throughout this task.
shell 
docker login
💡
Use a Personal Access Token (PAT) instead of your password — generate one at hub.docker.com → Account Settings → Security. PATs can be scoped to read/write and revoked individually.
6
Build and push the multi-arch image
One command builds both platform variants and pushes the manifest list to Docker Hub. Buildx compiles them in parallel where possible.
shell 
docker buildx build \
  --platform linux/amd64,linux/arm64 \
  -t yourname/multiarch-demo:latest \
  --push \
  .
expected output
[+] Building 18.3s (16/16) FINISHED
=> [linux/amd64] FROM golang:1.22-alpine 2.1s
=> [linux/arm64] FROM golang:1.22-alpine 2.1s
=> [linux/amd64] COPY . . 0.1s
=> [linux/arm64] COPY . . 0.1s
=> [linux/amd64] RUN GOOS=linux GOARCH=amd64 go build ... 4.2s
=> [linux/arm64] RUN GOOS=linux GOARCH=arm64 go build ... 4.8s
=> pushing manifest list: yourname/multiarch-demo:latest 1.3s
7
Verify the manifest list in the registry
Inspect the pushed tag to confirm it's a manifest list containing both platform digests — not a single-platform image.
shell 
docker buildx imagetools inspect yourname/multiarch-demo:latest
expected output
Name: docker.io/yourname/multiarch-demo:latest
MediaType: application/vnd.oci.image.index.v1+json
Digest: sha256:abc123...

Manifests:
Name: docker.io/yourname/multiarch-demo:latest@sha256:a1b2c3...
MediaType: application/vnd.oci.image.manifest.v1+json
Platform: linux/amd64

Name: docker.io/yourname/multiarch-demo:latest@sha256:d4e5f6...
MediaType: application/vnd.oci.image.manifest.v1+json
Platform: linux/arm64
✅
MediaType: application/vnd.oci.image.index.v1+json confirms this is a manifest list, not a plain image. Both linux/amd64 and linux/arm64 platform entries are present.
8
Pull and run — confirm the right arch is selected
Pull the image on your local machine. Docker reads your host's architecture and fetches the matching platform variant automatically.
shell 
docker run --rm -p 8080:8080 -d --name test yourname/multiarch-demo:latest
curl http://localhost:8080
docker rm -f test
expected output on an arm64 Mac (Apple Silicon)
arch=arm64 os=linux
expected output on an amd64 Linux server
arch=amd64 os=linux
ℹ️
To explicitly pull a specific platform — for testing, or when running on a different arch — add --platform linux/amd64 to the docker run command.
9
Target platforms reference
Common platform strings for --platform. You can target any subset — start with amd64 and arm64 for the most coverage.
linux/amd64 x86-64. Standard Intel/AMD — VPS servers, most CI runners, Intel Macs running Rosetta.
linux/arm64 AArch64. Apple Silicon (M1–M4), AWS Graviton 2/3, Oracle Ampere, Raspberry Pi 4/5.
linux/arm/v7 32-bit ARM. Raspberry Pi 2/3 running 32-bit OS, older embedded boards.
linux/arm/v6 ARMv6. Original Raspberry Pi (Pi 1, Pi Zero, Pi Zero W).
linux/riscv64 RISC-V 64-bit. Emerging open-source architecture — SiFive, StarFive boards.
linux/s390x IBM Z (mainframe). Required for images published to enterprise registries.
10
Push additional tags and version the manifest
In production, push a version tag alongside latest so previous manifests are preserved and you can roll back by tag.
shell 
# Push two tags in one build — latest and a semver
docker buildx build \
  --platform linux/amd64,linux/arm64 \
  -t yourname/multiarch-demo:latest \
  -t yourname/multiarch-demo:1.0.0 \
  --push \
  .
💡
Each additional -t creates a separate manifest list pointing at the same image digests — no extra bytes are stored, just extra pointers. This is free to do and the right production habit.
11
Clean up the builder instance
When you're done, switch back to the default builder and optionally remove the multiarch one.
shell 
# Switch back to the default builder
docker buildx use default

# Remove the multiarch builder (stops its container)
docker buildx rm multiarch
ℹ️
In a CI environment, create the builder at the start of the pipeline and let it be torn down with the runner. For local development, keep the multiarch builder around — it starts and stops quickly and caches layers between builds.
🔍 What happened under the hood
Step 01
🏭
BuildKit spawned
The docker-container driver started a BuildKit daemon in its own container, capable of cross-platform builds.
Step 02
⚡
Parallel builds
BuildKit ran the amd64 and arm64 build graphs concurrently — shared base layers were pulled once and reused for both.
Step 03
🔄
Cross-compilation
The build stage ran on the host CPU. GOOS and GOARCH directed the Go compiler to emit binaries for the target platform, not the host.
Step 04
📤
Layers pushed
Each platform's image layers were pushed to the registry separately, then the manifest list was written pointing to both digests.
Step 05
🗂️
Index written
The registry stored the OCI image index — a JSON document mapping os/arch keys to image digest values.
Step 06
🎯
Auto-selection on pull
Docker's daemon read the host's os/arch, queried the index, and fetched only the matching platform's layers — transparent to the user.
