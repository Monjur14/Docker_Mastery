📖 Required concepts
📄
What is docker-compose.yml?
A YAML file that declaratively describes your entire stack — services, networks, volumes, environment variables, port mappings, and build instructions. Docker Compose reads it and translates every key into the Docker API calls you were making by hand. It lives in version control alongside your code.
🌐
Automatic networking
Compose creates a default network named <project>_default automatically, and connects every service to it. Services reach each other by their service name — no manual docker network create required.
🏷️
Project name & namespacing
Compose prefixes every resource with the project name (defaulting to the folder name). A network becomes myapp_default, a volume becomes myapp_mongo-data. This lets multiple stacks coexist on one machine without name collisions.
⏳
depends_on vs. readiness
depends_on controls start order — Compose won't start the app service until Mongo's container is running. It does not wait for Mongo to be ready to accept connections. That's why the Node app still needs its retry loop (or a healthcheck, covered in Task 34).
🏗️
build vs. image
A service can either build from a local Dockerfile (the Node app) or pull a ready-made image from a registry (MongoDB). Compose handles both in the same file — build: . triggers a local build, image: mongo:7 pulls from Docker Hub.
🔄
Idempotency
Running docker compose up multiple times is safe — Compose only creates resources that don't already exist, and only recreates containers whose config has changed. This makes it safe to re-run after editing the compose file.
🔁 Manual commands → Compose keys

Every flag from Task 27 maps directly to a YAML key. Nothing is lost — it's all still there, just declared instead of imperative.

📄 The docker-compose.yml file

Every line below maps to something you typed by hand in Task 27. Comments explain each key as you read.

docker-compose.yml 
# ── Services ────────────────────────────────────────────────
# Each top-level key under 'services' becomes one container.
services:

  # ── MongoDB ─────────────────────────────────────────────────
  mongo:
    image: mongo:7               # pull from Docker Hub
    container_name: mongo        # fixes the name (optional but nice)
    restart: unless-stopped      # restart on crash, not on manual stop
    environment:
      MONGO_INITDB_DATABASE: mydb
    volumes:
      - mongo-data:/data/db       # named volume → /data/db inside container
    networks:
      - app-network

  # ── Node.js API ─────────────────────────────────────────────
  app:
    build: .                       # build from Dockerfile in current dir
    container_name: node-app
    restart: unless-stopped
    ports:
      - "3000:3000"                # host:container — same as -p 3000:3000
    environment:
      PORT: 3000
      MONGO_URI: mongodb://mongo:27017/mydb  # 'mongo' resolves via Docker DNS
    depends_on:
      - mongo                      # start mongo container first
    networks:
      - app-network

# ── Networks ─────────────────────────────────────────────────
# Declaring it explicitly lets you give it a custom name.
networks:
  app-network:
    driver: bridge

# ── Volumes ──────────────────────────────────────────────────
# Top-level declaration creates the volume if it doesn't exist.
# Omitting the value ({}) uses Docker's default local driver.
volumes:
  mongo-data:
📐
YAML indentation is strict. Use 2 spaces — never tabs. A misaligned key silently moves to the wrong parent and produces confusing errors. Most editors have a YAML plugin that highlights these issues.
🪜 Step-by-step guide
1
Clean up any Task 27 leftovers
If containers or the network from Task 27 are still running, remove them first. The named volume can stay — Compose will reuse it and your data will survive.
shell 
# Stop and remove containers from Task 27 if still running
docker rm -f node-app mongo
docker network rm app-network

# Keep the volume — Compose will reuse it
# docker volume rm mongo-data  ← skip this
2
Create docker-compose.yml
Create the file in your node-mongo-app/ directory and paste the content from the section above. Verify the file is valid with docker compose config.
shell 
# Validate the compose file — prints the merged config or errors
docker compose config
✅
If docker compose config prints back the merged YAML without errors, the file is syntactically valid. Fix any indentation errors it reports before proceeding.
3
Bring the stack up
--build forces a fresh image build before starting. -d runs everything in the background. This single command replaces all six manual steps from Task 27.
shell 
docker compose up --build -d
expected output
[+] Building 11.2s (9/9) FINISHED
=> [app] FROM node:20-alpine 2.1s
=> [app] RUN npm install --omit=dev 7.8s
=> [app] exporting to image 0.3s

[+] Running 4/4
✔ Network node-mongo-app_app-network Created 0.1s
✔ Volume "node-mongo-app_mongo-data" Created 0.0s
✔ Container mongo Started 0.5s
✔ Container node-app Started 0.8s
ℹ️
Notice the volume is prefixed: node-mongo-app_mongo-data. If you created mongo-data manually in Task 27, Compose creates a new namespaced one — your old data won't appear. See the note at the end of this task on external volumes.
4
Inspect the running stack
Compose has its own ps command that only shows containers belonging to this project.
shell 
docker compose ps
expected output
NAME IMAGE COMMAND SERVICE STATUS PORTS
mongo mongo:7 "docker-entrypoint.s…" mongo Up 30 seconds
node-app node-mongo-app "node server.js" app Up 28 seconds 0.0.0.0:3000->3000/tcp
5
Follow logs from all services
Compose multiplexes logs from every container into one stream, colour-coded by service name. Press Ctrl+C to stop following without stopping the containers.
shell 
# All services
docker compose logs -f

# One service only
docker compose logs -f app
expected output — interleaved, colour-coded
mongo | {"msg":"Waiting for connections","attr":{"port":27017}}
app | ⏳ Mongo not ready, retrying... (5 left)
app | ⏳ Mongo not ready, retrying... (4 left)
mongo | {"msg":"Connection accepted"}
app | ✅ Connected to MongoDB
app | 🚀 API on port 3000
6
Verify the API is working
Hit the health endpoint and create a note — exactly as in Task 27, but this time the whole stack started with one command.
shell 
curl http://localhost:3000/health

curl -X POST http://localhost:3000/notes \
  -H "Content-Type: application/json" \
  -d '{"text":"Written via Compose"}'

curl http://localhost:3000/notes
expected output
{"status":"ok","mongo":1}
{"_id":"…","text":"Written via Compose","createdAt":"…","__v":0}
[{"_id":"…","text":"Written via Compose","createdAt":"…","__v":0}]
7
Edit the app and hot-rebuild
Make a change to server.js — for example, update the health response. Then run up again. Compose rebuilds only the app service, leaves Mongo untouched, and swaps in the new container.
shell 
# After editing server.js, rebuild and restart only the app
docker compose up --build -d app
expected output — mongo untouched
[+] Building 2.1s (9/9) FINISHED (layer cache hit on npm install)
[+] Running 1/1
✔ Container node-app Started 0.4s
💡
The npm install layer is served from cache because package.json didn't change — the same caching strategy you set up in Task 14 pays off here.
8
Tear down the stack
down stops and removes containers and the network. The volume is kept by default — your data survives. Add -v only if you want to wipe the database.
shell 
# Stop and remove containers + network (keep volume)
docker compose down

# Also remove the volume — DESTROYS ALL DATA
docker compose down -v
expected output — docker compose down
[+] Running 3/3
✔ Container node-app Removed 1.2s
✔ Container mongo Removed 0.4s
✔ Network node-mongo-app_app-network Removed 0.1s
🔗 Bonus — reuse a volume from Task 27

Compose namespaces volumes by default. To use the bare mongo-data volume you created manually, declare it as external. Compose will use it as-is without prefixing.

docker-compose.yml — volumes section (external) 
volumes:
  mongo-data:
    external: true   # use the existing 'mongo-data' volume, don't namespace it
⚠️
docker compose up will fail with an error if external: true is set and the volume doesn't exist — Compose will not create it. Create it first with docker volume create mongo-data.
📋 Essential Compose commands
docker compose up -d Start all services in detached mode (build if needed)
docker compose up --build -d Force-rebuild images then start — use after code changes
docker compose down Stop and remove containers + network; keep volumes
docker compose down -v Also remove named volumes — destroys data, use with care
docker compose ps List containers belonging to this project only
docker compose logs -f [service] Stream logs — all services or one; Ctrl+C detaches safely
docker compose exec app sh Open a shell inside a running service container
docker compose restart app Restart one service without touching the others
docker compose config Validate and print the merged compose config — debug first
docker compose pull Pull the latest versions of all images declared in the file
