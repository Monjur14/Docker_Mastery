Task 30  ·  Intermediate

Environment
Variable Injection

Pass runtime config and secrets into containers via --env-file and Compose environment: fields — keeping secrets out of images and Dockerfiles, and config out of source control.

~20 minutes Completed Tasks 14 – 29 Docker Compose v2
📖Required concepts
🌍
Environment Variables in Containers
Every process inherits a set of key=value pairs from its environment. Containers are no different — when Docker starts a container, it can inject any number of variables that are immediately available to the running process via process.env, os.environ, or equivalent in any language.
🚫
Never Hardcode Secrets in Images
Anything written into a Dockerfile with ENV or COPY is baked into a layer forever — even if you delete it in a later step. Anyone who pulls the image can read that layer. Inject secrets at runtime, not build time.
📄
The .env File
A plain text file with one KEY=value pair per line. Compose automatically reads a .env in the project root for variable substitution in the compose file itself. For injecting variables into containers, you use --env-file or env_file: — these are different things, and the distinction matters.
🔢
Precedence: Which Value Wins?
When the same variable is set in multiple places, Docker uses a strict order of precedence. CLI flags beat compose file beats env_file beats Dockerfile defaults. Knowing the order lets you safely override values for local dev without touching the compose file.
🔁
Pass-through Variables
In a compose file, listing a variable name without a value — e.g. - DATABASE_URL — passes through the value from your shell environment. This lets you set secrets in your shell (or CI) and forward them in without writing them to any file.
🏗️
ARG vs ENV in Dockerfiles
ARG is a build-time variable — available during docker build only, gone at runtime. ENV sets a variable that persists into the running container. Use ARG for non-secret build config (e.g. Node version) and runtime injection for secrets.
🗂️Four ways to inject variables
CLI flag
-e / --env
Set one variable inline: -e PORT=3000. Repeatable. Highest precedence.
Best for: one-off overrides, quick testing.
CLI flag
--env-file
Load all vars from a file: --env-file .env.prod. One flag, many variables.
Best for: environment-specific config bundles.
Compose file
environment:
Declare vars in the compose file — with a value, or as pass-through (name only).
Best for: non-secret config committed to source.
Compose file
env_file:
Point to a file in the compose file: env_file: .env.local. File stays out of git.
Best for: secrets alongside a compose service.
🔬Command anatomy
docker run --env-file .env.prod -e LOG_LEVEL=debug myapp:1.0.0
docker

The Docker CLI.

run

Create and start a container from an image.

--env-file

Load all KEY=value lines from a file into the container's environment. Comments (#) and blank lines are ignored.

.env.prod

Path to the env file — relative to current working directory. Can be named anything; convention is .env, .env.prod, .env.local.

-e

Set a single variable inline. Overrides the same key from --env-file — CLI flags have higher precedence.

image

The image to run. Env vars are injected after the container starts — they're never written into the image.

🪜Step-by-step guide
1
Inspect a container's environment baseline
First, see what variables a plain container receives without any injection. The env command prints every variable in the container's environment.
shell
docker run --rm alpine env
expected output
PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin
HOSTNAME=3f8a2b1c9d7e
HOME=/root
ℹ️
Only a handful of OS defaults. Nothing from your host shell leaks in unless you explicitly pass it — containers are isolated by default.
2
Inject a single variable with -e
The fastest way — pass one key=value pair directly on the command line. Useful for quick tests and CI overrides.
shell
docker run --rm \
  -e NODE_ENV=production \
  -e PORT=3000 \
  alpine env
expected output
NODE_ENV=production
PORT=3000
PATH=/usr/local/sbin:...
HOSTNAME=9c2b4f1e8a3d
3
Create an env file for your app
An env file keeps all your config in one place and out of your command history. Create .env.local (and add it to .gitignore):
.env.local
# App config
NODE_ENV=development
PORT=3000
LOG_LEVEL=debug

# Database — never commit real credentials
DATABASE_URL=mongodb://user:password@db:27017/myapp
DATABASE_NAME=myapp_dev

# Third-party
STRIPE_SECRET_KEY=sk_test_abc123...
JWT_SECRET=supersecretkey-change-in-prod
🔐
Always add .env* to .gitignore before creating these files. A single accidental commit of real credentials to a public repo is an incident — secret scanning bots harvest them within seconds of the push.
4
Load the file with --env-file
Pass the file path to --env-file. All key=value pairs are injected at once. You can stack multiple files.
shell
docker run --rm \
  --env-file .env.local \
  alpine env | sort
expected output
DATABASE_NAME=myapp_dev
DATABASE_URL=mongodb://user:password@db:27017/myapp
JWT_SECRET=supersecretkey-change-in-prod
LOG_LEVEL=debug
NODE_ENV=development
PORT=3000
STRIPE_SECRET_KEY=sk_test_abc123...
💡
You can override any file value at runtime with -e: --env-file .env.local -e NODE_ENV=production. The -e flag wins — this is how you quickly flip one value without editing the file.
5
Use environment: in a Compose file
The environment: key injects variables into a specific service. Use a value to hardcode non-secrets; omit the value to pass through from your shell.
docker-compose.yml
services:
  app:
    build: .
    environment:
      # Hardcoded — safe to commit (non-secret config)
      NODE_ENV: production
      PORT: "3000"

      # Pass-through — value comes from your shell or CI
      # (no value after the colon)
      DATABASE_URL:
      JWT_SECRET:
      STRIPE_SECRET_KEY:
ℹ️
Pass-through variables with no value are silently absent if the variable isn't set in your shell — Docker won't error. Verify they're present inside the container after startup if your app requires them.
6
Use env_file: in a Compose service
Point a service at an env file directly in the compose file. Compose loads it when the service starts — no need to pass --env-file on the CLI every time.
docker-compose.yml
services:
  app:
    build: .
    env_file:
      - .env          # shared defaults (committed — no secrets)
      - .env.local   # local overrides (gitignored — secrets OK)
    environment:
      # Still override specific keys here if needed
      NODE_ENV: development
💡
Stack multiple files — later files override earlier ones for the same key. A common pattern: commit .env with safe defaults and gitignore .env.local for each developer's real secrets.
7
Use ${VARIABLE} substitution in the compose file itself
The compose file can reference variables from your shell or a root .env file using ${VAR} syntax. This is distinct from injecting into containers — it controls the compose file's own values.
docker-compose.yml
services:
  app:
    # Image tag comes from the shell or .env at project root
    image: myapp:${APP_VERSION:-latest}
    ports:
      # Host port is configurable per environment
      - "${HOST_PORT:-3000}:3000"
.env (committed — controls compose file, not container)
APP_VERSION=1.2.0
HOST_PORT=8080
⚠️
The root .env file is for compose file substitution only — it does NOT automatically inject into containers unless you also add env_file: .env to a service. This distinction trips up almost everyone the first time.
8
Verify what a running container sees
After docker compose up -d, confirm the container received the right variables by execing into it.
shell
# Exec into the running service container
docker compose exec app env | sort

# Or check one specific variable
docker compose exec app printenv DATABASE_URL
ℹ️
docker compose config shows the fully resolved compose file with all substitutions applied — useful for debugging ${VAR} references before you even start the containers.
9
Read env vars in your Node.js app
With variables injected, your app reads them at runtime via process.env. No library needed — this is standard Node.js.
index.js
// Read injected variables — no hardcoded values
const port    = process.env.PORT         || 3000;
const dbUrl   = process.env.DATABASE_URL;
const nodeEnv = process.env.NODE_ENV     || 'development';

if (!dbUrl) {
  console.error('DATABASE_URL is required');
  process.exit(1);   // fail fast if required secret is missing
}

console.log(`Starting in ${nodeEnv} mode on port ${port}`);
💡
Fail fast on missing required variables at startup. It's far easier to debug a container that refuses to start with a clear error than one that starts but fails silently 10 minutes later when it first tries to connect to a database.
10
Set a default in the Dockerfile with ENV
ENV in a Dockerfile sets a fallback that's always present — runtime injection overrides it. Only use ENV for non-secret defaults like port numbers or mode flags.
Dockerfile
FROM node:20-alpine
WORKDIR /app
COPY package*.json ./
RUN npm ci --omit=dev
COPY . .

# Safe defaults — overridden at runtime by -e or env_file
ENV NODE_ENV=production
ENV PORT=3000

# NEVER do this:
# ENV DATABASE_URL=mongodb://user:secret@host/db

EXPOSE 3000
CMD ["node", "index.js"]
🏆Precedence — which value wins

When the same variable is defined in multiple places, Docker resolves it top-down. Higher wins.

1
-e KEY=value (CLI flag)
Explicit inline override — always wins
highest
2
--env-file path (CLI flag)
File loaded via docker run
3
environment: in compose file
Hardcoded or pass-through in the YAML
4
env_file: in compose file
File reference declared per-service
5
ENV in Dockerfile
Baked-in default — lowest priority
ℹ️
Precedence only matters when the same key appears in multiple sources. Different keys from different sources are all merged together — there's no conflict.
🔐Security rules for env vars
🚫
Never use ENV in Dockerfiles for secrets
Every ENV instruction is a permanent layer visible to anyone who pulls the image — even if overwritten later. There is no safe way to delete it after the fact.
🚫
Never commit .env files with real secrets
Add .env* to .gitignore before you create them. A rewritten git history is a painful and often incomplete fix.
✅
Commit a .env.example with placeholder values
Document every variable your app needs without real values. Developers copy it to .env.local and fill in their own credentials.
✅
Use a secret manager in production
Env files are fine for local dev. In production, pull secrets from AWS Secrets Manager, HashiCorp Vault, or Docker Swarm secrets — never ship a plain-text secrets file to a server.
✅
Validate required vars at startup
Crash immediately with a clear message if a required variable is absent. This prevents subtler failures and makes missing-secret problems obvious in logs.
🔍What happened under the hood
Step 01
📋
File parsed
Docker CLI read the --env-file, stripping comments and blank lines.
Step 02
🔀
Sources merged
Variables from all sources were merged in precedence order — higher wins on conflict.
Step 03
📦
Container spec
The resolved key=value list was attached to the container's OCI spec before any process started.
Step 04
🚀
Process started
The container's init process received the variables in its environment before executing CMD.
Step 05
🔒
Image unchanged
No variable was written to any image layer — the image is identical regardless of what was injected at runtime.
Step 06
🔄
Re-injectable
Run the same image with different env vars for dev, staging, or prod — no rebuild needed.
