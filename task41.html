CI Pipeline
Integration

Set up a GitHub Actions workflow that automatically lints your Dockerfile, builds your image, runs a vulnerability scan, and pushes to Docker Hub on every push to main — with layer caching so it's fast.

~30 minutes Tasks 19 & 37 complete GitHub account + Docker Hub
🗺️What we're building

A single workflow file that runs five stages in sequence. Each stage is a gate — if any step fails, the push is blocked and no broken image reaches Docker Hub.

📖Required concepts
⚙️
GitHub Actions anatomy
A workflow is a YAML file in .github/workflows/. It contains one or more jobs, each running on a fresh virtual machine. Jobs contain steps — either shell commands (run:) or pre-built actions (uses:) from the marketplace. Steps within a job share the same filesystem; jobs run in parallel by default unless you use needs:.
🔐
GitHub Actions secrets
Sensitive values like Docker Hub credentials are stored in repository secrets (Settings → Secrets and variables → Actions). They're injected into the workflow via ${{ secrets.MY_SECRET }} syntax and are never logged or exposed in build output — even if a step prints them, GitHub redacts the value.
⚡
Build cache with GitHub Actions Cache
Docker Buildx can export layer cache to GitHub's cache store and restore it on the next run. The cache-from / cache-to options in docker/build-push-action handle this automatically. A warm cache turns a 3-minute build into a 20-second one when only app code changes.
🏷️
Image tagging strategy
The docker/metadata-action generates smart tags automatically: latest on main pushes, v1.2.3 on git tags, pr-42 on pull requests, and the short commit SHA on every push. This gives you full traceability — you can always roll back to an exact commit.
🔍
hadolint — Dockerfile linter
hadolint statically analyses your Dockerfile against a set of best-practice rules — catching things like apt-get install without pinned versions, missing --no-install-recommends, using ADD instead of COPY, and unpinned base images. It fails the CI build on violations above a configurable severity.
🛡️
trivy — image vulnerability scanner
trivy (by Aqua Security) scans a built image for known CVEs in OS packages and language dependencies. In the pipeline it runs after the build step — if CRITICAL vulnerabilities are found, the push is blocked. It can also output SARIF reports that appear natively in GitHub's Security tab.
Workflow YAML structure
on:

Defines when the workflow fires — push events, pull_request, schedule, or workflow_dispatch (manual). Can filter by branch or path.

jobs:

Top-level map of job IDs. Each job gets a fresh runner VM. Jobs run in parallel unless needs: creates a dependency chain.

steps:

Ordered list within a job. Each step is either a shell run: command or a reusable uses: action. Steps share the same workspace and environment.

secrets:

Repository secrets injected via ${{ secrets.NAME }}. Values are masked in all logs — never exposed even if accidentally printed.

context:

GitHub provides rich context objects: github.sha, github.ref_name, github.event_name, and more — available in any expression.

🪜Step-by-step guide
1
Set up the project structure
Create the GitHub Actions workflow directory inside your existing Docker project. The workflow file must live at exactly this path.
shell
mkdir -p .github/workflows
my-docker-app/
├── Dockerfile
├── docker-compose.yml
├── app.js (or your app files)
├── .hadolint.yaml ← linter config (Step 2)
└── .github/
    └── workflows/
        └── docker-publish.yml ← the workflow (Step 4)
2
Add Docker Hub credentials as repository secrets
Your Docker Hub username and a Personal Access Token (PAT) must be stored as secrets — never hardcoded. Create the PAT in Docker Hub under Account Settings → Security → New Access Token.
ℹ️
Go to your GitHub repo → Settings → Secrets and variables → Actions → New repository secret. Create two secrets:

DOCKERHUB_USERNAME — your Docker Hub username
DOCKERHUB_TOKEN — a Docker Hub Access Token (not your password)
🔐
Use an Access Token, not your password. Tokens can be scoped to read/write only and revoked individually without changing your account password. Create one at hub.docker.com → Account Settings → Security.
3
Configure hadolint
Create a .hadolint.yaml to configure which rules to enforce and which to ignore for your project. Commit this alongside the Dockerfile.
.hadolint.yaml
# Fail the build on warnings and above
failure-threshold: warning

# Rules to ignore (add your own justified exceptions)
ignore:
  - DL3008  # pin versions in apt — we'll accept unpinned for now
  - DL3018  # pin versions in apk

# Trusted registries — images from these don't need digest pinning
trustedRegistries:
  - docker.io
  - ghcr.io
4
Write the complete workflow file
Create .github/workflows/docker-publish.yml. Read through each section — the inline comments explain every decision.
.github/workflows/docker-publish.yml
name: Docker Build, Scan & Publish

# ── Triggers ──────────────────────────────────────
on:
  push:
    branches: ["main"]
    tags:    ["v*.*.*"]     # also fire on semver tags
  pull_request:
    branches: ["main"]       # build+scan PRs, but don't push

# ── Global env ────────────────────────────────────
env:
  IMAGE_NAME: ${{ secrets.DOCKERHUB_USERNAME }}/my-app

jobs:

  # ── Job 1: Lint ───────────────────────────────────
  lint:
    name: Lint Dockerfile
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Run hadolint
        uses: hadolint/hadolint-action@v3.1.0
        with:
          dockerfile: Dockerfile
          config: .hadolint.yaml

  # ── Job 2: Build, Scan & Push ─────────────────────
  build-and-push:
    name: Build → Scan → Push
    runs-on: ubuntu-latest
    needs: lint            # only runs if lint passes
    permissions:
      contents: read
      security-events: write  # for SARIF upload to Security tab

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      # Set up Buildx for cache support and multi-arch
      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v3

      # Log in only on pushes — not on pull requests
      - name: Log in to Docker Hub
        if: github.event_name != 'pull_request'
        uses: docker/login-action@v3
        with:
          username: ${{ secrets.DOCKERHUB_USERNAME }}
          password: ${{ secrets.DOCKERHUB_TOKEN }}

      # Generate smart tags from git context
      - name: Extract Docker metadata
        id: meta
        uses: docker/metadata-action@v5
        with:
          images: ${{ env.IMAGE_NAME }}
          tags: |
            # latest on main branch pushes
            type=raw,value=latest,enable={{is_default_branch}}
            # semver tags: v1.2.3 → 1.2.3, 1.2, 1
            type=semver,pattern={{version}}
            type=semver,pattern={{major}}.{{minor}}
            # short SHA on every push for traceability
            type=sha,prefix=sha-,format=short
            # pr-42 label on pull requests
            type=ref,event=pr

      # Build with layer cache — load locally for the scan step
      - name: Build image (cached)
        uses: docker/build-push-action@v5
        with:
          context: .
          push: false           # don't push yet — scan first
          load: true            # load into local Docker for trivy
          tags: ${{ steps.meta.outputs.tags }}
          labels: ${{ steps.meta.outputs.labels }}
          cache-from: type=gha   # restore from GitHub Actions Cache
          cache-to: type=gha,mode=max  # save all layers

      # Scan the built image for CVEs
      - name: Scan image with Trivy
        uses: aquasecurity/trivy-action@master
        with:
          image-ref: ${{ env.IMAGE_NAME }}:latest
          format: sarif
          output: trivy-results.sarif
          severity: CRITICAL,HIGH   # fail on critical + high
          exit-code: '1'           # non-zero = block the push

      # Upload SARIF to GitHub Security tab (always, even on failure)
      - name: Upload Trivy SARIF report
        uses: github/codeql-action/upload-sarif@v3
        if: always()
        with:
          sarif_file: trivy-results.sarif

      # Push to Docker Hub — only if scan passed & not a PR
      - name: Push image to Docker Hub
        if: github.event_name != 'pull_request'
        uses: docker/build-push-action@v5
        with:
          context: .
          push: true
          tags: ${{ steps.meta.outputs.tags }}
          labels: ${{ steps.meta.outputs.labels }}
          cache-from: type=gha   # warm cache — almost instant
          cache-to: type=gha,mode=max
⚡
Why build twice? The first build uses load: true to make the image available locally for Trivy to scan. The final push step rebuilds from the warm cache — because load and push can't be used together with Buildx in the same invocation. With a hot cache, the second build takes only a few seconds.
5
Commit and push to trigger the workflow
Add all files, commit, and push to main. GitHub automatically detects the new workflow file and queues a run.
shell
git add .github/workflows/docker-publish.yml .hadolint.yaml
git commit -m "ci: add Docker build, scan and publish workflow"
git push origin main
ℹ️
Navigate to your repo on GitHub → Actions tab. You'll see the workflow run appear within seconds. Click into it to watch each step execute in real time with live log streaming.

7
Understand the tag strategy in practice
The tags generated by docker/metadata-action change based on what triggered the workflow. Here's the full matrix:
Git event Tags pushed to Docker Hub
push to main latest, sha-a1b2c3d
git tag v1.2.3 1.2.3, 1.2, latest, sha-a1b2c3d
pull request #42 pr-42, sha-a1b2c3d — not pushed, only built+scanned
💡
To trigger a versioned release, push a semver git tag: git tag v1.0.0 && git push origin v1.0.0. The workflow fires again and pushes the 1.0.0, 1.0, and latest tags atomically.

9
Handle a failing Trivy scan
When Trivy finds a CRITICAL CVE, the push step is skipped and the run fails. Here's how to triage and fix it.
trivy scan failure output
CRITICAL CVE-2024-1234 libssl3 3.0.8-1 Fix: upgrade to 3.0.11
HIGH CVE-2024-5678 libc6 2.35 Fix: upgrade to 2.36

2 vulnerabilities found (exit code 1)
✗ Push image to Docker Hub — skipped (previous step failed)
Dockerfile — fix: use a newer base image with patches
# Before: unpinned, may have old vulnerable packages
FROM node:20-alpine

# After: pin to a specific digest for reproducibility
# or bump to a newer patch that has the fix
FROM node:20.11-alpine3.19
⚠️
If a CVE exists in your base image but no fix is available yet, you can temporarily add it to a Trivy ignore file (.trivyignore) with a comment explaining when to revisit. Don't use this as a permanent workaround.
10
Add a scheduled weekly scan
CVEs are discovered continuously — even images that passed last week may be vulnerable now. Add a scheduled trigger to re-scan the latest published image weekly, without requiring a code change.
docker-publish.yml — add to the on: block
on:
  push:
    branches: ["main"]
    tags:    ["v*.*.*"]
  pull_request:
    branches: ["main"]
  # Re-scan every Monday at 08:00 UTC
  schedule:
    - cron: '0 8 * * 1'
💡
Scheduled runs on the default branch use the latest committed workflow file. If a new CVE is found by the weekly scan, GitHub will email the repository owner and mark the Security tab with the findings from the uploaded SARIF report.
