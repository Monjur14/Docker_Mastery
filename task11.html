Task 11  ·  Beginner

Pull Different Images
from Docker Hub

Search and pull several official images — alpine, python, and node — and explore the image registry from the CLI.

~8 minutes Docker installed Terminal / Shell
📖 Required concepts
🏪
Official images
Official images on Docker Hub (like alpine, python, node) are curated and maintained either by Docker or the upstream project itself. They live under the library/ namespace and are generally the safest, best-documented starting point for any given tool or language.
🔎
docker search
docker search queries Docker Hub from the CLI and returns matching repositories along with star counts and whether they're official. It's a quick way to discover images without opening a browser.
🏷️
Tags select the variant
A single repository like python holds many tags — different versions and base OS variants (e.g. python:3.12, python:3.12-slim, python:3.12-alpine). Omitting a tag defaults to :latest, which is convenient but not reproducible — production setups should always pin a specific tag.
🪶
Why Alpine images are tiny
Alpine Linux is a minimal distro built around musl libc and BusyBox instead of glibc and GNU coreutils. It's often under 10MB, which is why so many language images ship an -alpine variant for smaller production images — at the cost of occasional compatibility quirks with software expecting glibc.
🧱
Shared base layers
If two images share a common base layer (e.g. both built FROM debian:bookworm), Docker only downloads and stores that layer once. Pulling several images can be faster and cheaper on disk than the sum of their individual sizes suggests.
📜
docker image ls
docker image ls (an alias of docker images) lists everything stored in your local registry cache — repository, tag, image ID, creation date, and size — so you can see exactly what you've pulled.
🔬 Command anatomy
docker pull python:3.12-slim
docker

The CLI binary — how you talk to the Docker daemon from your terminal.

pull

Subcommand that downloads an image (and its layers) from a registry without running a container from it.

python

Repository name. Without a namespace prefix, Docker assumes the official library/python repository.

:3.12-slim

Tag — selects a specific version and variant. Omit it and Docker defaults to :latest.

🪜 Step-by-step guide
1
Search Docker Hub from the CLI
Search for "alpine" and look at the results — pay attention to which repositories are marked official.
shell 
docker search alpine
expected output
NAME DESCRIPTION STARS OFFICIAL
alpine A minimal Docker image based on Alpine... 11000 [OK]
alpine/git A simple git container running in alpine 250
alpine/socat Run socat command in alpine container 120
2
Pull the Alpine image
Pull the official Alpine Linux image — one of the smallest general-purpose base images available.
shell 
docker pull alpine:latest
3
Pull two Python variants
Pull the full Python image and the slim variant, so you can compare their sizes directly afterward.
shell 
docker pull python:3.12
docker pull python:3.12-slim
4
Pull the Node.js LTS image
Pull Node's current long-term-support release, tagged with its codename alias.
shell 
docker pull node:lts
5
List everything you've pulled
Compare sizes across images and tags — this is where Alpine's minimalism becomes obvious.
shell 
docker images
expected output
REPOSITORY TAG IMAGE ID CREATED SIZE
node lts 3a5f9c1d2b4e 2 weeks ago 1.1GB
python 3.12 6e2a8d4f1c9b 3 weeks ago 1.02GB
python 3.12-slim 9b1c7a3e5f2d 3 weeks ago 130MB
alpine latest 051f94a92aac 6 weeks ago 7.8MB
💡
Notice how alpine is roughly 150x smaller than the full python image, and the -slim variant sits in between. Base image choice has a real impact on pull time, attack surface, and deployment speed.
6
Explore what's inside an image
Start an interactive shell in Alpine to poke around and confirm it really is that minimal.
shell 
docker run -it --rm alpine sh
ℹ️
Inside the shell try cat /etc/os-release to confirm you're in Alpine, then exit to leave. The --rm flag auto-deletes the container on exit since it's just for exploring.
7
Verify a language runtime inside its image
Run the Python image's interpreter directly with docker run to confirm the version installed.
shell 
docker run --rm python:3.12-slim python3 --version
docker run --rm node:lts node --version
expected output
Python 3.12.4
v20.15.1
8
Inspect image details without running it
Use docker image inspect to see architecture, layers, and config for the Node image without starting a container.
shell 
docker image inspect node:lts --format '{{.Architecture}} / {{.Os}}'
9
Clean up (optional)
Remove any of the images you no longer want to keep locally.
shell 
# Remove specific images by repo:tag
docker rmi python:3.12 node:lts

# Or remove all unused images at once
docker image prune -a
📐 Image variant reference
Image Typical use
alpine Tiny general-purpose base for building minimal custom images; not a language runtime by itself.
python:3.12 Full Debian-based Python image with build tools included — largest, most compatible option.
python:3.12-slim Debian-based but stripped of build tools and docs — a good middle ground for production.
node:lts Current Node.js long-term-support release, Debian-based, ready for most Node apps out of the box.
