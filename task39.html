Task 39  ·  Advanced

Self-hosted
Private Registry

Run the official registry:2 container locally, push a custom image to it, and pull that image from another machine — no Docker Hub account required.

~15 minutes Docker installed Two machines or VMs
📖 Required concepts
🏛️
What a registry is
A registry is a server that stores and serves Docker images. Docker Hub is a public registry. Running registry:2 gives you a private registry you control entirely — useful for air-gapped environments, CI pipelines, or keeping proprietary images off the public internet.
🏷️
Image naming encodes the registry
Docker infers which registry to use from the image name. nginx pulls from Docker Hub. localhost:5000/nginx pushes to and pulls from your local registry on port 5000. The host:port prefix is how Docker knows where to send the request.
🔓
Insecure registries
By default Docker only pushes to or pulls from registries over HTTPS. A local registry on localhost is exempt, but a remote machine on your LAN is not. You must either configure TLS or tell the Docker daemon on the client machine to treat that IP as an insecure registry in daemon.json.
💾
Persistent storage with a volume
The registry container stores images under /var/lib/registry inside the container. Without a bind mount or named volume, all pushed images are lost when the container is removed. Always attach a volume to a production registry.
🔁
Retag before push
You can't push an image named myapp:1.0 to a private registry — Docker doesn't know which registry to target. You must first retag it with the registry's address: localhost:5000/myapp:1.0. The retag creates an alias; the image data isn't duplicated.
🔍
Registry API
registry:2 exposes the Docker Registry HTTP API v2. You can query it directly with curl — listing repositories, tags, and manifests — without needing the Docker CLI at all.

🔬 Command anatomy
docker tag alpine:latest localhost:5000/alpine:latest
docker tag

Creates an alias for an existing local image. No data is copied — both names point to the same layers.

alpine:latest

Source image already on your machine. Can be any local image name.

localhost:5000/

Registry prefix — tells Docker to push this image to the registry listening on localhost:5000 rather than Docker Hub.

alpine:latest

Repository name and tag inside the registry. Can differ from the source name.

🪜 Step-by-step guide
1
Machine A — registry host
Start the registry container
Run registry:2 in detached mode, publish port 5000, and attach a named volume so pushed images survive container restarts.
shell 
docker run -d \
  --name registry \
  --restart always \
  -p 5000:5000 \
  -v registry-data:/var/lib/registry \
  registry:2
💡
--restart always ensures the registry comes back up automatically if the host reboots — essential for any long-lived registry.
2
Machine A — registry host
Verify the registry is responding
Hit the v2 API root — a 200 response confirms the registry is healthy and ready to accept pushes.
shell 
curl -s http://localhost:5000/v2/
expected output
{}
ℹ️
An empty JSON object {} is the correct response from a healthy registry. Any other response — or a connection error — means the container isn't running yet.
3
Machine A — registry host
Pull a source image to push
Grab Alpine as a lightweight test image to push to your registry.
shell 
docker pull alpine:latest
4
Machine A — registry host
Retag the image with your registry prefix
Create an alias that points to your local registry. Without this prefix Docker would try to push to Docker Hub instead.
shell 
docker tag alpine:latest localhost:5000/alpine:latest
5
Machine A — registry host
Push the image to your registry
Docker reads the registry address from the image name and uploads the layers directly to your local registry container.
shell 
docker push localhost:5000/alpine:latest
expected output
The push refers to repository [localhost:5000/alpine]
78a822fe2a2d: Pushed
latest: digest: sha256:1308bd... size: 528
6
Machine A — registry host
List repositories via the API
Confirm the image is stored in the registry by querying the catalog endpoint directly.
shell 
# List all repositories in the registry
curl -s http://localhost:5000/v2/_catalog

# List tags for a specific repository
curl -s http://localhost:5000/v2/alpine/tags/list
expected output
{"repositories":["alpine"]}
{"name":"alpine","tags":["latest"]}
7
Machine A — registry host
Delete the local copy and pull it back
Remove the local image, then pull it from your registry to prove the round trip works — this is the same test a remote machine would perform.
shell 
# Remove both local copies
docker rmi alpine:latest localhost:5000/alpine:latest

# Pull from your registry
docker pull localhost:5000/alpine:latest
expected output
latest: Pulling from localhost:5000/alpine
c926b61bad3b: Pull complete
Status: Downloaded newer image for localhost:5000/alpine:latest
8
Machine B — configure insecure registry
Allow a remote machine to reach the registry
On Machine B, edit daemon.json to trust the registry host's IP over plain HTTP, then restart the Docker daemon.
json — /etc/docker/daemon.json on Machine B 
{
  "insecure-registries": ["192.168.1.100:5000"]
}
shell — Machine B 
sudo systemctl restart docker
⚠️
Replace 192.168.1.100 with the actual IP of Machine A. For anything beyond a LAN test, configure TLS instead of using an insecure registry — see the hardening note in the next step.
9
Machine B — pull from registry
Pull the image on Machine B
Use the registry host's IP in the image name. Docker resolves it to your private registry — no Docker Hub involved.
shell — Machine B 
docker pull 192.168.1.100:5000/alpine:latest
expected output
latest: Pulling from 192.168.1.100:5000/alpine
c926b61bad3b: Pull complete
Status: Downloaded newer image for 192.168.1.100:5000/alpine:latest
10
Machine A — cleanup
Clean up
Stop and remove the registry container. The named volume keeps your images safe — remove it separately only if you want to wipe the registry data too.
shell 
# Stop and remove the registry container
docker rm -f registry

# Optional: also wipe the stored images
docker volume rm registry-data
🔍 What happened under the hood
Step 01
🏷️
Tag resolves target
The localhost:5000/ prefix told Docker which registry to use at push time.
Step 02
🔐
Auth skipped
localhost is implicitly trusted by Docker — no TLS or credentials needed for this basic setup.
Step 03
📤
Layers uploaded
The daemon sent each image layer as a blob to the registry's HTTP API, skipping any already present.
Step 04
💾
Stored on volume
registry:2 wrote the blobs and manifest to /var/lib/registry, backed by the named volume.
Step 05
📥
Pull resolves registry
Machine B parsed the IP prefix in the image name, checked daemon.json, and fetched layers over HTTP.
Step 06
✅
Round trip complete
Machine B has the exact same image layers that Machine A pushed, verified by content digest.
📐 Registry API quick reference
Endpoint What it returns
GET /v2/ Health check. Returns {} if the registry is up and the API version is supported.
GET /v2/_catalog Lists all repository names stored in the registry.
GET /v2/<name>/tags/list Lists all tags available for a specific repository.
GET /v2/<name>/manifests/<tag> Returns the full image manifest for a given repository and tag.
DELETE /v2/<name>/manifests/<digest> Marks a manifest for deletion. Run garbage collection afterward to reclaim disk space.
