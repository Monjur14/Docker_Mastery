Task 20  ·  Intermediate

Local Directory
Bind Mounts

Mount a host source-code directory into a container using --mount or -v to enable live-reload during development — edit a file on your host and see it reflected instantly inside the container.

~15 minutes Docker installed Terminal / Shell
📖 Required concepts
🔗
What is a bind mount?
A bind mount maps a specific path on the host directly into a path inside the container. Unlike a named volume (which Docker manages), a bind mount points at a real directory you own — so any file you edit on your laptop immediately appears inside the running container, and vice-versa.
🆚
Bind mounts vs. named volumes
Bind mounts reference a host path you provide — great for source code. Named volumes are fully managed by Docker, stored in Docker's storage area, and portable across machines — better for database data. Both use the same -v / --mount flags.
⚡
Live reload in development
Tools like nodemon, uvicorn --reload, or webpack-dev-server watch the filesystem for changes and restart the server automatically. When your source directory is bind-mounted, those watchers see your edits immediately — no rebuild, no container restart needed.
📌
Absolute paths required
Docker requires the host path to be absolute. Use $(pwd) in bash/zsh to resolve the current directory at runtime, so your command works wherever you've cd'd to. On Windows PowerShell use ${PWD}.
🔒
Read-only mounts
Appending :ro to the mount makes it read-only inside the container — the process can read your files but cannot modify them. Useful for config files or secrets you don't want a buggy container to overwrite on your host.
🪄
--mount vs. -v syntax
-v /host:/container is the older, terse shorthand. --mount type=bind,source=…,target=… is newer, more explicit, and gives better error messages. Both do the same thing — this task uses both so you recognise either.
🔬 Command anatomy
docker run -v $(pwd)/app:/app -p 3000:3000 node:20-alpine
docker

The CLI binary — how you talk to the Docker daemon from your terminal.

run

Create and start a new container from the specified image.

-v

Short form of --volume. Declares a bind mount in host_path:container_path format.

$(pwd)/app:/app

The host-side path left of the colon, the container-side path right. Docker mirrors them in real time.

node:20-alpine

The image to run — a lightweight Node.js 20 image used in this task's live-reload demo.

🪜 Step-by-step guide
1
Create a small project directory on your host
Set up a minimal Node.js HTTP server — this is the code you'll edit live without rebuilding any image.
shell 
mkdir bind-demo && cd bind-demo
mkdir app
app/server.js 
const http = require('http');
const server = http.createServer((req, res) => {
  res.end('Hello from v1!\n');
});
server.listen(3000, () => console.log('Listening on :3000'));
app/package.json 
{
  "name": "bind-demo",
  "scripts": {
    "start": "nodemon server.js"
  },
  "dependencies": {
    "nodemon": "^3.0.0"
  }
}
2
Understand what the mount will look like
Before running, visualise how your host directory maps to the container filesystem.
🖥️  Host (your machine)
$(pwd)/app/
├── server.js
└── package.json
bind mount
📦  Container
/app/
├── server.js
└── package.json
ℹ️
Changes flow both ways in real time. Edit server.js on your host → the container sees it immediately. A container writing to /app → files appear on your host. This is why bind mounts are powerful — and why you should be careful with write access.
3
Run the container with a bind mount (-v shorthand)
Mount ./app into /app in the container, install dependencies, then start nodemon — all in one command.
shell 
docker run -it --name live-server \
  -v "$(pwd)/app:/app" \
  -p 3000:3000 \
  -w /app \
  node:20-alpine \
  sh -c "npm install && npm start"
expected output
added 31 packages in 4s
[nodemon] 3.0.2
[nodemon] watching path(s): *.*
[nodemon] starting `node server.js`
Listening on :3000
💡
-w /app sets the working directory inside the container so npm install and npm start run in the right place. Always quote the -v value to handle paths with spaces.
4
Test the server is live
Open a new terminal tab and hit the port mapped to your host.
shell — new terminal tab 
curl http://localhost:3000
expected output
Hello from v1!
5
Edit the file on your host — watch it reload live
Without stopping the container, open app/server.js in your editor and change the response text.
app/server.js — edited line 
  res.end('Hello from v2 — live reload works!\n');

Save the file. In the container's terminal you'll see nodemon restart automatically:

container terminal (automatic)
[nodemon] restarting due to changes...
[nodemon] starting `node server.js`
Listening on :3000
shell — new terminal tab 
curl http://localhost:3000
expected output
Hello from v2 — live reload works!
6
Same mount with the explicit --mount syntax
The newer --mount flag is more verbose but clearer — you'll see it in production Compose files and documentation.
shell 
docker run -it --rm \
  --mount type=bind,source="$(pwd)/app",target=/app \
  -p 3000:3000 \
  -w /app \
  node:20-alpine \
  sh -c "npm install && npm start"
type=bind Tells Docker this is a bind mount (as opposed to volume or tmpfs).
source=… Absolute path on the host. The directory must already exist or Docker will error.
target=… The path inside the container where the host directory will be mounted.
readonly Optional — add ,readonly at the end to prevent the container from writing to your host.
7
Confirm the mount with docker inspect
Verify the bind mount is correctly registered in the container's metadata.
shell — new terminal tab 
docker inspect live-server --format '{{json .Mounts}}'
expected output
[{
"Type": "bind",
"Source": "/home/user/bind-demo/app",
"Destination": "/app",
"Mode": "",
"RW": true
}]
ℹ️
RW: true confirms the mount is read-write. If you had added :ro or ,readonly you'd see RW: false here instead.
8
Clean up
Stop and remove the container. Your source files on the host are untouched — the bind mount doesn't own them.
shell 
docker rm -f live-server
✅
Your bind-demo/app/ directory and all source files remain exactly as you left them. Removing the container has no effect on host files — that's the whole point of a bind mount.
🔍 What happened under the hood
Step 01
📂
Host path resolved
Docker resolved $(pwd)/app to an absolute path and registered the mount in the container config.
Step 02
🔗
Kernel bind mount
The Linux kernel's mount --bind syscall wired the host directory into the container's filesystem namespace.
Step 03
👁️
Shared inode table
Both sides share the same inodes — no copying. A file write on either side is immediately visible to the other.
Step 04
🔔
Filesystem events
Nodemon received inotify events when you saved — the same events the host OS would fire — and restarted the process.
Step 05
🗑️
Unmount on stop
When the container stopped, Docker unmounted the bind path from the container namespace. The host directory was never touched.
Step 06
🏠
Host files intact
Deleting the container only removes the container layer — bind-mounted host files are never managed or deleted by Docker.
🎯 What you learned
  • How a bind mount maps a specific host path directly into a container's filesystem
  • The difference between bind mounts (host-managed) and named volumes (Docker-managed)
  • How to use both -v host:container shorthand and the explicit --mount type=bind,… syntax
  • How file edits on the host propagate instantly inside the container for live reload
  • How to make a mount read-only with :ro or ,readonly
  • How to verify mounts with docker inspect and confirm the Type, Source, and RW fields
