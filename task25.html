Task 25  ·  Intermediate

Run MongoDB
in Docker

Start a MongoDB container, persist its data with a named volume so documents survive container deletion, then insert and query documents using mongosh — the official MongoDB shell.

~20 minutes Tasks 21–24 complete Terminal / Shell
📖 Required concepts
🍃
What is MongoDB?
MongoDB is a document database — it stores data as JSON-like documents (called BSON) rather than rows and columns. Documents are grouped into collections (analogous to tables) inside databases. It's schema-flexible: documents in the same collection can have different fields.
💾
Why data persistence matters
A container's writable layer is deleted when the container is removed. For a database, this would mean losing all your data. By attaching a named Docker volume to MongoDB's data directory (/data/db), the data lives on the volume — independent of the container's lifecycle.
🐚
mongosh
mongosh is the modern MongoDB Shell — a Node.js REPL that connects to a MongoDB instance and lets you run queries, insert documents, and inspect databases interactively. It's bundled into the official mongo Docker image so you can run it with docker exec.
🔐
Authentication with env vars
The official MongoDB image respects environment variables to set up an initial root user on first run: MONGO_INITDB_ROOT_USERNAME and MONGO_INITDB_ROOT_PASSWORD. If omitted, MongoDB starts without authentication — fine for local dev, never for production.
📄
Documents & collections
In MongoDB, a document is a JSON object like {"name": "Alice", "age": 30}. MongoDB auto-assigns an _id field (an ObjectId) to each document. A collection is a group of documents — you don't need to define its schema upfront.
🔢
Default port: 27017
MongoDB listens on port 27017 by default. To connect to it from your host machine or another container, you publish this port with -p 27017:27017. Connection strings for apps follow the pattern mongodb://user:pass@host:27017.
🔬 Command anatomy
docker run -d --name mongo-dev -p 27017:27017 -v mongo-data:/data/db -e MONGO_INITDB_ROOT_USERNAME=admin -e MONGO_INITDB_ROOT_PASSWORD=secret mongo:7
docker run

Create and start a new container from the specified image.

-d

Detached mode — run the container in the background and print its ID. MongoDB is a long-running service; you don't want it tying up your terminal.

--name

Name the container mongo-dev so you can reference it by name in subsequent exec and stop commands.

-p

Publish port 27017 on the host, forwarding to 27017 inside the container. Allows your apps and GUI clients to connect from localhost:27017.

-v

Mount the named volume mongo-data at /data/db — MongoDB's default data directory. Docker creates the volume automatically if it doesn't exist.

-e

Set environment variables for the initial root credentials. Only takes effect if the data directory is empty (first run on a fresh volume).

mongo:7

The official MongoDB image pinned to major version 7. Avoids accidentally pulling a breaking version bump later.

🪜 Step-by-step guide
1
Create a named volume for persistence
Create the volume explicitly before running the container. This makes it clear the volume is intentional and not an anonymous throwaway.
shell 
docker volume create mongo-data
expected output
mongo-data
2
Start the MongoDB container
Run MongoDB in detached mode with the volume mounted, port published, and root credentials set. This pulls the image on first run.
shell 
docker run -d \
  --name mongo-dev \
  -p 27017:27017 \
  -v mongo-data:/data/db \
  -e MONGO_INITDB_ROOT_USERNAME=admin \
  -e MONGO_INITDB_ROOT_PASSWORD=secret \
  mongo:7
⚠️
Don't use these credentials in production. For local development they're fine, but in any real environment use strong passwords and store them in Docker secrets or an env file that is never committed to version control.
3
Confirm MongoDB is ready
MongoDB takes a moment to initialize. Check the logs and wait for the "Waiting for connections" line before proceeding.
shell 
docker logs mongo-dev --tail 20
look for this line
{"t":{"$date":"2024-01-15T10:00:00.000+00:00"},"s":"I", "c":"NETWORK",
"msg":"Waiting for connections","attr":{"port":27017,"ssl":"off"}}
💡
You can also run docker logs -f mongo-dev to stream logs in real time. Press Ctrl+C to stop following — this won't stop the container.
4
Open a mongosh session
Use docker exec to open the MongoDB shell inside the running container, authenticated as the root user you configured.
shell 
docker exec -it mongo-dev \
  mongosh -u admin -p secret --authenticationDatabase admin
expected output
Current Mongosh Log ID: 65a4c2b3e1f8d2a0b4c6e8f0
Connecting to: mongodb://127.0.0.1:27017/?...
Using MongoDB: 7.0.4
Using Mongosh: 2.1.1

test> ▌
ℹ️
The prompt test> means you're in the default test database. All commands from here run inside mongosh inside the container.
5
Create a database and insert documents
Switch to a new database and insert a few documents into a collection. MongoDB creates both on the fly — no schema setup needed.
mongosh 
// Switch to (or create) a database called bookshop
use bookshop

// Insert a single document
db.books.insertOne({
  title: "The Pragmatic Programmer",
  authors: ["David Thomas", "Andrew Hunt"],
  year: 2019,
  tags: ["engineering", "career"]
})

// Insert multiple documents at once
db.books.insertMany([
  { title: "Clean Code", authors: ["Robert C. Martin"], year: 2008 },
  { title: "Designing Data-Intensive Applications", authors: ["Martin Kleppmann"], year: 2017 }
])
insertOne response
{
acknowledged: true,
insertedId: ObjectId('65a4c2b3e1f8d2a0b4c6e8f1')
}
6
Query the collection
Read back the documents you just inserted using find(), filter by field, and count results.
mongosh 
// Get all documents in the collection
db.books.find()

// Filter: books published after 2010
db.books.find({ year: { $gt: 2010 } })

// Count total documents
db.books.countDocuments()

// Find one specific book by title
db.books.findOne({ title: "Clean Code" })

// List all databases
show dbs

// List collections in current database
show collections
find() output (one document shown)
[
{
_id: ObjectId('65a4c2b3e1f8d2a0b4c6e8f1'),
title: 'The Pragmatic Programmer',
authors: [ 'David Thomas', 'Andrew Hunt' ],
year: 2019,
tags: [ 'engineering', 'career' ]
},
...
]
🍃
Notice the auto-generated _id field — MongoDB added it automatically. Every document gets a unique ObjectId unless you specify your own _id.
7
Update and delete documents
Practice the full CRUD cycle — update a field on an existing document, then delete one.
mongosh 
// Add a "rating" field to Clean Code
db.books.updateOne(
  { title: "Clean Code" },
  { $set: { rating: 4.8 } }
)

// Delete a document by title
db.books.deleteOne({ title: "Clean Code" })

// Confirm it's gone
db.books.countDocuments()
8
Exit mongosh and prove data persists
Exit the shell, stop and delete the container, then start a fresh one on the same volume. Your data should survive.
mongosh — exit shell 
exit
shell — stop and remove the container 
docker stop mongo-dev
docker rm mongo-dev
shell — start a brand new container on the same volume 
docker run -d --name mongo-dev -p 27017:27017 \
  -v mongo-data:/data/db \
  -e MONGO_INITDB_ROOT_USERNAME=admin \
  -e MONGO_INITDB_ROOT_PASSWORD=secret \
  mongo:7

# Re-open mongosh and verify books are still there
docker exec -it mongo-dev \
  mongosh -u admin -p secret --authenticationDatabase admin
mongosh — verify 
use bookshop
db.books.find()
✅
Your books are still there — the container was deleted but the volume survived. This is the core value of named volumes for databases: the data lifecycle is decoupled from the container lifecycle.
9
Clean up
Stop and remove the container, then optionally remove the volume if you're done with the data.
shell 
docker stop mongo-dev
docker rm mongo-dev

# Remove the volume only if you no longer need the data
docker volume rm mongo-data
⚠️
docker volume rm permanently deletes all data stored in the volume. Docker will refuse to remove a volume that is currently mounted by a running container.
🔍 What happened under the hood
Step 01
📦
Image pulled
mongo:7 layers downloaded from Docker Hub — includes mongod and mongosh.
Step 02
💾
Volume mounted
Docker mounted mongo-data at /data/db inside the container.
Step 03
🔐
Root user created
MongoDB's init scripts read the env vars and created the admin user on first start.
Step 04
🐚
Shell connected
mongosh opened a connection to mongod on localhost:27017 with auth.
Step 05
📄
BSON on disk
Documents written as BSON files inside the volume at /data/db.
Step 06
♻️
Container replaced
Old container deleted, new one reattached to the same volume — data intact.
