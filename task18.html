Task 18  ·  Intermediate

Modify a Dockerfile
& Rebuild

Change your Node.js app, bump the version tag, and rebuild — then observe how Docker's layer cache makes subsequent builds dramatically faster by reusing unchanged steps.

~20 minutes Task 17 completed Node.js app + Dockerfile
📖 Required concepts
⚡
Layer Cache
Every Dockerfile instruction creates a layer. Docker caches each layer by its content hash. On a rebuild, if nothing above a layer has changed, Docker reuses the cached result and prints "CACHED" — skipping that step entirely. This makes rebuilds fast.
💥
Cache Invalidation
When a layer's content changes, its cache is invalidated — and so is every layer after it. This is why instruction order matters: put things that change rarely (npm install) before things that change often (COPY . .).
🏷️
Semantic Versioning with Tags
Each build gets a new version tag like myapp:1.1.0. This creates a new image — the old myapp:1.0.0 still exists unchanged. Tags let you roll back or compare versions instantly.
📋
Optimal Layer Order
The golden rule: least-frequently-changed instructions go first. Copy only package.json before running npm install, then copy your source code. This way dependency installation is cached unless package.json actually changes.
🔢
ENV for Version Tracking
Setting ENV APP_VERSION inside the Dockerfile bakes the version into the image. Your app can read process.env.APP_VERSION at runtime — no need to hard-code it inside your source files.
🚫
--no-cache Flag
Passing --no-cache to docker build forces every layer to re-execute from scratch — useful when you suspect stale cached layers or want a fully fresh image (e.g. picking up OS security patches).
🔬 Command anatomy
docker build -t myapp:1.1.0 .
docker

The Docker CLI — sends the build command to the daemon.

build

Subcommand that reads a Dockerfile and builds a new image layer by layer.

-t

Tag flag — names the resulting image. Format is name:tag. Without it you'd get an unnamed image with only an ID.

myapp:1.1.0

The image name and semantic version tag. Creates a distinct image separate from myapp:1.0.0.

.

The build context — the directory Docker sends to the daemon. The . means the current directory.

🪜 Step-by-step guide
1
Review your starting point
Navigate into your project directory from Task 17. Confirm your files are in place and the existing image is tagged as myapp:1.0.0.
shell 
# Go to your project directory
cd myapp

# Check files exist
ls
# Should show: Dockerfile  index.js  package.json  .dockerignore

# Confirm the 1.0.0 image exists
docker images myapp
expected output
REPOSITORY TAG IMAGE ID CREATED SIZE
myapp 1.0.0 a4f3b2c1d9e7 2 hours ago 185MB
2
Do a baseline build and note the timing
Before making any changes, rebuild from scratch with --no-cache to get a real baseline. This forces every step to run — note how long it takes.
shell 
docker build --no-cache -t myapp:1.0.0 .
expected output (no cache — all steps execute)
[+] Building 18.4s (9/9) FINISHED
=> [internal] load build definition from Dockerfile
=> [1/5] FROM docker.io/library/node:18-alpine
=> [2/5] WORKDIR /app
=> [3/5] COPY package.json .
=> [4/5] RUN npm install 12.1s
=> [5/5] COPY . .
ℹ️
The slow step is always RUN npm install — it downloads packages from the internet. Remember this time; you'll see it disappear after the next rebuild.
3
Edit your app — change the response message
Open index.js and update the response text. This simulates a real feature change without touching package.json.
diff — index.js
@@ index.js @@ const http = require('http'); const PORT = process.env.PORT || 3000; - res.end('Hello from myapp v1.0.0\n'); + res.end(`Hello from myapp v${process.env.APP_VERSION || '1.1.0'}\n`);
💡
Using process.env.APP_VERSION instead of a hard-coded string means you only need to update the version in the Dockerfile — not inside the app code.
4
Update the Dockerfile — bump version, add ENV
Edit the Dockerfile to set the new version via ENV and expose the version as a build argument. Notice the order: npm install comes before COPY . ..
diff — Dockerfile
FROM node:18-alpine WORKDIR /app -ENV APP_VERSION=1.0.0 +ENV APP_VERSION=1.1.0 COPY package.json . RUN npm install COPY . . EXPOSE 3000 CMD ["node", "index.js"]

Here's the full updated Dockerfile for reference:

dockerfile 
FROM    node:18-alpine
WORKDIR /app
ENV     APP_VERSION=1.1.0

# Copy only package.json first — so npm install is cached
# unless dependencies actually change
COPY    package.json .
RUN     npm install --production

# Copy source code last — changes here won't bust npm cache
COPY    . .
EXPOSE  3000
CMD     ["node", "index.js"]
5
Rebuild with the new version tag
Build a new image tagged myapp:1.1.0. Watch closely for the word CACHED next to each step in the output.
shell 
docker build -t myapp:1.1.0 .
expected output (with cache — notice the speed)
[+] Building 1.3s (9/9) FINISHED
=> [internal] load build definition from Dockerfile
=> CACHED [1/5] FROM docker.io/library/node:18-alpine
=> CACHED [2/5] WORKDIR /app
=> CACHED [3/5] COPY package.json .
=> CACHED [4/5] RUN npm install --production
=> [5/5] COPY . . 0.1s
=> exporting to image
🚀
18 seconds down to 1.3 seconds — that's the cache at work. Steps 1–4 are CACHED because package.json didn't change. Only the source copy and ENV change triggered a re-run.
6
Verify both image versions exist
List your images — you should now have both 1.0.0 and 1.1.0 side by side.
shell 
docker images myapp
expected output
REPOSITORY TAG IMAGE ID CREATED SIZE
myapp 1.1.0 c9d2e3f4a1b8 10 seconds ago 185MB
myapp 1.0.0 a4f3b2c1d9e7 2 hours ago 185MB
ℹ️
Both versions coexist. The old image wasn't replaced — only a new one was added. You can roll back by simply running the old tag.
7
Run the new version and verify the change
Start a container from myapp:1.1.0 and call the endpoint to confirm the updated response.
shell 
docker run -d -p 3000:3000 --name myapp-v2 myapp:1.1.0
curl http://localhost:3000
expected output
Hello from myapp v1.1.0
8
Deliberately bust the cache — change a dependency
Now see what happens when you modify package.json. Add a new package — this invalidates the npm install layer and everything after it.
shell 
# Install a tiny package locally and save it to package.json
npm install --save ms

# Now rebuild — npm install will NOT be cached this time
docker build -t myapp:1.2.0 .
expected output (cache bust on npm install)
[+] Building 15.8s (9/9) FINISHED
=> CACHED [1/5] FROM docker.io/library/node:18-alpine
=> CACHED [2/5] WORKDIR /app
=> [3/5] COPY package.json . 0.1s
=> [4/5] RUN npm install 13.2s
=> [5/5] COPY . . 0.1s
⚠️
Expected! Changing package.json invalidated the COPY package.json layer — and every layer after it (including RUN npm install) had to re-execute. This is exactly how the cache cascade works.
9
Clean up
Stop running containers and optionally remove old image versions you no longer need.
shell 
# Stop and remove the running container
docker stop myapp-v2
docker rm myapp-v2

# Remove a specific old image version
docker rmi myapp:1.0.0

# Or prune all dangling (untagged) images at once
docker image prune
⚡ Cache behaviour reference

Understanding what triggers a cache miss helps you structure Dockerfiles for maximum speed.

What you changed First missed layer npm install re-runs?
index.js only COPY . . CACHED
ENV APP_VERSION in Dockerfile ENV APP_VERSION CACHED (if ENV is after RUN)
package.json (add/remove dep) COPY package.json . MISS — re-runs
Base image tag (e.g. node:18→20) FROM MISS — everything re-runs
Using --no-cache flag All layers MISS — forced full rebuild

🎯 What you learned
  • How Docker's layer cache works — unchanged layers are reused, changed layers (and everything after) re-execute
  • Why Dockerfile instruction order matters: stable steps first, volatile steps last
  • How to copy only package.json before npm install to cache dependencies separately from source code
  • How to bump a version tag to produce a new image without destroying the old one
  • How to use ENV APP_VERSION to bake a version into the image and read it at runtime
  • What --no-cache does and when to use it for a clean rebuild
