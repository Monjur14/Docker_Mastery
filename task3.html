Roadmap / Task 3 Beginner
Task 03  ·  Beginner

Detached Mode
& Port Mapping

Run Nginx in the background with -d, map container port 80 to host port 8080 with -p, and open it in your browser without blocking your terminal.

~7 minutes Docker installed Terminal + Browser
📖 Required concepts
🖥️
Foreground vs. detached mode
By default, docker run runs in the foreground — it attaches to your terminal and blocks it until the container stops. Detached mode (-d) starts the container in the background and immediately returns control of your terminal, printing only the container ID.
🔌
Why containers are isolated by default
A container has its own private network namespace. Even though Nginx inside the container listens on port 80, that port is not reachable from your host machine unless you explicitly publish it — this isolation is a security feature, not a bug.
🔀
Port mapping (publishing)
The -p HOST:CONTAINER flag forwards traffic from a port on your host machine to a port inside the container. -p 8080:80 means "anything hitting host port 8080 gets forwarded to port 80 inside the container."
🏷️
Naming containers
Without --name, Docker auto-generates a random name like brave_curie. Giving your container an explicit name makes it much easier to reference in later commands like docker logs or docker stop.
📜
Reattaching to logs
Since detached containers don't print to your terminal live, docker logs lets you view everything the container has written to stdout/stderr since it started — add -f to follow it live.
🛑
Stop vs. kill vs. remove
docker stop sends a graceful shutdown signal (SIGTERM, then SIGKILL after a timeout). docker kill terminates immediately. Neither deletes the container — that's what docker rm is for.
🔬 Command anatomy
docker run -d -p 8080:80 --name web nginx
docker

The CLI binary — how you talk to the Docker daemon from your terminal.

run

Creates a new container from an image and starts it.

-d

Short for --detach. Runs the container in the background and returns the terminal prompt immediately.

-p

Short for --publish. Format is HOST_PORT:CONTAINER_PORT — here, host 8080 maps to container 80.

--name

Assigns a human-readable name (web) instead of a random one.

nginx

The image to run — Docker pulls the official nginx:latest image from Docker Hub if it's not already cached.

Your browser
localhost:8080
forwards to
Nginx in container
0.0.0.0:80
The -p 8080:80 flag creates this bridge. Nginx itself never knows its traffic arrived via port 8080.
🪜 Step-by-step guide
1
Check port 8080 is free
Make sure nothing else on your host is already using port 8080 before you map it.
shell 
# macOS / Linux
lsof -i :8080

# Windows (PowerShell)
netstat -ano | findstr :8080
ℹ️
No output means the port is free — good to go. If something is using it, pick a different host port (e.g. 8081) in the next step.
2
Run Nginx in detached mode with port mapping
This starts Nginx in the background, maps host port 8080 to container port 80, and gives it a memorable name.
shell 
docker run -d -p 8080:80 --name web nginx
expected output
a7f3c9d21e4b8f0c6a1d5e9f2b7c4a8d3e6f1b0c9d2a7e4f8b1c3d6e9a0f2b7c
💡
In detached mode, Docker prints only the full container ID and returns your prompt immediately — no logs, no "Hello from Docker" banner.
3
Confirm it's running
docker ps (without -a) lists only currently running containers.
shell 
docker ps
expected output
CONTAINER ID IMAGE COMMAND STATUS PORTS NAMES
a7f3c9d21e4b nginx "/docker-entrypoint.…" Up 8 seconds 0.0.0.0:8080->80/tcp web
ℹ️
The PORTS column confirms the mapping: 0.0.0.0:8080->80/tcp means host port 8080 (all interfaces) forwards to container port 80.
4
Open it in your browser
Navigate to the mapped host port. You should see the default Nginx welcome page.
url 
http://localhost:8080
✅
You'll see "Welcome to nginx!" in the browser. That request left your browser, hit the host's network stack on 8080, and was forwarded by Docker into port 80 inside the container.
5
Test from the terminal too
You don't need a browser to verify it — curl works just as well and is handy for scripting.
shell 
curl -I http://localhost:8080
expected output
HTTP/1.1 200 OK
Server: nginx/1.27.0
Content-Type: text/html
6
View the container's logs
Every request you just made was logged by Nginx inside the container. Pull those logs into your terminal.
shell 
# One-time dump
docker logs web

# Live follow (Ctrl+C to stop watching)
docker logs -f web
ℹ️
docker logs -f only stops watching — it never stops the container itself. Press Ctrl+C any time.
7
Stop and remove the container
Gracefully stop the running container, then delete it. Both steps are needed since stop alone doesn't free the name.
shell 
docker stop web
docker rm web
💡
Shortcut: docker rm -f web force-stops and removes in one command — handy for quick cleanup during practice.
🔍 What happened under the hood
Step 01
📦
Container created
Daemon created a container from the nginx image with its own network namespace.
Step 02
🔀
iptables/NAT rule added
Docker configured a NAT rule forwarding host:8080 traffic into the container's network namespace on port 80.
Step 03
🚀
Detached from terminal
The daemon kept the container running in the background and returned your shell prompt.
Step 04
📡
Nginx listens on :80
Inside its isolated namespace, Nginx has no idea a mapping even exists — it just binds port 80 as usual.
Step 05
🌐
Browser request forwarded
Your request to localhost:8080 was rewritten by the NAT rule and delivered to the container.
Step 06
📝
Access logged
Nginx wrote the request to stdout, which Docker captured for docker logs to display.
🎯 What you learned
  • Detached mode (-d) runs containers in the background and frees your terminal
  • Container ports are isolated by default — -p HOST:CONTAINER is required to expose them
  • How to name containers with --name for easier reference
  • How to read the PORTS column in docker ps
  • How to inspect a detached container's output with docker logs and docker logs -f
  • The difference between stop, kill, and rm
