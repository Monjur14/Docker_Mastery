Task 14  ·  Intermediate

Build Your First
Dockerfile

Write a Dockerfile for a simple Node.js app — copy source, install dependencies, expose a port, and define the startup command. Turn your code into a portable image.

~20 minutes Docker + Node.js installed Terminal + text editor
📖 Required concepts
📄
What is a Dockerfile?
A Dockerfile is a plain-text recipe that tells Docker how to build a custom image. Each line is an instruction that adds a layer. Think of it as a shell script that Docker executes step-by-step inside a clean environment, producing a reproducible snapshot at the end.
🏗️
Build Context
When you run docker build, Docker sends your project folder — the build context — to the daemon. Instructions like COPY pull files from this context into the image.
🧅
Layer Caching
Each Dockerfile instruction creates an image layer. Docker caches these layers. If nothing changes between builds, Docker reuses the cached layer instead of re-running the instruction. This makes rebuilds fast — but only if you order instructions carefully (stable things first, frequently-changing code last).
🔢
Base Images
Every Dockerfile starts with a base image via FROM. For Node.js, node:20-alpine is a popular choice — it includes Node and npm on a tiny Alpine Linux base, keeping the final image small.
📂
WORKDIR
WORKDIR sets the working directory inside the container for all subsequent instructions. It's like running cd /app before every command. If the directory doesn't exist, Docker creates it. Always set a WORKDIR — never leave files scattered in the root filesystem.
🚪
EXPOSE vs. Port Mapping
EXPOSE in a Dockerfile is documentation — it says which port the app listens on. It does not publish the port to your host. Actual publishing happens at runtime with -p 3000:3000 in docker run (covered in Task 15).
🔬 Dockerfile instructions at a glance
Instruction What it does Why you need it
FROM Sets the base image to build on top of Every image must start somewhere — here we get Node + npm for free
WORKDIR Sets the working directory inside the container Keeps files organized; avoids cluttering the root filesystem
COPY Copies files from the build context into the image Gets your source code and package files into the container
RUN Executes a shell command during the build Installs dependencies — result is frozen into the image layer
EXPOSE Documents which port the app listens on Signals intent to operators and tools (does not publish the port)
CMD Default command to run when a container starts Starts your Node.js app; can be overridden at runtime
📁 Project structure

Create a new folder for your project. You'll write three files — a simple Express app, a package.json, and the Dockerfile itself.

📁 project layout
my-node-app/
├── Dockerfile <-- you'll write this
├── package.json
└── server.js
🪜 Step-by-step guide
1
Create the project folder
Make a new directory and navigate into it. Everything you create here becomes the build context Docker sends to the daemon.
shell 
mkdir my-node-app && cd my-node-app
2
Write the Node.js app — server.js
A minimal Express server that responds with a greeting on port 3000. This is what Docker will package and run.
server.js 
const express = require('express');
const app = express();
const PORT = process.env.PORT || 3000;

app.get('/', (req, res) => {
  res.send('Hello from inside Docker! 🐳');
});

app.listen(PORT, () => {
  console.log(`Server running on port ${PORT}`);
});
ℹ️
process.env.PORT lets you override the port via an environment variable at runtime — a good practice even in simple apps.
3
Write package.json
Defines your app's metadata and lists Express as a dependency. Docker will use this file to install dependencies during the build.
package.json 
{
  "name": "my-node-app",
  "version": "1.0.0",
  "description": "A simple Dockerized Node.js app",
  "main": "server.js",
  "scripts": {
    "start": "node server.js"
  },
  "dependencies": {
    "express": "^4.18.2"
  }
}
4
Write the Dockerfile
This is the main event. Create a file named exactly Dockerfile (no extension) in your project root. Every line below is explained in the section that follows.
Dockerfile 
# 1. Start from the official Node.js Alpine image
FROM node:20-alpine

# 2. Set the working directory inside the container
WORKDIR /app

# 3. Copy package files first (for layer cache efficiency)
COPY package*.json ./

# 4. Install production dependencies
RUN npm install --omit=dev

# 5. Copy the rest of your source code
COPY . .

# 6. Document which port the app listens on
EXPOSE 3000

# 7. Define the command to start the app
CMD ["node", "server.js"]
💡
Why copy package.json before the source code? If your source code changes but package.json didn't, Docker reuses the cached npm install layer — saving significant time on every rebuild.
5
Verify your project layout
Before building, confirm all three files are in place.
shell 
ls -1
expected output
Dockerfile
package.json
server.js
6
Build the image
Run docker build from your project directory. The -t flag tags the image with a human-readable name. The final . means "use the current directory as the build context."
shell 
docker build -t my-node-app .
expected output
[+] Building 18.3s (9/9) FINISHED
=> [internal] load build definition from Dockerfile 0.1s
=> [internal] load .dockerignore 0.1s
=> [internal] load metadata for docker.io/library/node 1.2s
=> [1/5] FROM node:20-alpine 5.4s
=> [2/5] WORKDIR /app 0.1s
=> [3/5] COPY package*.json ./ 0.1s
=> [4/5] RUN npm install --omit=dev 9.8s
=> [5/5] COPY . . 0.1s
=> exporting to image 0.2s
=> naming to docker.io/library/my-node-app:latest 0.0s
✅
Each [N/5] line corresponds to a Dockerfile instruction. Watch as Docker executes each one in order and caches the result.
7
Verify the image was created
List local images. Your new my-node-app image should appear alongside its size.
shell 
docker images my-node-app
expected output
REPOSITORY TAG IMAGE ID CREATED SIZE
my-node-app latest a3f8b2c91d02 10 seconds ago 178MB
ℹ️
~178 MB for a Node.js app might seem large, but the node:20-alpine base is already much smaller than alternatives. Task 31 (Multistage Builds) will show you how to shrink this further.
8
Observe cache in action — rebuild
Run the build again without changing any files. Notice how every layer is now served from cache — the entire build completes in under a second.
shell 
docker build -t my-node-app .
expected output — cached build
[+] Building 0.3s (9/9) FINISHED
=> [internal] load build definition from Dockerfile 0.0s
=> [1/5] FROM node:20-alpine 0.0s
=> CACHED [2/5] WORKDIR /app 0.0s
=> CACHED [3/5] COPY package*.json ./ 0.0s
=> CACHED [4/5] RUN npm install --omit=dev 0.0s
=> CACHED [5/5] COPY . . 0.0s
=> exporting to image 0.0s
💡
Now try editing server.js and rebuilding. Only the last two layers (COPY . . onward) are re-executed — npm install is still served from cache because package.json didn't change.
9
Inspect the image layers
Use docker history to see each layer, its size, and the command that created it.
shell 
docker history my-node-app
ℹ️
The heaviest layer will be npm install (pulling Express and its dependencies). The layer from COPY . . should be tiny — just your few source files.
🔬 Dockerfile anatomy

Here's how the two-phase COPY pattern maps to cache efficiency:

FROM node:20-alpine base ~130 MB · pulled once
↓
WORKDIR Create /app directory <1 KB · stable
↓
COPY package*.json only ~2 KB · rarely changes
↓
RUN npm install --omit=dev ~45 MB · cached until deps change
↓
COPY Source code (. .) ~5 KB · changes often → rebuild from here
↓
CMD node server.js metadata only · 0 bytes
🧅
Key insight: when you edit server.js, only the green layer and below are invalidated. The heavy npm install layer stays cached. This is why instruction order in a Dockerfile matters so much.
🔍 What happened under the hood
Step 01
📤
Context sent
Your project folder was tarred and sent to the Docker daemon.
Step 02
⬇️
Base pulled
node:20-alpine was pulled from Docker Hub (layers cached for future use).
Step 03
🏗️
Layers built
Each instruction created a new read-only layer on top of the previous.
Step 04
📦
npm installed
Express and its dependencies were fetched and frozen into a layer.
Step 05
🏷️
Image tagged
The final image was tagged as my-node-app:latest.
Step 06
💾
Stored locally
The image sits in Docker's local store, ready to be run or pushed.
