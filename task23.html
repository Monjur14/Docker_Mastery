Task 23  ·  Intermediate

Run a
MySQL Container

Start MySQL with environment variables for credentials, expose it on the host, persist data with a volume, and connect to it from a database client — inside or outside the container.

~15 minutes Docker installed Port 3306 free on host
📖Required concepts
🗄️
Why run MySQL in Docker?
Containerised MySQL means zero host installation — no system package manager, no version conflicts, no lingering daemons. You can run MySQL 5.7 and 8.4 side-by-side on the same machine, swap versions in seconds, and nuke the whole instance with docker rm.
🔑
Configuration via environment variables
The official MySQL image reads its initial setup from environment variables passed with -e. These set the root password, create a database, and seed a user on first startup — no interactive setup wizard required.
🔌
Port 3306
MySQL listens on port 3306 inside the container. With -p 3306:3306 you map it to the same port on the host, so any local client (TablePlus, DBeaver, mysql CLI) can connect to 127.0.0.1:3306 as normal.
💾
Data persistence with volumes
MySQL stores its data files in /var/lib/mysql inside the container. Without a volume, all your data disappears when the container is removed. Mounting a named volume at that path keeps your data safe across restarts and re-creations.
⏳
MySQL takes time to start
On first launch, MySQL initialises its data directory, sets up system tables, and applies the env-var config — this takes 5–20 seconds. Connection attempts before it's ready will be refused. Always wait for the "ready for connections" log line before connecting.
🔐
Never use these credentials in production
The env vars in this task set hardcoded passwords — fine for local development, not acceptable in any production or shared environment. In production, use Docker secrets or a secrets manager; never embed passwords in command history or source control.
🔑MySQL environment variables
Variable Example value What it does
MYSQL_ROOT_PASSWORD rootpass Required. Sets the password for the root superuser. required
MYSQL_DATABASE appdb Creates a database with this name on first startup. optional
MYSQL_USER appuser Creates a non-root user with full access to MYSQL_DATABASE. optional
MYSQL_PASSWORD apppass Password for MYSQL_USER. Required if MYSQL_USER is set. optional
MYSQL_ALLOW_EMPTY_PASSWORD yes Allows root login with no password — useful for throwaway local shells only. optional
MYSQL_RANDOM_ROOT_PASSWORD yes Generates a random root password and prints it to logs on first boot. optional
🔬Command anatomy
docker run -d --name mysql-dev -p 3306:3306 -v mysql-data:/var/lib/mysql -e MYSQL_ROOT_PASSWORD=rootpass -e MYSQL_DATABASE=appdb -e MYSQL_USER=appuser -e MYSQL_PASSWORD=apppass mysql:8.4
docker run

Create and start a new container.

-d

Detached mode — run in the background and print the container ID. Essential for long-running services like databases.

--name

Give the container a memorable name (mysql-dev) so you can reference it without copying IDs.

-p

Publish port — maps host port 3306 to container port 3306, making MySQL reachable from the host at 127.0.0.1:3306.

-v

Mount the named volume mysql-data at /var/lib/mysql so data survives container restarts and deletions.

-e

Set environment variables for MySQL's entrypoint script to configure root password, database name, and app user on first boot.

mysql:8.4

The image to use. Pinning a specific version (8.4) rather than latest prevents unexpected upgrades.

🪜Step-by-step guide
1
Pull the MySQL image
Fetch the image explicitly first so you can see the download progress before running. Pinning a version instead of latest prevents surprise upgrades.
shell
docker pull mysql:8.4
expected output
8.4: Pulling from library/mysql
a480a496ba95: Pull complete
...
Status: Downloaded newer image for mysql:8.4
docker.io/library/mysql:8.4
2
Run MySQL with credentials and a named volume
This single command starts MySQL in the background, sets up root and an app user, creates a database, and persists all data to a named volume.
shell
docker run -d \
  --name mysql-dev \
  -p 3306:3306 \
  -v mysql-data:/var/lib/mysql \
  -e MYSQL_ROOT_PASSWORD=rootpass \
  -e MYSQL_DATABASE=appdb \
  -e MYSQL_USER=appuser \
  -e MYSQL_PASSWORD=apppass \
  mysql:8.4
⚠️
This command puts passwords in your shell history. For real projects, load credentials from a .env file with --env-file .env instead of passing -e flags inline.
3
Wait for MySQL to be ready
MySQL initialises its data directory on the first boot — this takes several seconds. Tail the logs and wait for the "ready for connections" message before attempting to connect.
shell
docker logs -f mysql-dev
look for these lines (may take 10–20 s)
2024-01-15T10:23:11.000Z 0 [System] [MY-010931] Starting MySQL 8.4 ...
2024-01-15T10:23:14.000Z 1 [System] [MY-013576] InnoDB initialization has started.
2024-01-15T10:23:16.000Z 1 [System] [MY-013577] InnoDB initialization has ended.
...
2024-01-15T10:23:22.000Z 0 [System] [MY-010931] /usr/sbin/mysqld: ready for connections.
Version: '8.4.0' socket: '/var/run/mysqld/mysqld.sock' port: 3306 MySQL Community Server - GPL
💡
Press Ctrl + C to stop following the logs — the container keeps running. The -f flag just tails the stream; it doesn't attach to the process.
4
Connect using the MySQL CLI — inside the container
The simplest way to verify the setup: exec into the running container and open a mysql shell. No client installation on the host needed.
shell
docker exec -it mysql-dev \
  mysql -u appuser -papppass appdb
expected output
mysql: [Warning] Using a password on the command line interface can be insecure.
Welcome to the MySQL monitor. Commands end with ; or \g.
Your MySQL connection id is 8
Server version: 8.4.0 MySQL Community Server - GPL

mysql>
ℹ️
Note there is no space between -p and the password — -papppass — this is the MySQL CLI convention. If you use -p apppass with a space, MySQL prompts you interactively and treats "apppass" as the database name.
5
Run some SQL to verify everything works
At the mysql> prompt, confirm you're in the right database, create a table, insert a row, and query it back.
sql
-- Confirm current database
SELECT DATABASE();

-- Create a table
CREATE TABLE users (
  id   INT AUTO_INCREMENT PRIMARY KEY,
  name VARCHAR(100)
);

-- Insert a row
INSERT INTO users (name) VALUES ('Docker');

-- Query it back
SELECT * FROM users;
expected output
+------------+
| DATABASE() |
+------------+
| appdb |
+------------+

Query OK, 0 rows affected (0.02 sec)
Query OK, 1 row affected (0.01 sec)

+----+--------+
| id | name |
+----+--------+
| 1 | Docker |
+----+--------+
💡
Type exit or press Ctrl + D to leave the MySQL shell and return to your host terminal.
6
Connect from a GUI client on your host
Because port 3306 is published to the host, any desktop database client can connect to 127.0.0.1 just like a locally installed MySQL server.
connection settings
Host     : 127.0.0.1
Port     : 3306
User     : appuser
Password : apppass
Database : appdb
new connection · mysql
Server Host : localhost
Port        : 3306
Database    : appdb
Username    : appuser
Password    : apppass
ℹ️
DBeaver may prompt you to download the MySQL JDBC driver the first time — click Download to let it install automatically.
shell — requires mysql client installed on host
mysql -h 127.0.0.1 -P 3306 \
     -u appuser -papppass appdb
ℹ️
Use -h 127.0.0.1 (not localhost) to force a TCP connection. localhost on some systems triggers a socket connection, which won't work against the container.
7
Verify data persists across restarts
Stop and remove the container, then re-create it with the same volume. The data you inserted in step 5 should still be there, proving the volume is doing its job.
shell
# Remove the container (NOT the volume)
docker rm -f mysql-dev

# Re-create with the same volume — data survives
docker run -d \
  --name mysql-dev \
  -p 3306:3306 \
  -v mysql-data:/var/lib/mysql \
  -e MYSQL_ROOT_PASSWORD=rootpass \
  -e MYSQL_DATABASE=appdb \
  -e MYSQL_USER=appuser \
  -e MYSQL_PASSWORD=apppass \
  mysql:8.4

# Wait for ready, then query — row is still there
docker exec -it mysql-dev \
  mysql -u appuser -papppass appdb \
  -e "SELECT * FROM users;"
8
Connect as root for admin tasks
Some operations — creating extra users, inspecting all databases, checking server status — require the root account.
shell
docker exec -it mysql-dev \
  mysql -u root -prootpass
sql — inside root shell
-- List all databases
SHOW DATABASES;

-- List all users
SELECT user, host FROM mysql.user;

-- Check server status
SHOW STATUS LIKE 'Uptime';
9
Clean up
Stop and remove the container. The volume is left intact by default — add the second command only if you want to wipe all data too.
shell
# Remove the container only
docker rm -f mysql-dev

# Also remove the data volume (wipes all data — irreversible)
docker volume rm mysql-data
🔍What happened under the hood
Step 01
📂
Volume mounted
Docker created mysql-data and mounted it at /var/lib/mysql before the process started.
Step 02
⚙️
Entrypoint ran
The image's entrypoint script detected an empty data dir and ran mysqld --initialize.
Step 03
🔑
Credentials applied
Root password was set, appdb was created, and appuser was granted access — all from env vars.
Step 04
🌐
Port published
Docker's proxy bound host port 3306 to the container's port 3306, making MySQL reachable from the host.
Step 05
🗄️
mysqld started
After init, the entrypoint launched the main mysqld process and began accepting TCP connections.
Step 06
💾
Data persisted
All writes (tables, rows) went to the volume. Removing the container left the volume — and all data — intact.
🎯What you learned
  • How to configure MySQL on first boot using -e environment variables — no interactive setup needed
  • The purpose of each MySQL env var: root password, database name, app user and password
  • Why you must wait for the "ready for connections" log line before attempting any client connection
  • How to connect from inside the container via docker exec, and from the host via a published port
  • The -h 127.0.0.1 vs localhost distinction when connecting from outside the container
  • How a named volume at /var/lib/mysql keeps your data safe across container deletions and re-creations
