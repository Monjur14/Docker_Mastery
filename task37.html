Task 37  ·  Advanced

Scan Images for
Vulnerabilities

Use docker scout and trivy to scan a custom image for known CVEs, read the security report, and apply fixes that actually shrink your attack surface.

~25 minutes Docker Desktop 4.17+ or Trivy installed A built image to scan (Task 15+)
📖Required concepts
🐛
What is a CVE?
A Common Vulnerability and Exposure is a publicly disclosed security flaw in a specific version of a piece of software, assigned a unique ID like CVE-2024-21626. The National Vulnerability Database rates each one's severity. When your image contains an OS package or npm library with a known CVE, scanners flag it.
🔍
How Image Scanners Work
A scanner unpacks your image layers and builds an inventory of every installed package — OS packages (apt/apk), language packages (npm/pip/gems), and binaries. It then cross-references each version against a vulnerability database (NVD, GitHub Advisories, OS vendor advisories) to find known CVEs.
📊
CVSS Score
The Common Vulnerability Scoring System gives each CVE a numeric score from 0 to 10. It measures exploitability (how easy to attack?), impact (how bad if exploited?), and scope (can it jump to other systems?). Scores map to severity labels: Critical, High, Medium, Low.
🩹
Fixed vs Unfixed CVEs
A scanner distinguishes between CVEs that have a patched version available (actionable — upgrade the package) and ones that are unfixed (the vendor hasn't released a patch yet). Focus on fixed CVEs first — they're the easiest wins with the highest impact.
🏔️
Base Image is the Biggest Factor
Most CVEs in a container image come from the OS base layer, not your code. Switching from node:20 (Debian, ~200 packages) to node:20-alpine (musl libc, ~15 packages) or node:20-slim typically eliminates 80–90% of all CVEs in one change.
📌
Pinning and Rebuilding
New CVEs are disclosed constantly. An image that was clean last month may have Critical vulnerabilities today. The remedy is to rebuild regularly (at least weekly in CI) using a fresh pull of the base image — even if your code hasn't changed — so OS packages stay up to date.
🔬Anatomy of a CVE identifier
CVE - 2024 - 21626
CVE

Common Vulnerabilities and Exposures — the naming standard maintained by MITRE. All public vulnerability databases use this ID format.

year

The year the vulnerability was assigned a CVE ID — not necessarily when the bug was introduced or discovered. High-profile bugs from the past often get IDs years later.

number

A sequential identifier within that year. Lower numbers were assigned earlier. Numbers above 10000 are common for busy years. Look up any CVE at nvd.nist.gov.

🛠️Docker Scout vs Trivy

First-party scanner integrated into Docker Desktop and the CLI. Zero setup if you already have Docker Desktop 4.17+.

ℹ️
This task covers both tools. Use Scout for quick local scans with Docker Desktop. Use Trivy in CI pipelines and when you need JSON output, threshold enforcement, or to scan non-image targets.
🪜Step-by-step guide
1
Build a deliberately vulnerable image to scan
Start with an older base image — this guarantees a realistic number of CVEs to work with, so the scan output is interesting rather than empty.
Dockerfile.scan-demo
# Intentionally older image — produces scan results worth reading
FROM node:18

WORKDIR /app
COPY package*.json ./
RUN npm install
COPY . .

EXPOSE 3000
CMD ["node", "index.js"]
shell
docker build -f Dockerfile.scan-demo -t myapp:scan-demo .
2
Install Trivy
Trivy has packages for every platform. Choose the method for your OS:
macOS — homebrew
brew install aquasecurity/trivy/trivy
Linux — install script
curl -sfL https://raw.githubusercontent.com/aquasecurity/trivy/main/contrib/install.sh | sh -s -- -b /usr/local/bin
any platform — run as a container (no install)
docker run --rm -v /var/run/docker.sock:/var/run/docker.sock \
  aquasec/trivy image myapp:scan-demo
💡
Running Trivy as a container is the fastest way to try it — no install, always the latest version. Mount the Docker socket so Trivy can read local images directly.
3
Run your first Trivy scan
The default scan checks OS packages and language dependencies. Trivy downloads its vulnerability database on the first run.
shell
trivy image myapp:scan-demo

4
Filter to only Critical and High
Use --severity to focus on what needs immediate action. Ignoring LOW and MEDIUM noise helps you triage efficiently.
shell
# Show only Critical and High — your immediate to-do list
trivy image --severity CRITICAL,HIGH myapp:scan-demo

# Show only issues that have a fix available
trivy image --severity CRITICAL,HIGH --ignore-unfixed myapp:scan-demo
💡
--ignore-unfixed hides CVEs where no patch exists yet — there's nothing you can do about them right now. This gives you a clean actionable list.
5
Scan with Docker Scout
If you have Docker Desktop 4.17+ or the Scout CLI, run a quickview and a full CVE report. Scout is built into the docker command — no separate install needed.
shell
# One-line summary of CVE counts by severity
docker scout quickview myapp:scan-demo

# Full CVE list with fix recommendations
docker scout cves myapp:scan-demo

# Filter to Critical/High only
docker scout cves --only-severity critical,high myapp:scan-demo

# Compare your image against the recommended base image
docker scout recommendations myapp:scan-demo

6
Apply Fix #1 — switch to a minimal base image
The single highest-impact change: replace node:18 (Debian, full) with node:20-alpine. Alpine ships with ~15 OS packages versus Debian's ~200.
Dockerfile — after base image upgrade
# Before: node:18  → 87 CVEs, 3 Critical
# After:  node:20-alpine → typically <5 CVEs, 0 Critical
FROM node:20-alpine

WORKDIR /app
COPY package*.json ./
RUN npm ci --omit=dev   # omit devDependencies too
COPY . .

EXPOSE 3000
CMD ["node", "index.js"]
shell
docker build -t myapp:hardened .
trivy image --severity CRITICAL,HIGH myapp:hardened
scan after switching to alpine

✅
Zero Critical and High CVEs — down from 17 — with a single line change. The image size also dropped from ~392 MB to ~145 MB as a bonus.
7
Apply Fix #2 — audit and update npm packages
CVEs also come from your node_modules. Use npm's built-in audit and Trivy's package scanner together.
shell
# Audit your local node_modules for known CVEs npm audit # Auto-fix compatible upgrades (semver-safe) npm audit fix # Also scan just the npm packages inside a built image trivy image --scanners vuln --pkg-types library myapp:hardened
ℹ️
npm audit fix --force will bump major versions that may include breaking changes. Review its output carefully before committing. Prefer the non-force version first.
8
Export a JSON report for CI
In a CI pipeline you need machine-readable output to gate a build. Trivy's JSON and SARIF formats integrate with GitHub Actions, GitLab, and most security platforms.
shell
# JSON report for programmatic processing trivy image --format json --output scan-report.json myapp:hardened # SARIF format for GitHub Code Scanning tab trivy image --format sarif --output trivy-results.sarif myapp:hardened # Exit with non-zero code if any CRITICAL found — blocks CI trivy image --exit-code 1 --severity CRITICAL myapp:hardened
💡
--exit-code 1 makes Trivy exit with a failure code when it finds CVEs above your threshold. This is how you enforce a policy: CI fails the build and blocks the image from being pushed to production.
9
Add a CI scan step to your workflow
A scan that only runs locally isn't a security policy. Add Trivy to your GitHub Actions workflow so every pushed image is scanned automatically.
.github/workflows/scan.yml
name: Build and Scan on: [push] jobs: scan: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - name: Build image run: docker build -t myapp:${{ github.sha }} . - name: Run Trivy scan uses: aquasecurity/trivy-action@master with: image-ref: 'myapp:${{ github.sha }}' format: 'sarif' output: 'trivy-results.sarif' severity: 'CRITICAL,HIGH' exit-code: '1' # fail build on findings - name: Upload results to GitHub Security tab uses: github/codeql-action/upload-sarif@v3 if: always() # upload even if scan failed with: sarif_file: 'trivy-results.sarif'
10
Scan a Dockerfile for misconfigurations
Trivy can also lint your Dockerfile for security misconfigurations before you even build — catching issues like running as root, missing HEALTHCHECK, or pinning images to latest.
shell
# Scan the Dockerfile itself for misconfigurations trivy config . # Scan a specific file trivy config --file-patterns 'dockerfile:Dockerfile' .

🩹Common CVE sources and fixes
Root cause Typical severity Fix
Outdated OS base image
Critical/High
Switch to node:20-alpine or rebuild weekly with --pull
Full Debian/Ubuntu base
High
Switch to -alpine or -slim variant — removes 90% of packages
Vulnerable npm dependency
High
Run npm audit fix and update package-lock.json
devDependencies in image
Medium
Use npm ci --omit=dev or multistage build to exclude them
Installed build tools (gcc, make)
Medium
Use multistage build — install tools in builder stage, copy only artifacts
Unfixed CVE (no patch exists)
Varies
Add to .trivyignore with expiry date and risk acknowledgement
🔍What happened under the hood
Step 01
📦
Layers unpacked
Trivy pulled and unpacked each image layer into a virtual filesystem for analysis.
Step 02
📋
SBOM built
Every installed package — OS, npm, pip, gems — was inventoried into a Software Bill of Materials.
Step 03
🗄️
DB queried
Each package version was cross-referenced against NVD, GitHub Advisories, and OS vendor advisories.
Step 04
📊
CVEs matched
Matching CVEs were collected with their CVSS scores, severity, and fixed-version data.
Step 05
🖨️
Report rendered
Results were formatted as a table, JSON, or SARIF depending on your output flag.
Step 06
🚦
Exit code set
With --exit-code 1, Trivy exited non-zero if findings exceeded the severity threshold.
