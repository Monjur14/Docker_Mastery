Task 24  ·  Intermediate

Connect a Backend App
to MySQL

Network a Node.js app container to your MySQL container using a shared Docker bridge network, so they talk by container name — no hardcoded IPs, no host ports between them.

~20 minutes Task 23 completed Node.js (for building image)
📖Required concepts
🌐
Containers talk via Docker networks
Two containers can only reach each other if they're on the same Docker network. On a user-defined bridge network, each container is reachable by its container name as a hostname — Docker's embedded DNS resolves it. No need to know or hardcode IP addresses.
🚫
The default bridge doesn't do DNS
The default Docker bridge network (bridge) does not support container name resolution. You must create a user-defined network with docker network create to get automatic DNS between containers.
🔗
The host is not the MySQL host
From inside the app container, localhost refers to the app container itself — not MySQL. The MySQL host is the container name you gave it (e.g. mysql-dev). This is the most common mistake when connecting containers.
⏳
Apps must handle MySQL startup delay
MySQL takes several seconds to initialise. If your app tries to connect before MySQL is ready, it will fail. Good apps use a retry loop with backoff — or wait-for scripts — rather than crashing immediately on a refused connection.
📦
The app is also a container
You write a small Node.js script, package it into a Docker image with a Dockerfile, and run it as a container alongside MySQL. The app image only needs Node.js; it does not need MySQL installed inside it.
🔑
Credentials via environment variables
Pass database credentials to the app container with -e flags — the same pattern used to configure MySQL itself. The app reads them with process.env, keeping secrets out of source code.
🏗️What we're building
app-net (bridge)
node-app
port 3000 → host
mysql-dev:3306
mysql-dev
port 3306 (internal)

Both containers share app-net. The app reaches MySQL at hostname mysql-dev, port 3306 — only the app port is published to the host.

🪜Step-by-step guide
1
Create a shared network
User-defined bridge networks give containers automatic DNS — they can reach each other by container name. Create one before starting either container.
shell
docker network create app-net
expected output
a3f7c2d18e4b9012... (network ID)
2
Start MySQL on the shared network
Same as Task 23 but now with --network app-net added. Notice we still publish port 3306 to the host — useful for direct client access — but the app container won't use that path.
shell
docker run -d \
  --name mysql-dev \
  --network app-net \
  -p 3306:3306 \
  -v mysql-data:/var/lib/mysql \
  -e MYSQL_ROOT_PASSWORD=rootpass \
  -e MYSQL_DATABASE=appdb \
  -e MYSQL_USER=appuser \
  -e MYSQL_PASSWORD=apppass \
  mysql:8.4
💡
Already have a mysql-dev container from Task 23? Connect it to the new network without recreating it: docker network connect app-net mysql-dev
3
Create the app project files
Build a minimal directory with three files: the Node.js script, its package file, and a Dockerfile to containerise it.
node-app/
  ├── app.js
  ├── package.json
  └── Dockerfile
shell
mkdir node-app && cd node-app
4
Write the Node.js app
A small script that reads DB credentials from environment variables, retries the connection until MySQL is ready, runs a query, and exits.
app.js 
const mysql = require('mysql2/promise');

const config = {
  host:     process.env.DB_HOST     || 'localhost',
  port:     process.env.DB_PORT     || 3306,
  user:     process.env.DB_USER     || 'appuser',
  password: process.env.DB_PASSWORD || 'apppass',
  database: process.env.DB_NAME     || 'appdb',
};

async function connectWithRetry(retries = 10, delay = 3000) {
  for (let i = 1; i <= retries; i++) {
    try {
      const conn = await mysql.createConnection(config);
      console.log(`✅ Connected to MySQL at ${config.host}:${config.port}`);
      return conn;
    } catch (err) {
      console.log(`⏳ Attempt ${i}/${retries} — waiting for MySQL... (${err.message})`);
      if (i === retries) throw err;
      await new Promise(r => setTimeout(r, delay));
    }
  }
}

async function main() {
  const conn = await connectWithRetry();

  // Create table if it doesn't exist
  await conn.execute(`
    CREATE TABLE IF NOT EXISTS messages (
      id      INT AUTO_INCREMENT PRIMARY KEY,
      text    VARCHAR(255),
      created TIMESTAMP DEFAULT CURRENT_TIMESTAMP
    )
  `);

  // Insert a row
  await conn.execute(
    'INSERT INTO messages (text) VALUES (?)',
    ['Hello from the app container!']
  );

  // Read it back
  const [rows] = await conn.execute('SELECT * FROM messages');
  console.log('\n📋 Messages in database:');
  console.table(rows);

  await conn.end();
  console.log('\n✅ Done — connection closed.');
}

main().catch(err => {
  console.error('❌ Fatal error:', err.message);
  process.exit(1);
});
ℹ️
The retry loop is critical — it keeps trying every 3 seconds for up to 30 seconds while MySQL initialises. Without it, the app would crash instantly if MySQL isn't ready on the first attempt.
5
Create package.json
Declare the single dependency — mysql2 — a fast, Promise-native MySQL client for Node.js.
package.json
{
  "name": "node-app",
  "version": "1.0.0",
  "main": "app.js",
  "dependencies": {
    "mysql2": "^3.9.0"
  }
}
6
Write the Dockerfile
A minimal Dockerfile that installs dependencies and runs the script. Using node:22-alpine keeps the image small.
Dockerfile
FROM node:22-alpine

WORKDIR /app

# Install dependencies first (better layer caching)
COPY package.json .
RUN npm install --omit=dev

# Copy app source
COPY app.js .

CMD ["node", "app.js"]
7
Build the app image
Run this from inside the node-app/ directory. The . at the end is the build context — Docker sends the entire current directory to the daemon.
shell
docker build -t node-app:1.0 .
expected output
[+] Building 12.4s (9/9) FINISHED
=> [internal] load build definition from Dockerfile
=> [1/4] FROM docker.io/library/node:22-alpine
=> [2/4] WORKDIR /app
=> [3/4] COPY package.json .
=> [4/4] RUN npm install --omit=dev
=> exporting to image
=> naming to docker.io/library/node-app:1.0
8
Wait for MySQL to be ready
If MySQL was just started, wait until it logs "ready for connections" before running the app — or let the app's retry loop handle it (it will, up to 10 attempts).
shell
docker logs -f mysql-dev 2>&1 | grep "ready for connections"
💡
This command exits automatically once the pattern is matched — no need to Ctrl+C. If MySQL was already running from Task 23, skip this step entirely.
9
Run the app container
Connect it to app-net and pass the MySQL connection details as environment variables. The key one is DB_HOST=mysql-dev — the container name, not an IP address.
shell
docker run --rm \
  --name node-app \
  --network app-net \
  -e DB_HOST=mysql-dev \
  -e DB_PORT=3306 \
  -e DB_USER=appuser \
  -e DB_PASSWORD=apppass \
  -e DB_NAME=appdb \
  node-app:1.0
ℹ️
--rm automatically removes the container when it exits. Since this app runs to completion and exits, there's no reason to keep the container around afterwards.
10
Read the output
Watch the retry loop, the successful connection, and the query result all print in sequence.
expected output
⏳ Attempt 1/10 — waiting for MySQL... (connect ECONNREFUSED)
⏳ Attempt 2/10 — waiting for MySQL... (connect ECONNREFUSED)
✅ Connected to MySQL at mysql-dev:3306

📋 Messages in database:
┌─────────┬────┬──────────────────────────────────┬─────────────────────┐
│ (index) │ id │ text │ created │
├─────────┼────┼──────────────────────────────────┼─────────────────────┤
│ 0 │ 1 │ 'Hello from the app container!' │ 2024-01-15T10:31... │
└─────────┴────┴──────────────────────────────────┴─────────────────────┘

✅ Done — connection closed.
💡
Run the same docker run command again — you'll see a second row appear in the table. Each run inserts a new message, proving the data is persisting in the MySQL volume.
11
Inspect the network to confirm both containers are connected
Verify Docker's view of the network and see which containers are attached with their assigned IPs.
shell
docker network inspect app-net
expected output (abridged)
[
{
"Name": "app-net",
"Driver": "bridge",
"Containers": {
"...": { "Name": "mysql-dev", "IPv4Address": "172.18.0.2/16" },
}
}
]
12
Diagnose a connection problem (optional)
If the app can't reach MySQL, debug from inside the app network using a one-shot Alpine container on the same network.
shell
# Can we resolve mysql-dev by name?
docker run --rm --network app-net alpine \
  nslookup mysql-dev

# Can we reach port 3306?
docker run --rm --network app-net alpine \
  sh -c "nc -zv mysql-dev 3306"
⚠️
If nslookup mysql-dev fails, the MySQL container isn't on app-net. Check with docker network inspect app-net and reconnect if needed: docker network connect app-net mysql-dev
13
Clean up
Stop MySQL, remove the network, and optionally wipe the data volume.
shell
# Stop and remove MySQL
docker rm -f mysql-dev

# Remove the network
docker network rm app-net

# Remove the data volume (only if done with this data)
docker volume rm mysql-data

# Remove the app image
docker rmi node-app:1.0
🔍What happened under the hood
Step 01
🌐
Network created
Docker created a virtual bridge interface on the host and a private subnet for app-net.
Step 02
🔍
DNS registered
When each container joined the network, Docker's embedded DNS registered its container name as a resolvable hostname.
Step 03
📡
Name resolved
The app looked up mysql-dev — Docker DNS returned the container's private IP on app-net.
Step 04
🔗
TCP connection
The app opened a TCP connection to MySQL's private IP on port 3306 — entirely inside Docker's virtual network.
Step 05
🔑
Auth handshake
MySQL verified the credentials (appuser / apppass) and granted access to appdb.
Step 06
✅
Query executed
SQL ran inside MySQL, results streamed back over the same TCP connection, app printed the table and exited cleanly.
🎯What you learned
  • Why user-defined bridge networks are required for container-to-container DNS — the default bridge network doesn't support name resolution
  • That from inside an app container, localhost is the app itself — the MySQL host is the container name on the shared network
  • How to build a Node.js app image with a Dockerfile and connect it to a database at runtime via environment variables
  • Why retry logic is essential when one container depends on another that takes time to initialise
  • How to inspect a network with docker network inspect and debug connectivity with a one-shot Alpine container
  • The full flow: create network → start DB → build image → run app — the manual version of what Docker Compose automates
