Task 42  ·  Advanced

Deploy a Container
on a VPS

SSH into a cloud server, install Docker, and deploy your stack in production with docker compose up -d. Go from a blank Ubuntu server to a live, public-facing app.

~45 minutes VPS + SSH key Terminal / Shell
📖 Required concepts
🖥️
What is a VPS?
A Virtual Private Server is a rented Linux machine in the cloud with a public IP address. Providers like DigitalOcean, Hetzner, Linode, and Vultr offer 1-CPU/1 GB RAM instances for $4–6/month — enough to host side projects, APIs, and small web apps.
🔑
SSH key authentication
SSH keys replace passwords for server login. You generate a key pair — private key stays on your machine, public key goes onto the server. Every provider lets you paste your public key at VPS creation so you can log in immediately without a password.
🚀
docker compose up -d
The -d flag (detached) starts all services in the background and returns your shell immediately. The containers keep running after you close the SSH session. This is the standard production launch command for a Compose stack.
🔁
restart: unless-stopped
Adding restart: unless-stopped to a Compose service tells Docker to automatically restart it after crashes or server reboots — until you explicitly stop it with docker compose down.
🛡️
Firewall basics — ufw
A fresh VPS has all ports open by default. ufw (Uncomplicated Firewall) lets you close everything and whitelist only what you need — typically port 22 (SSH), 80 (HTTP), and 443 (HTTPS). Anything else is blocked at the OS level.
🔄
Updating a running stack
The production update workflow is: pull new code or images on the server, then run docker compose pull && docker compose up -d. Compose replaces only the containers whose image or config changed, keeping the rest running.

🔬 Command anatomy
docker compose -f compose.yml up -d --pull always
docker compose

The Compose CLI — orchestrates multi-container applications from a YAML definition file.

-f compose.yml

Explicit file path. Optional if the file is named compose.yml or docker-compose.yml in the current directory.

up

Create and start all services defined in the file. Recreates containers whose config has changed.

-d

Detached — run everything in the background. Your SSH session can close without stopping the stack.

--pull always

Pull the latest version of every image before starting, even if a local copy exists. Ensures production runs current code.

🪜 Step-by-step guide
1
Generate an SSH key pair (if you don't have one)
Run this on your local machine. The public key is what you upload to the VPS provider — the private key never leaves your laptop.
local machine 
# Generate a modern ed25519 key pair
ssh-keygen -t ed25519 -C "deploy@myapp" -f ~/.ssh/vps_deploy

# Print the public key — paste this into your VPS provider's SSH key field
cat ~/.ssh/vps_deploy.pub
💡
If you already have an ~/.ssh/id_ed25519.pub or ~/.ssh/id_rsa.pub, use that — no need to generate a new one.
2
Provision the VPS and SSH in
Create an Ubuntu 24.04 droplet/instance at your provider with your SSH public key attached. Once it's running, connect from your local terminal.
local machine 
# Replace with your server's actual IP address
ssh -i ~/.ssh/vps_deploy root@YOUR_SERVER_IP
expected output
Welcome to Ubuntu 24.04 LTS (GNU/Linux 6.8.0-31-generic x86_64)
root@ubuntu-vps:~#
ℹ️
Some providers use ubuntu instead of root as the default username — check your provider's documentation. The prompt you see confirms you're on the server.
3
Update the system and lock down the firewall
All commands from here run on the server. Update packages and configure ufw to only allow SSH, HTTP, and HTTPS.
server — root@vps 
# Update all system packages
apt-get update -y && apt-get upgrade -y

# Configure the firewall
ufw default deny incoming
ufw default allow outgoing
ufw allow 22/tcp    # SSH — do this BEFORE enabling or you'll lock yourself out
ufw allow 80/tcp    # HTTP
ufw allow 443/tcp   # HTTPS
ufw --force enable

# Confirm status
ufw status
⚠️
Always allow port 22 before enabling ufw. If you enable the firewall first without the SSH rule, you will be permanently locked out of the server.
4
Install Docker and Docker Compose
Use Docker's official install script — the fastest and most reliable method for Ubuntu. It installs the engine, CLI, and Compose plugin in one go.
server — root@vps 
# Download and run Docker's official install script
curl -fsSL https://get.docker.com | sh

# Verify the installation
docker version
docker compose version
expected output (compose version)
Docker Compose version v2.27.0
ℹ️
The install script adds the Docker apt repository and installs the latest stable release. On subsequent visits, apt-get upgrade will keep Docker up to date automatically.
5
Create a deploy directory and write the Compose file
Set up a directory for your project on the server, then create the compose.yml. This example deploys nginx as a stand-in — swap it for your actual app image.
server — root@vps 
mkdir -p /opt/myapp && cd /opt/myapp
/opt/myapp/compose.yml 
services:
  web:
    image: nginx:alpine
    ports:
      - "80:80"
    volumes:
      - ./html:/usr/share/nginx/html:ro
    restart: unless-stopped

  db:
    image: postgres:16
    environment:
      POSTGRES_USER: app
      POSTGRES_PASSWORD: "${DB_PASSWORD}"
      POSTGRES_DB: myapp
    volumes:
      - pgdata:/var/lib/postgresql/data
    restart: unless-stopped

volumes:
  pgdata:
server — create a test HTML page 
mkdir html
echo '<h1>Deployed with Docker 🐳</h1>' > html/index.html
6
Create the .env file for secrets
Store sensitive values in a .env file that Compose reads automatically — never hardcode secrets in the Compose file itself.
server — root@vps 
cat > /opt/myapp/.env <<'EOF'
DB_PASSWORD=your-strong-password-here
EOF

# Lock it down so only root can read it
chmod 600 /opt/myapp/.env
⚠️
Add .env to your .gitignore — never commit secrets to version control. On the server, chmod 600 ensures only root can read the file.
7
Deploy — bring the stack up in detached mode
Pull the images and start all services in the background. Docker Compose reads the .env file automatically.
server — root@vps 
cd /opt/myapp
docker compose up -d --pull always
expected output
[+] Pulling 2/2
✔ web Pulled 3.2s
✔ db Pulled 8.1s
[+] Running 3/3
✔ Volume "myapp_pgdata" Created
✔ Container myapp-db-1 Started
✔ Container myapp-web-1 Started
8
Verify the stack is running and accessible
Check that containers are up on the server, then hit the public IP from your local machine to confirm the app is reachable from the internet.
server — root@vps 
# Check all containers are Up
docker compose ps

# Quick local test on the server itself
curl http://localhost
expected output — compose ps
NAME IMAGE STATUS PORTS
myapp-web-1 nginx:alpine Up 12 seconds 0.0.0.0:80->80/tcp
myapp-db-1 postgres:16 Up 12 seconds
local machine — test from the internet 
curl http://YOUR_SERVER_IP
expected output
<h1>Deployed with Docker 🐳</h1>
✅
Your container is now serving traffic from the internet. Any browser hitting http://YOUR_SERVER_IP will reach the nginx container running on your VPS.
9
Ship updates to production
The standard update cycle — edit your Compose file or push a new image tag, then pull and redeploy. Only changed containers restart.
local machine — copy updated files to server 
# Copy updated compose.yml or app files to the server
scp -i ~/.ssh/vps_deploy compose.yml root@YOUR_SERVER_IP:/opt/myapp/
server — pull new images and redeploy 
cd /opt/myapp
docker compose pull           # fetch latest image versions
docker compose up -d         # recreate only changed containers
docker image prune -f        # remove old image versions to free disk
💡
Compose is smart about restarts — if only the web image changed, only that container is recreated. The db container and its volume are left completely untouched.
10
Essential production operations
The commands you'll reach for every time you SSH into a production server.
server — day-to-day operations 
# See what's running and its health
docker compose ps

# Follow logs for all services (Ctrl+C to stop)
docker compose logs -f

# Follow logs for a specific service only
docker compose logs -f web

# Live resource usage
docker stats

# Stop everything (containers removed, volumes kept)
docker compose down

# Stop and wipe volumes — DESTRUCTIVE, loses all DB data
docker compose down -v
⚠️
docker compose down -v deletes all named volumes including your database. In production, only run this when you deliberately want to wipe persistent data. Most of the time, plain docker compose down is what you want.
11
Confirm auto-restart survives a reboot
Verify that restart: unless-stopped works — reboot the server and confirm containers come back automatically.
server — root@vps 
# Reboot the server
reboot

# Wait ~30 seconds, then SSH back in and check
docker compose -f /opt/myapp/compose.yml ps
expected output after reboot
myapp-web-1 Up 18 seconds 0.0.0.0:80->80/tcp
myapp-db-1 Up 18 seconds
✅
Containers restarted automatically — the Docker daemon starts on boot and relaunches any container with a restart policy set. No cron jobs or systemd units needed.
🔍 What happened under the hood
Step 01
🔑
SSH handshake
Your private key proved identity to the server's authorized_keys file — no password ever transmitted.
Step 02
🛡️
ufw configured
Netfilter rules were written to the kernel — all inbound packets except SSH/HTTP/HTTPS are dropped before reaching any process.
Step 03
⬇️
Images pulled
Docker pulled nginx and postgres layers from Docker Hub and cached them locally on the VPS disk.
Step 04
🌐
Port bound to host
80:80 told Docker to bind the VPS's public network interface to the container — internet traffic reaches nginx directly.
Step 05
🔁
Restart policy set
Docker registered each container's restart policy in its metadata — the daemon re-reads this on boot and relaunches the container automatically.
Step 06
📦
Volume persists data
Postgres writes to the named volume on VPS disk — survived the reboot and will survive any future container recreation.
