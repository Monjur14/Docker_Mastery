Task 21  ·  Intermediate

Named Docker Volumes
for Databases

Create a persistent Docker volume, attach it to a PostgreSQL container, write data, delete the container, and prove the data is still there when you bring it back up.

~15 minutes Docker installed Terminal / Shell
📖 Required concepts
💾
Why containers lose data by default
Every container has a writable layer on top of its image layers. Anything written there exists only for the lifetime of that container — delete the container and the data is gone. Databases need storage that outlives the container.
📦
What is a named volume?
A named volume is a Docker-managed storage unit with a human-readable name like pgdata. Docker stores its contents on the host (under /var/lib/docker/volumes/), manages its lifecycle separately from any container, and lets you reattach it to a new container at any time.
🆚
Named volumes vs. bind mounts
Bind mounts point at a specific host path you manage — perfect for source code you edit on your machine. Named volumes are fully managed by Docker — no host path to worry about, portable across machines and operating systems, and the right choice for database storage.
🐘
PostgreSQL's data directory
PostgreSQL stores everything — tables, indexes, WAL files — under a single directory called PGDATA. The official Docker image defaults this to /var/lib/postgresql/data. Mount your volume there and every write Postgres makes is persisted outside the container.
⚙️
Volume lifecycle
Volumes exist independently of containers. You can create them before any container runs, attach them to multiple containers (carefully — not simultaneously for most databases), inspect their contents, and delete them explicitly. docker rm never deletes a volume unless you add -v.
🔑
PostgreSQL environment variables
The official postgres image is configured entirely via environment variables: POSTGRES_USER, POSTGRES_PASSWORD, and POSTGRES_DB are enough to create a ready-to-use database on first boot — no config files needed.
🔬 Command anatomy
docker run -d -v pgdata:/var/lib/postgresql/data -e POSTGRES_PASSWORD=secret postgres:16
docker

The CLI binary — how you talk to the Docker daemon from your terminal.

run

Create and start a new container from the specified image.

-d

Detached mode — run in the background and print the container ID. The database keeps running after the command returns.

pgdata:/var/…

Mount the named volume pgdata at the path Postgres writes all its data to. Docker creates the volume automatically if it doesn't exist yet.

-e

Set an environment variable inside the container. The postgres image reads these to initialise the database on first run.

postgres:16

The official PostgreSQL 16 image from Docker Hub — pinned to a major version rather than latest for stability.

🪜 Step-by-step guide
1
Create the named volume explicitly
Creating the volume upfront makes intent clear. Docker would auto-create it on docker run too, but doing it first means you can inspect it before attaching anything.
shell 
docker volume create pgdata
expected output
pgdata
shell 
docker volume ls
expected output
DRIVER VOLUME NAME
local pgdata

3
Start a PostgreSQL container with the volume attached
Run Postgres in the background, mounting pgdata at its data directory. The three -e variables initialise the database on first boot.
shell 
docker run -d \
  --name pg16-db \
  -v pgdata:/var/lib/postgresql/data \
  -e POSTGRES_USER=admin \
  -e POSTGRES_PASSWORD=secret \
  -e POSTGRES_DB=myapp \
  -p 5432:5432 \
  postgres:16
ℹ️
The first run initialises the cluster — it takes a few seconds. Use docker logs -f pg16-db and wait for "database system is ready to accept connections" before moving on.
4
Wait for Postgres to be ready
Tail the logs until the ready message appears, then press Ctrl+C.
shell 
docker logs -f pg16-db
expected output (last few lines)
PostgreSQL init process complete; ready for start up.
...
LOG: database system is ready to accept connections
5
Write data to the database
Exec into the container, open psql, create a table, and insert a row — the data we'll prove survives container deletion.
shell 
docker exec -it pg16-db psql -U admin -d myapp
psql prompt 
CREATE TABLE greetings (id SERIAL PRIMARY KEY, message TEXT);
INSERT INTO greetings (message) VALUES ('Data survives!');
SELECT * FROM greetings;
\q
expected output
CREATE TABLE
INSERT 0 1
id | message
----+----------------
1 | Data survives!
(1 row)
6
Delete the container completely
Force-remove the running container. This is the moment of truth — a container with ephemeral storage would lose everything here.
shell 
docker rm -f pg16-db
shell 
# Confirm the container is gone
docker ps -a --filter name=pg16-db
expected output
CONTAINER ID IMAGE COMMAND CREATED STATUS PORTS NAMES
shell 
# But the volume is still here
docker volume ls
expected output
DRIVER VOLUME NAME
local pgdata
✅
The container is gone but pgdata still exists. Docker never deletes a volume when you remove a container — you have to explicitly run docker volume rm pgdata or pass docker rm -fv to delete both together.
7
Spin up a brand-new container — attach the same volume
Start a fresh Postgres container and mount the same pgdata volume. Because the data directory already exists and is initialised, Postgres skips the init phase and boots straight into the existing cluster.
shell 
docker run -d \
  --name pg16-db-2 \
  -v pgdata:/var/lib/postgresql/data \
  -e POSTGRES_USER=admin \
  -e POSTGRES_PASSWORD=secret \
  -e POSTGRES_DB=myapp \
  -p 5432:5432 \
  postgres:16
ℹ️
Wait for "ready to accept connections" again before querying — docker logs -f pg16-db-2.
8
Query the new container — verify data survived
Connect to the new container and confirm the table and row written in Step 5 are still there.
shell 
docker exec -it pg16-db-2 psql -U admin -d myapp -c 'SELECT * FROM greetings;'
expected output
id | message
----+----------------
1 | Data survives!
(1 row)
✅
The row created in a deleted container is visible in a brand-new one. The named volume is the source of truth — not the container.
9
Inspect the volume metadata
See exactly where on the host Docker is storing the volume's data and which containers are using it.
shell 
docker volume inspect pgdata
expected output
[
{
"CreatedAt": "2026-07-02T09:00:00Z",
"Driver": "local",
"Mountpoint": "/var/lib/docker/volumes/pgdata/_data",
"Name": "pgdata",
"Scope": "local"
}
]
⚠️
On Mac and Windows, the Mountpoint path is inside a Linux VM — you can't browse it directly in Finder or Explorer. On Linux you can ls that path directly and see Postgres's raw data files.
10
Clean up
Remove the container and, when you're done, the volume. Note that you must delete them separately.
shell 
# Stop and remove the container
docker rm -f pg16-db-2

# Remove the volume (this permanently deletes all data)
docker volume rm pgdata
⚠️
docker volume rm is irreversible. In a real project, omit this step — you want the volume to persist between deployments. Only delete it when you mean to wipe the database entirely.
🔍 What happened under the hood
Step 01
📁
Volume created
Docker allocated a directory under /var/lib/docker/volumes/pgdata/ on the host, independent of any container.
Step 02
🔗
Volume mounted
The kernel bind-mounted the volume directory into the container at /var/lib/postgresql/data.
Step 03
🐘
Cluster initialised
Postgres wrote its initial data files to the mounted volume path — on the host's disk, not in the container layer.
Step 04
💾
Data written to host
Every SQL write flushed to disk landed in the volume directory on the host — not the throwaway container layer.
Step 05
🗑️
Container deleted
Docker removed the container's writable layer only. The volume directory on the host was untouched.
Step 06
♻️
New container, same data
The new container found an already-initialised volume, skipped init, and read back the exact rows written by the old container.
🎯 What you learned
  • Why containers lose data by default, and why databases need a volume
  • How to create and list named volumes with docker volume create and docker volume ls
  • How to attach a volume to a Postgres container and configure it with environment variables
  • That docker rm never deletes a volume — volumes and containers have independent lifecycles
  • How to verify persistence by querying a brand-new container mounted on the same volume
  • How to read volume metadata and the host Mountpoint with docker volume inspect
