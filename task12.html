Task 12  ·  Beginner

Inspect Image
Metadata

Use docker image inspect and docker history to explore layers, environment variables, and config baked into an image.

~8 minutes Docker installed Terminal / Shell
📖 Required concepts
🧾
Images carry more than files
Every image bundles a config object alongside its filesystem layers — default command, entrypoint, exposed ports, environment variables, working directory, and the user it runs as. This is exactly what gets copied into every container started from it.
🔬
docker image inspect
docker image inspect dumps the full JSON metadata for an image — architecture, OS, config, layer digests (RootFS.Layers), creation time, and more. It's the same data Docker itself reads before running a container.
🕰️
docker history
docker history lists every layer that makes up an image in build order, showing the Dockerfile instruction that created it and the size it added. It's effectively a reverse-engineered view of the Dockerfile.
🎯
--format and Go templates
Both commands accept --format with a Go template string, letting you pull out one specific field (like {{.Config.Env}}) instead of scrolling through the entire JSON blob by hand.
🌱
ENV vars are baked in, not just runtime
Environment variables set with ENV in a Dockerfile become part of the image itself — every container from that image inherits them automatically, before you ever pass -e flags of your own.
🧩
CMD vs. ENTRYPOINT
Both fields live in the image config and control what runs when a container starts. ENTRYPOINT is the fixed executable; CMD supplies default arguments to it (or is the whole command if no entrypoint is set). Inspecting an image shows you exactly what a bare docker run will actually execute.
🔬 Command anatomy
docker image inspect --format '{{.Config.Env}}' python:3.12-slim
docker

The CLI binary — how you talk to the Docker daemon from your terminal.

image inspect

Subcommand that returns the full JSON metadata document for a local image.

--format '...'

Optional Go template — extracts one field instead of printing the whole JSON document.

python:3.12-slim

The image (repository:tag) to inspect. Must already exist locally.

🪜 Step-by-step guide
1
Make sure you have an image to inspect
Pull the Nginx image if you don't already have it locally.
shell 
docker pull nginx:latest
2
Dump the full metadata JSON
Run docker image inspect with no formatting to see the raw structure. It's long — skim for the top-level keys.
shell 
docker image inspect nginx:latest
ℹ️
The output is a JSON array with one object, since you inspected a single image. Key sections to look for: Config, RootFS, Architecture, and Created.
3
Extract just the environment variables
Use --format to pull only the Env array baked into the image.
shell 
docker image inspect --format '{{.Config.Env}}' nginx:latest
expected output
[PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin NGINX_VERSION=1.27.0 PKG_RELEASE=1~bookworm]
4
Check the default command and entrypoint
See exactly what process starts when you run this image with no extra arguments.
shell 
docker image inspect --format 'Entrypoint: {{.Config.Entrypoint}}  Cmd: {{.Config.Cmd}}' nginx:latest
expected output
Entrypoint: [/docker-entrypoint.sh] Cmd: [nginx -g daemon off;]
5
Check exposed ports and working directory
Two more common config fields worth knowing how to read.
shell 
docker image inspect --format 'ExposedPorts: {{.Config.ExposedPorts}}  WorkDir: {{.Config.WorkingDir}}' nginx:latest
expected output
ExposedPorts: map[80/tcp:{}] WorkDir:
6
Walk the build history layer by layer
Use docker history to see the Dockerfile instructions that built this image, in order, with the size each one added.
shell 
docker history nginx:latest
expected output
IMAGE CREATED CREATED BY SIZE
a1b2c3d4e5f6 3 weeks ago CMD ["nginx" "-g" "daemon off;"] 0B
<missing> 3 weeks ago STOPSIGNAL SIGQUIT 0B
<missing> 3 weeks ago EXPOSE map[80/tcp:{}] 0B
<missing> 3 weeks ago ENTRYPOINT ["/docker-entrypoint.sh"] 0B
<missing> 3 weeks ago COPY docker-entrypoint.sh / 1.2kB
<missing> 3 weeks ago RUN apt-get update && apt-get install -y ... 62MB
<missing> 3 weeks ago ENV NGINX_VERSION=1.27.0 0B
💡
Rows marked <missing> are intermediate layers with no separate image ID of their own — normal for images pulled from a registry. The largest single layer is almost always a RUN apt-get install or similar package-install step.
7
See untruncated build commands
By default long RUN commands get cut off. Add --no-trunc to see them in full.
shell 
docker history --no-trunc nginx:latest
8
Compare metadata across two images
Inspect Alpine's config the same way and notice how much sparser it is — no baked-in ENV, no exposed ports, no entrypoint script.
shell 
docker image inspect --format 'Env: {{.Config.Env}}  Cmd: {{.Config.Cmd}}' alpine:latest
expected output
Env: [PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin] Cmd: [/bin/sh]
📐 Config field reference
Field What it tells you
.Config.Env Environment variables baked into the image, inherited by every container started from it.
.Config.Cmd Default command/arguments run when no command is given to docker run.
.Config.Entrypoint The fixed executable that Cmd supplies arguments to, if one is set.
.Config.ExposedPorts Ports the image documents as listening — informational only, doesn't publish them automatically.
.Config.WorkingDir Directory a container starts in when it runs, if the Dockerfile set one.
.RootFS.Layers Content-addressed digests of every filesystem layer, in stacking order.
🎯 What you learned
  • How to view an image's full metadata JSON with docker image inspect
  • How to extract a single field using --format and a Go template
  • Where baked-in environment variables, CMD, and ENTRYPOINT live in an image's config
  • How to walk an image's build history layer by layer with docker history
  • How to reveal untruncated build commands with --no-trunc
  • That exposed ports in metadata are documentation only, not an automatic publish
