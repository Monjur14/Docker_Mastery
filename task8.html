Roadmap / Task 8 Beginner
Task 08  ·  Beginner

Interactive
Shell Access

Spin up an Ubuntu container and drop into its bash terminal interactively. Learn to run Linux commands inside a live container, explore the filesystem, and understand how -it flags wire your terminal to the container process.

~15 minutes Tasks 1–7 complete Terminal / Shell
📖 Required concepts
🖥️
What "interactive" means
By default, docker run starts a container and lets its process run in the background without any keyboard input from you. Interactive mode attaches your terminal's stdin to the container so you can type commands into it as if it were a real machine.
🔌
The -i and -t flags
-i (interactive) keeps stdin open so you can send keystrokes to the container. -t (TTY) allocates a pseudo-terminal that makes the shell behave like a real one — with a prompt, colors, arrow-key history, and Ctrl+C handling. You almost always use both together as -it.
🏗️
Isolated filesystem
The container has its own complete Linux filesystem, separate from your host. It has /etc, /var, /home, its own process table, and its own network interfaces. Changes you make inside the container do not affect your host machine.
🔄
Container lifetime = process lifetime
The container stays alive as long as its main process runs. When you run bash interactively, the container lives until you type exit or press Ctrl+D. At that point bash exits, and the container stops automatically.
📦
The ubuntu image
The official ubuntu image on Docker Hub is a minimal Ubuntu base — it contains the OS filesystem and package manager (apt) but strips out many tools to keep the image small (about 77MB). You'll often need to apt-get install tools you need inside it.
🔗
exec vs run
docker run creates a brand new container from an image. docker exec opens an additional shell inside an already-running container. This task covers both — you'll use run first, then exec to re-enter a named container.
🔬 Command anatomy
docker run -it --name my-ubuntu ubuntu bash
docker

The CLI binary — how you talk to the Docker daemon from your terminal.

run

Create and start a new container from an image.

-it

Combined -i (keep stdin open) + -t (allocate a pseudo-TTY). Together they give you a live, interactive terminal session.

--name

Assign a memorable name to the container instead of a random one. Makes it easier to reference later with docker exec or docker start.

ubuntu

The image to use. Pulls ubuntu:latest from Docker Hub if not cached.

bash

The command to run inside the container, overriding the image's default. This drops you into a bash shell as root.

Flag What it does
-i
Keep stdin open even if not attached. Without this, your keystrokes won't reach the container.
-t
Allocate a pseudo-TTY. Makes the shell display a prompt, handle Ctrl+C, and render colors correctly.
--name
Give the container a fixed name so you can reference it by name instead of its auto-generated ID.
--rm
Auto-remove the container when it exits. Useful for throwaway sessions so you don't accumulate stopped containers.
🪜 Step-by-step guide
1
Start an interactive Ubuntu container
This command pulls Ubuntu if needed, creates a container, and drops you straight into a bash shell. Your prompt will change to a root shell inside the container.
shell — host 
docker run -it --name my-ubuntu ubuntu bash
expected output
Unable to find image 'ubuntu:latest' locally
latest: Pulling from library/ubuntu
7b1a6ab2e44d: Pull complete
Status: Downloaded newer image for ubuntu:latest

root@a1b2c3d4e5f6:/# ▌
ℹ️
The prompt root@a1b2c3d4e5f6:/# means you're now inside the container. The hex string is the container ID, and # indicates you're running as root. All commands you type from here run inside the container.
2
Explore the container filesystem
Run a few standard Linux commands to confirm you're inside an isolated Ubuntu environment.
shell — inside container 
# Where are we?
pwd

# What OS is this?
cat /etc/os-release

# List root filesystem
ls /

# Check running processes (just bash + ps)
ps aux
expected output (cat /etc/os-release)
PRETTY_NAME="Ubuntu 24.04.1 LTS"
NAME="Ubuntu"
VERSION_ID="24.04"
VERSION="24.04.1 LTS (Noble Numbat)"
ID=ubuntu
💡
Notice that ps aux only shows two processes — bash and ps itself. The container is fully isolated; you can't see your host's processes from inside.
3
Install a package with apt
The Ubuntu image is minimal — tools like curl aren't pre-installed. Install one to see how apt works inside a container.
shell — inside container 
# Update package list
apt-get update

# Install curl (-y skips the confirmation prompt)
apt-get install -y curl

# Confirm it works
curl --version
⚠️
These changes are ephemeral. Packages installed inside a running container are lost when the container is removed. To persist software, add it to a Dockerfile — that's covered in Task 10.
4
Create a file inside the container
Write a file to understand container isolation — it will exist inside the container but not on your host.
shell — inside container 
echo "I exist only inside this container" > /tmp/hello.txt
cat /tmp/hello.txt
expected output
I exist only inside this container
5
Detach without stopping the container
Press Ctrl+P then Ctrl+Q to detach from the container's terminal while leaving it running. This is the escape hatch — the container stays alive in the background.
keyboard shortcut 
# Inside the container — press these keys in sequence:
Ctrl+P  →  Ctrl+Q
expected output (back on host)
read escape sequence
$ ▌
ℹ️
You're back on your host shell. The container is still running in the background. Confirm it with docker ps — you should see my-ubuntu listed with status Up.
6
Re-enter the running container with exec
Use docker exec to open a new bash session inside the already-running container. Your file from Step 4 should still be there.
shell — host 
docker exec -it my-ubuntu bash
shell — inside container (verify file persists) 
cat /tmp/hello.txt
expected output
I exist only inside this container
💡
exec vs run: docker exec opens a new process inside an existing container. docker run would create a brand new container. Using exec here means we're in the same container, with the same filesystem state.
7
Exit and confirm isolation
Type exit to leave the exec shell (the container keeps running since bash was launched by exec, not as the main process). Then verify the file doesn't exist on your host.
shell — inside container 
exit
shell — host (confirm isolation) 
# This file doesn't exist on your host machine
cat /tmp/hello.txt
expected output
cat: /tmp/hello.txt: No such file or directory
✅
The file truly doesn't exist on your host — that's filesystem isolation working correctly. The container's writable layer is completely separate from your machine's /tmp.
8
Stop the container
The container is still running in the background (its original bash session is active). Stop it by name.
shell — host 
docker stop my-ubuntu
ℹ️
Stopping a container sends SIGTERM to its main process, giving it 10 seconds to shut down gracefully, then SIGKILL. The container's state is preserved as "Exited" — it can be restarted with docker start -i my-ubuntu.
9
Try the --rm flag for throwaway sessions
For quick one-off sessions you don't need to keep, add --rm. The container deletes itself as soon as you exit — no cleanup needed.
shell — host 
docker run -it --rm ubuntu bash
shell — inside container 
# Do whatever you need, then exit
echo "Hello from a throwaway container"
exit
💡
After exit, run docker ps -a — the container won't appear because --rm deleted it automatically. Perfect for quick debugging sessions.
10
Clean up
Remove the named container and optionally the Ubuntu image.
shell — host 
# Remove the named container
docker rm my-ubuntu

# Remove the ubuntu image (optional — useful in other tasks)
docker rmi ubuntu
