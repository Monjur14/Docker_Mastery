Task 05  ·  Intermediate

Container Logs
& Troubleshooting

View and follow real-time stdout logs from a running container using docker logs --follow. Learn to diagnose crashed containers, tail output, and debug live processes.

~10 minutes Docker installed Terminal / Shell
📖 Required concepts
📝
stdout / stderr capture
Docker captures everything a container's main process writes to stdout and stderr. It doesn't read log files inside the container by default — it reads the process's output streams directly, via a logging driver.
🚚
Logging drivers
By default Docker uses the json-file driver, storing each line as JSON on disk. Other drivers (journald, syslog, awslogs, etc.) forward logs elsewhere. docker logs only works with drivers that support reading back.
👀
Following vs. one-shot reads
docker logs alone prints everything captured so far and exits. Adding --follow (-f) keeps the stream open, printing new lines as they're written — similar to tail -f on a log file.
🩺
Logs after a crash
Logs are stored per-container, not per-process — so docker logs still works on a stopped or crashed container. This is usually the first place to look when something exits unexpectedly.
🔢
Exit codes
When a container stops, it reports an exit code. 0 means success. Non-zero codes (like 1 for a general error or 137 for an out-of-memory kill) point you toward the cause before you even read a single log line.
🔧
Live debugging with exec
Logs tell you what happened; docker exec lets you get a shell inside a running container to poke around live — check processes, files, or network state that logs alone won't show.
🔬 Command anatomy
docker logs --follow log-demo
docker

The CLI binary — how you talk to the Docker daemon from your terminal.

logs

Subcommand that fetches the captured stdout/stderr output for a specific container.

--follow

Shorthand -f. Keeps the connection open and streams new log lines as they arrive, instead of exiting after the existing output.

log-demo

The container's name or ID. You can find this with docker ps.

🪜 Step-by-step guide
1
Start a container that emits logs over time
Run a small BusyBox loop in the background that prints a timestamped line every 2 seconds — a stand-in for a real app's log stream.
shell 
docker run -d --name log-demo busybox sh -c 'i=0; while true; do echo "tick $i: $(date)"; i=$((i+1)); sleep 2; done'
ℹ️
-d runs the container in the background (detached) so it keeps running and printing logs after the command returns.
2
View existing logs (one-shot)
Fetch everything logged so far, then return control to your terminal.
shell 
docker logs log-demo
expected output
tick 0: Wed Jul 1 09:12:01 UTC 2026
tick 1: Wed Jul 1 09:12:03 UTC 2026
tick 2: Wed Jul 1 09:12:05 UTC 2026
3
Follow logs in real time
Add --follow to keep the stream open — new lines appear as the container writes them, just like tail -f.
shell 
docker logs --follow log-demo
expected outputstreaming
tick 14: Wed Jul 1 09:12:29 UTC 2026
tick 15: Wed Jul 1 09:12:31 UTC 2026
tick 16: Wed Jul 1 09:12:33 UTC 2026
... new lines keep appearing every 2 seconds
💡
Press Ctrl+C to stop following. This only detaches your terminal from the stream — the container keeps running.
4
Narrow the output
For a noisy container, combine flags to see only recent, timestamped lines instead of scrolling through everything.
--tail N Show only the last N lines instead of the full history.
-t Prefix each line with an ISO 8601 timestamp Docker recorded when it captured it.
--since TIME Show logs since a given time — e.g. 10m or an RFC3339 timestamp.
shell 
docker logs --tail 5 -t --follow log-demo
5
Simulate a crash and read the logs afterward
Start a container that fails on purpose, then confirm docker logs still works even though it already exited.
shell 
# A container that logs an error, then exits with a failure code
docker run --name crash-demo busybox sh -c 'echo "starting up..."; echo "FATAL: config file not found" >&2; exit 1'
expected output
starting up...
FATAL: config file not found
shell 
# Confirm the exit code and re-read the logs from a stopped container
docker ps -a --filter name=crash-demo
docker logs crash-demo
expected output
CONTAINER ID IMAGE COMMAND CREATED STATUS NAMES
7f3a9c1e2b0d busybox "sh -c …" 3 seconds ago Exited (1) 2 seconds ago crash-demo
⚠️
Exited (1) is a general error exit code. Common non-zero codes: 1 unhandled application error, 126 command not executable, 127 command not found, 137 killed (often out-of-memory, SIGKILL).
6
Inspect for more detail
Logs tell you what was printed; docker inspect tells you the machine-readable facts — including the exact exit code and whether it was OOM-killed.
shell 
docker inspect crash-demo --format 'ExitCode: {{.State.ExitCode}}  OOMKilled: {{.State.OOMKilled}}'
expected output
ExitCode: 1 OOMKilled: false
7
Get a live shell for deeper debugging
When logs aren't enough, exec into the running log-demo container to look around from the inside.
shell 
docker exec -it log-demo sh
ℹ️
A stopped container has no running process to exec into — docker exec only works on containers that are currently up. Type exit to leave the shell.
8
Clean up
Stop and remove the containers used in this task.
shell 
docker rm -f log-demo crash-demo
🔍 What happened under the hood
Step 01
🖨️
Process writes output
The container's main process writes lines to stdout/stderr as it runs.
Step 02
💾
Daemon captures it
The logging driver (json-file by default) persists each line to disk on the host.
Step 03
📡
CLI requests logs
docker logs asks the daemon to read back what's stored.
Step 04
🔗
Stream stays open
With --follow, the daemon keeps pushing new lines over the same connection.
Step 05
🧩
Exit code recorded
When the process ends, its exit status is stored in the container's metadata.
Step 06
🕵️
Post-mortem read
Logs and inspect data remain available after the container stops, for debugging later.
🎯 What you learned
  • How Docker captures a container's stdout/stderr via a logging driver
  • The difference between a one-shot docker logs read and a live --follow stream
  • How to narrow output with --tail, -t, and --since
  • That logs remain readable after a container stops or crashes
  • How to read exit codes and use docker inspect for deeper failure details
  • How to get a live shell inside a running container with docker exec
