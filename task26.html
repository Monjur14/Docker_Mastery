Task 26  ·  Intermediate

Custom
Bridge Networks

Create a user-defined bridge network and connect two containers to it. Once on the same network, containers can reach each other by container name — no IP addresses, no --link, no hacks.

~20 minutes Tasks 8 & 25 complete Terminal / Shell
📖Required concepts
🌉
What is a bridge network?
A bridge network is a virtual switch inside your host. Docker creates one by default (called bridge), but it has a critical limitation: containers on the default bridge can only find each other by IP address — which changes every restart. A user-defined bridge adds automatic DNS so containers find each other by name.
🔍
Automatic DNS resolution
When you attach a container to a user-defined network, Docker's embedded DNS server automatically registers the container's name and any aliases as hostnames. Any other container on the same network can ping or curl my-container and it just works — even if the IP changes.
🔒
Network isolation
Containers on different user-defined networks are isolated from each other by default — they cannot communicate. A container can be attached to multiple networks simultaneously, acting as a bridge between them. This lets you design precise access control: a frontend talks to the app layer, but never directly to the database.
⚡
--network vs docker network connect
You can join a network at container creation with --network, or connect a running container to an additional network with docker network connect. Both work. You'll use both approaches in this task.
🆚
Default bridge vs user-defined bridge
The default bridge (docker0) has no DNS, requires --link for name resolution (deprecated), and all containers share it. A user-defined bridge has DNS, better isolation, and is the recommended approach for any multi-container setup. Docker Compose always creates user-defined bridges automatically.
📡
Published ports vs container networking
Published ports (-p) expose a container to your host machine. Containers talking to each other on the same network don't need published ports — they communicate directly on the internal network using the container's native port. Publishing is only needed for host or external access.
Feature Default bridge User-defined bridge
DNS by container name ✗ No ✓ Yes
Automatic IP discovery ✗ Must use --link ✓ Built-in
Isolation between networks ✗ Shared ✓ Isolated
Recommended for production ✗ No ✓ Yes
Used by Docker Compose ✗ No ✓ Always
🗺️What we're building
Target network topology
app-net (user-defined bridge)
web-server
nginx · port 80
⟷ by name
api-server
alpine · curl
host machine — port 8080 → web-server:80

You'll create a network called app-net, start an nginx container named web-server on it, then start an Alpine container named api-server on the same network and prove it can reach web-server by name.

🔬Command anatomy
docker network create --driver bridge app-net
docker

The Docker CLI binary.

network create

Subcommand to create a new Docker network. The created network persists until you explicitly remove it.

--driver

bridge is the default driver for single-host networking. Other drivers include overlay (multi-host Swarm) and host. Can be omitted since bridge is the default.

app-net

The name of your new network. You'll reference this in --network flags when starting containers.

docker run -d --name web-server --network app-net -p 8080:80 nginx
--name

Gives the container the name web-server. This is the hostname other containers on app-net will use to reach it.

--network

Attaches the container to app-net instead of the default bridge. Docker registers its name in the network's DNS immediately.

-p 8080:80

Publishes port 80 to your host on 8080. Needed for you to visit it in a browser — other containers on app-net don't need this to reach it.

🪜Step-by-step guide
1
Create the user-defined bridge network
Create a new network called app-net. This is the virtual switch both containers will share.
shell
docker network create --driver bridge app-net
expected output
a3f1c8d2e9b04a7f6c2e1d8b3a5f0e9c7d2a1b4f8e3c6d0a9b2e5f1c4d7a0
ℹ️
Docker returns the full network ID. Verify it was created with docker network ls — you should see app-net with driver bridge.
2
List networks to confirm
See all networks Docker knows about, including the three built-in ones and your new one.
shell
docker network ls
expected output
NETWORK ID NAME DRIVER SCOPE
b1c2d3e4f5a6 bridge bridge local
c7d8e9f0a1b2 host host local
d3e4f5a6b7c8 none null local
a3f1c8d2e9b0 app-net bridge local
3
Start the first container on app-net
Launch an nginx container named web-server on the network. Publishing port 8080 lets you verify it from your browser, but it's the --name that matters for inter-container DNS.
shell
docker run -d \
  --name web-server \
  --network app-net \
  -p 8080:80 \
  nginx
💡
Open http://localhost:8080 in your browser — you should see the nginx welcome page. This confirms the container is running and the host port mapping works.
4
Start the second container on the same network
Launch a lightweight Alpine container named api-server on app-net. It will stay alive with a sleep command so you can exec into it.
shell
docker run -d \
  --name api-server \
  --network app-net \
  alpine sleep infinity
ℹ️
sleep infinity keeps the Alpine container running forever without doing anything — a common trick for keeping a shell-only container alive so you can exec into it.
5
Prove DNS resolution works — ping by name
Exec into api-server and ping web-server by name. No IP address — just the container name.
shell
docker exec -it api-server ping -c 3 web-server
expected output
PING web-server (172.18.0.2): 56 data bytes
64 bytes from 172.18.0.2: seq=0 ttl=64 time=0.142 ms
64 bytes from 172.18.0.2: seq=1 ttl=64 time=0.098 ms
64 bytes from 172.18.0.2: seq=2 ttl=64 time=0.104 ms

--- web-server ping statistics ---
3 packets transmitted, 3 packets received, 0% packet loss
🌉
Docker's embedded DNS resolved web-server to its private IP on the network. Notice the IP is in the 172.18.x.x range — that's the subnet Docker allocated for app-net.
6
Make an HTTP request by container name
Install curl inside Alpine, then fetch the nginx welcome page using the container name as the hostname — exactly how microservices talk to each other in production.
shell
# Install curl inside api-server
docker exec api-server apk add --no-cache curl

# Fetch nginx's welcome page using the container name as host
docker exec api-server curl -s http://web-server/
expected output (truncated)
<!DOCTYPE html>
<html>
<head><title>Welcome to nginx!</title></head>
...
✅
The request went from api-server → app-net DNS → web-server:80 entirely inside Docker's virtual network. No host ports involved — this is how container-to-container communication works in real stacks.
7
Inspect the network
Use docker network inspect to see which containers are connected, their assigned IPs, and the network's subnet.
shell
docker network inspect app-net
expected output (key fields)
[{
"Name": "app-net",
"Driver": "bridge",
"IPAM": { "Config": [{ "Subnet": "172.18.0.0/16" }] },
"Containers": {
"...": { "Name": "web-server", "IPv4Address": "172.18.0.2/16" },
"...": { "Name": "api-server", "IPv4Address": "172.18.0.3/16" }
}
}]
8
Prove isolation — start a container outside the network
Start a third Alpine container without --network app-net and try to ping web-server from it. It should fail.
shell
# No --network flag → lands on the default bridge
docker run --rm alpine ping -c 2 web-server
expected output
ping: bad address 'web-server'
🔒
Isolation working correctly. The default bridge has no DNS for user-defined networks, and there's no route to app-net's subnet. Containers that shouldn't talk to each other simply can't — without any firewall rules to maintain.
9
Connect a running container to an additional network
You can attach a running container to another network without restarting it. Create a second network and connect web-server to it — now it's on two networks simultaneously.
shell
# Create a second network
docker network create db-net

# Connect the running web-server to it — no restart needed
docker network connect db-net web-server

# Confirm web-server now shows two networks
docker inspect web-server --format '{{json .NetworkSettings.Networks}}'
ℹ️
docker network connect is how you add a network to an already-running container. The reverse — docker network disconnect — removes it. Both take effect immediately.
10
Clean up
Stop and remove both containers, then remove the networks. Networks can only be removed once no containers are connected to them.
shell
# Stop and remove containers
docker stop web-server api-server
docker rm web-server api-server

# Remove the networks
docker network rm app-net db-net

# Or remove all unused networks at once
docker network prune
🔍What happened under the hood
Step 01
🌉
Virtual switch
Docker created a new Linux bridge interface (br-xxxx) on the host for app-net.
Step 02
📡
DNS registered
Each container's name was registered in Docker's embedded DNS server (127.0.0.11 inside containers).
Step 03
🔌
veth pair created
Docker created a virtual ethernet pair per container: one end in the container, one end on the bridge.
Step 04
🔍
Name resolved
"web-server" → DNS lookup → 172.18.0.2 → routed via bridge to web-server's veth interface.
Step 05
🔒
Isolation enforced
iptables rules prevent traffic crossing between different bridge networks by default.
Step 06
⚡
Hot-connect
docker network connect added a second veth pair to web-server without stopping the container.
🎯What you learned
  • How to create a user-defined bridge network with docker network create and why it's better than the default bridge
  • That containers on the same user-defined network resolve each other by container name — no IP addresses needed
  • How to attach containers to a specific network with --network at creation time
  • That containers on different networks are isolated — a container not on app-net cannot reach its members
  • How to hot-connect a running container to an additional network with docker network connect
  • How to inspect a network's members and subnet with docker network inspect
